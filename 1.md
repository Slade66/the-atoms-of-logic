# 设计

一个函数读起来应该**只在一个抽象层级**上。在"业务流程"层级，如果把 SQL、并发、模板渲染都塞进来，就同时混了三个层级，读者被迫在"我在看流程"和"我在看具体实现细节"之间反复横跳。

## 2

开发软件的第一步，其实不是打开编辑器写代码，而是想清楚你到底要做什么。 

失败，往往不是因为技术不行，而是一开始没搞清楚需求和边界就动手了。

明确核心问题： 软件的功能很多——比如滴答清单有任务、清单、日历、番茄钟、习惯打卡、协作……你不可能一上来全做。先问自己一个问题：这个软件要解决的最核心痛点是什么？把这个想清楚，能帮你砍掉 80% 不重要的功能。


定义最小可用版本（MVP）： 在第一个版本里，只保留最不可或缺的功能。比如：对一个任务管理软件来说，MVP 通常就是：能新建任务、能标记完成、能查看任务列表。其他的（标签、提醒、子任务、同步）全部往后排。能跑起来、能用，比功能齐全重要得多。

### Project Eunomia

任务是"我要做什么"，时间轴是"我的时间实际花在了哪"，习惯是"我能不能持续地做"。三者通过"任务"这个核心对象串起来——尤其是时间模块会引用任务

后端用你的 Go + Kratos： 这块是你的强项，直接发挥。它给前端提供数据接口、负责把手机和电脑的数据同步对齐。Kratos 原生支持 gRPC 和 HTTP，跟前端对接很顺。

架构： 本地优先——界面只读写设备本地数据，永远即时；后台再把数据同步到你的 Go 服务。


前端： Vue，用 Tauri 打包成 macOS 和 Android 的原生应用（启动快、体积小）。

Tauri 用 Vue 写界面，外面套一层 Tauri，它会编译出真正的 macOS 和 Android 原生安装包。它用的是系统自带的 webview，不像 Electron 那样把整个 Chromium 打包进去，所以体积小、启动快。Tauri 2.0 之后正式支持安卓和 iOS，桌面端也一直支持 macOS

单仓库：我是一个人开发，最省心的是一个 monorepo，仓库叫 eunomia，里面分 server/（你的 Kratos 后端）和 web/（Vue 前端）两个目录。

模块化单体：任务、时间轴、习惯、Agent 这四个，不是四个独立服务，而是同一个服务内部的四个模块。

一个进程、一次部署，内部按领域分清楚。所以你用默认的单服务骨架就对了，千万别用文档里那个 --nomod 大仓库模式（那个是给"多个独立服务"准备的，会把你引向微服务

Kratos 的单服务骨架是分层的，模块在每一层各加一组文件。
eunomia/
├── api/                        # proto 定义：你的数据模型 + 接口，唯一的源头
│   ├── task/v1/task.proto
│   ├── timeline/v1/timeline.proto
│   ├── habit/v1/habit.proto
│   └── agent/v1/agent.proto
├── cmd/eunomia/
│   ├── main.go                 # 程序入口
│   ├── wire.go                 # 依赖注入声明
│   └── wire_gen.go             # wire 自动生成
├── configs/
│   └── config.yaml             # MySQL 连接等配置放这
└── internal/
    ├── conf/                   # 配置的 proto
    ├── server/
    │   ├── http.go             # 注册所有模块的 HTTP 路由
    │   ├── grpc.go             # 注册所有模块的 gRPC
    │   └── server.go
    ├── service/                # 接口层：实现 proto 里的 service，转发给 biz
    │   ├── task.go
    │   ├── timeline.go
    │   ├── habit.go
    │   └── agent.go
    ├── biz/                    # 业务逻辑层（核心领域逻辑）
    │   ├── task.go
    │   ├── timeline.go
    │   ├── habit.go
    │   └── agent.go
    └── data/                   # 数据访问层：你的 MySQL 操作都在这
        ├── data.go             # DB 连接初始化
        ├── task.go
        ├── timeline.go
        ├── habit.go
        └── agent.go

请求从 api（proto 定的接口）进来，先到 service（接口层，做参数转换），再交给 biz（业务逻辑，不关心是 HTTP 还是 MySQL），biz 需要数据时调 data（去操作 MySQL），最后 server 负责把 HTTP 和 gRPC 端口起起来、把所有模块的 service 注册上去。

Agent 模块在这种结构下有个白捡的好处： 因为四个模块在同一个进程里，Agent 的 biz 层可以直接调用任务和时间轴的 biz 层去拿数据，不用走网络、不用调接口。这正是当初选单体而不是微服务，给你省下的麻烦。

server/ 放你的 Go 单体后端，web/ 放 Vue 前端。这就是个人项目最清爽的单仓库（monorepo）
