## govmomi 启用与使用 CBT 的具体步骤

使用 `govmomi` 库启用和操作虚拟机 CBT（Change Block Tracking）块变化追踪，具体分为以下几步：

1. **启用 CBT**：在 `govmomi` 中，通过调用 `VirtualMachine.Reconfigure(...)` 将配置中的 `ChangeTrackingEnabled` 开关打开：
   ```go
   spec := types.VirtualMachineConfigSpec{
       ChangeTrackingEnabled: types.NewBool(true),
   }
   task, err := vm.Reconfigure(ctx, spec)
   if err != nil {
       return err
   }
   if err := task.Wait(ctx); err != nil {
       return err
   }
   ```
2. **创建 Snapshot（静态一致性点）**：在备份或迁移读取磁盘前，必须先建立快照，使后续的读取行为基于一个静止且一致的数据状态：
   ```go
   snapTask, err := vm.CreateSnapshot(ctx, "cbt-base", "backup snapshot", false, true)
   if err != nil {
       return err
   }
   if err := snapTask.Wait(ctx); err != nil {
       return err
   }
   ```
3. **按磁盘保存 `changeId`**：需要特别注意，`changeId` 不是按整台虚拟机保存的，而是按每块磁盘独立生成的。在第一次基线同步时，需要将磁盘 Key、Snapshot 引用以及该时刻的 `changeId` 一同持久化记录下来，作为后续增量对比的基准标识。
4. **查询 Changed Extents**：后续调用 `QueryChangedDiskAreas(...)`，传入保存的对比基准标识，以分页的形式分段获取该磁盘在快照前后的物理变更区间。

## QueryChangedDiskAreas 返回数据与增量逻辑分析

- **输出对象分析**：`QueryChangedDiskAreas` API 会返回一个 `DiskChangeInfo` 结构体，其中主要包含本次查询的起始偏移量、总范围长度、变更块区间列表（`changedArea`），以及用于下一轮增量对比的物理状态标识。
- **增量匹配逻辑**：在第一次全量同步时，通常传入 `"*"` 作为初始对比标识，以获取磁盘当前所有已分配的物理区域；在后续增量同步时，传入上一轮保存的 `changeId`，即可仅查询自该时刻起发生改变的数据区域。
- **返回 JSON 示例**：
  ```json
  {
    "startOffset": 0,
    "length": 10737418240,
    "changedArea": [
      {
        "startOffset": 1048576,
        "length": 65536
      },
      {
        "startOffset": 5242880,
        "length": 131072
      }
    ]
  }
  ```
- **示例数据含义**：上述 JSON 意味着在查询起始点为 0、覆盖 10GB 范围的磁盘时，有两段连续区域发生了改变（一段是从 1MB 偏移处开始的 64KB 数据，另一段是从 5MB 偏移处开始的 128KB 数据）。数据迁移程序只需定向读取和写入这两段 extents，而无需对整块磁盘做物理扫描。
- **物理区间映射的意义**：CBT 的输出与文件系统和应用层逻辑无关，它仅仅是最底层的块设备物理区间差异，因此能够非常纯粹、无损地映射给下层的读盘与写盘逻辑。

## VMware 到 FusionCompute 的三阶段同步推荐路线

针对 VMware 虚拟机迁移至 FusionCompute，推荐采用以下三阶段同步机制以最小化业务停机时间：

1. **第一阶段：全量基线同步**：为目标盘创建 CBT 并建立首个快照，全量读取源端数据并写入目标卷，同时保存每块盘对应的初始 `changeId`。此过程在后台执行，业务无感知。
2. **第二阶段：多轮增量同步**：业务运行期间，定期重新建立快照，调用 `QueryChangedDiskAreas` 传入前一次保存的 `changeId` 以获取变化块，仅对增量块进行读取和直写同步，持续逼近两端的一致状态，缩短割接所需的时间。
3. **第三阶段：最终切换割接**：在预定停机窗口内挂起或关闭源端虚拟机以阻断数据写入，执行最后一次终极增量同步。在数据彻底对齐后，进行驱动注入适配，并在目标 FusionCompute 平台拉起占位虚拟机上线。

## CBT 块变更标记与数据读取机制的局限性

需要明确的是，CBT 本身只具有**物理块标记**的作用：
它仅仅告诉我们“哪些磁盘块发生了改变”，但并不具备“数据传输或读取通道”的功能。

这意味着，它无法直接提供从哪个已被官方验证的安全数据源（例如特定的快照文件或 NFC 租约通道）中，将这些变动的物理块以完全匹配快照时刻的一致性状态读取出来的机制。数据的实际拉取、分发和传输，需要上层程序通过具体的 API（如 VMware NFC 接口）结合 `Range` 参数行使读取并转发。
