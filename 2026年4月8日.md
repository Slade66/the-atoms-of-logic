
### 🧠 用 govmomi 使用 CBT：先开关，再建快照，再查变化

- **第一步：启用 CBT** `govmomi` 里通过 `VirtualMachine.Reconfigure(...)` 打开 `ChangeTrackingEnabled`。这是把 VM 的 change tracking 开关真正打开。

    ```go
    spec := types.VirtualMachineConfigSpec{
        ChangeTrackingEnabled: types.NewBool(true),
    }
    task, err := vm.Reconfigure(ctx, spec)
    if err != nil {
        return err
    }
    if err := task.Wait(ctx); err != nil {
        return err
    }
    ```

- **第二步：创建 snapshot** 备份或迁移前通常先建快照，让读取基于一个静态一致点。

    ```go
    snapTask, err := vm.CreateSnapshot(ctx, "cbt-base", "backup snapshot", false, true)
    if err != nil {
        return err
    }
    if err := snapTask.Wait(ctx); err != nil {
        return err
    }
    ```

- **第三步：按磁盘保存 `changeId`** `changeId` 不是按整台 VM 保存，而是按每块磁盘保存。第一次基线同步时，要把磁盘 key、snapshot 引用和 `changeId` 一起记录下来，后续增量才有对比基准。

- **第四步：查询 changed extents** `QueryChangedDiskAreas(...)` 会返回某块磁盘在两个时间点之间的变化范围。你可以按磁盘、按 offset 分页拿到变化区间，再只处理这些区间。


### 🧠 QueryChangedDiskAreas：CBT 的查询输出长什么样

- **输出对象** `QueryChangedDiskAreas` 返回的是 `DiskChangeInfo`，里面包含查询区间、变更区间列表，以及用于下次继续增量的状态信息。

- **增量逻辑** 第一次调用时常用 `"*"` 获取已分配区域；后续调用则传入上次保存的 `changeId`，查询自该标识以来发生变化的区域。

- **JSON 样例** 典型输出可以理解成下面这样：

    ```json
    {
      "startOffset": 0,
      "length": 10737418240,
      "changedArea": [
        {
          "startOffset": 1048576,
          "length": 65536
        },
        {
          "startOffset": 5242880,
          "length": 131072
        }
      ]
    }
    ```

- **样例含义** 这表示查询从 0 字节开始，覆盖 10GB 范围，结果里返回了两段连续变化区域，备份程序只需要读取这些 extent，而不需要把整盘重新扫一遍。

- **查询边界的意义** 它输出的不是“文件级别”的差异，也不是“应用级别”的差异，而是最底层的磁盘区间差异。对于上层程序来说，这已经足够再往下映射成读取、同步、导入或写盘动作。


### 🧠 Go 后端集成 VDDK：控制面用 govmomi，数据面用 cgo

- **推荐拆法** 把项目拆成两层最清晰：`pkg/vsphere` 负责连接 vCenter、查 VM、建快照、查 `changeId`；`pkg/vddk` 负责通过 cgo 包装 VDDK 的初始化、连接、打开磁盘和读取数据。

- **最小骨架** 下面是一个非常简化的接入示意，重点是看结构，不是完整生产代码。

    ```go
    /*
    #cgo CFLAGS: -I/path/to/vddk/include
    #cgo LDFLAGS: -L/path/to/vddk/lib64 -lvixDiskLib
    #include "vixDiskLib.h"
    #include <stdlib.h>
    */
    import "C"

    import (
        "fmt"
        "unsafe"
    )

    func main() {
        libDir := C.CString("/path/to/vddk/lib64")
        defer C.free(unsafe.Pointer(libDir))

        err := C.VixDiskLib_InitEx(6, 0, nil, nil, nil, libDir, nil)
        if err != C.VIX_OK {
            fmt.Printf("Init failed: %d\n", err)
            return
        }
        defer C.VixDiskLib_Exit()
    }
    ```

- **集成注意点** C 字符串要记得释放，VDDK 调用时最好固定到同一个 OS 线程，缓冲区也要注意扇区对齐。连接时如果要指定传输模式，可以走 `VixDiskLib_ConnectEx`。

- **架构上的好处** 这样拆以后，Go 代码仍然负责业务编排、状态管理和异常恢复，VDDK 只做它最擅长的磁盘访问。你不会把所有 VMware 细节都挤进业务层，也更容易替换或隔离底层实现。

### 🧠 纯 Go 能不能做：能，但更适合全量，不适合高效块级增量

- **纯 Go 的可行范围** 如果你的目标只是全量迁移，纯 Go 是完全可以做的。它可以用 `govmomi` 负责快照和 `HttpNfcLease`，再通过 HTTP 流式下载 VMDK 数据。

- **它的短板** 纯 Go 的官方能力更像是“控制面够用，数据面偏弱”。CBT 能告诉你变化范围，但如果没有像 VDDK 这样的数据面库，块级随机读取会比较别扭。

- **什么时候继续用纯 Go** 如果你要先做 MVP、磁盘不大、或者更在意部署简单而不是极致性能，纯 Go 会更轻。

- **什么时候不建议只用纯 Go** 如果你要做生产级增量迁移、多轮追平、严格压缩停机窗口，那就更适合引入 VDDK。

- **这个判断的核心** 不是“纯 Go 能不能连上 VMware”，而是“纯 Go 能不能优雅地把 CBT 返回的变化区间变成高效的数据面读写”。如果答案只是勉强能做，那工程上通常就会变成权衡明显的次优方案。

### 🧠 面向 VMware → FusionCompute 的推荐路线

- **第一阶段：全量基线** 先创建 CBT 并建快照，全量读盘，把初始数据同步到目标端，同时保存每块盘的 `changeId`。

- **第二阶段：多轮增量** 每轮都重新建快照，再调用 `QueryChangedDiskAreas` 拿 changed extents，只同步变化块，持续缩小两端差异。

- **第三阶段：最终切换** 停机后做最后一次增量，收口完成后在目标平台拉起虚机。

### CBT 只是块变化，不能拉数据

CBT 只告诉我们“哪些块变了”，但没有直接告诉我们“该从哪个已被官方验证的数据源，把这些块按快照时刻正确读出来”
