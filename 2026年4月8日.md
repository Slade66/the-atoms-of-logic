
### 🧠 用 govmomi 使用 CBT：先开关，再建快照，再查变化

- **第一步：启用 CBT** `govmomi` 里通过 `VirtualMachine.Reconfigure(...)` 打开 `ChangeTrackingEnabled`。这是把 VM 的 change tracking 开关真正打开。

    ```go
    spec := types.VirtualMachineConfigSpec{
        ChangeTrackingEnabled: types.NewBool(true),
    }
    task, err := vm.Reconfigure(ctx, spec)
    if err != nil {
        return err
    }
    if err := task.Wait(ctx); err != nil {
        return err
    }
    ```

- **第二步：创建 snapshot** 备份或迁移前通常先建快照，让读取基于一个静态一致点。

    ```go
    snapTask, err := vm.CreateSnapshot(ctx, "cbt-base", "backup snapshot", false, true)
    if err != nil {
        return err
    }
    if err := snapTask.Wait(ctx); err != nil {
        return err
    }
    ```

- **第三步：按磁盘保存 `changeId`** `changeId` 不是按整台 VM 保存，而是按每块磁盘保存。第一次基线同步时，要把磁盘 key、snapshot 引用和 `changeId` 一起记录下来，后续增量才有对比基准。

- **第四步：查询 changed extents** `QueryChangedDiskAreas(...)` 会返回某块磁盘在两个时间点之间的变化范围。你可以按磁盘、按 offset 分页拿到变化区间，再只处理这些区间。


### 🧠 QueryChangedDiskAreas：CBT 的查询输出长什么样

- **输出对象** `QueryChangedDiskAreas` 返回的是 `DiskChangeInfo`，里面包含查询区间、变更区间列表，以及用于下次继续增量的状态信息。

- **增量逻辑** 第一次调用时常用 `"*"` 获取已分配区域；后续调用则传入上次保存的 `changeId`，查询自该标识以来发生变化的区域。

- **JSON 样例** 典型输出可以理解成下面这样：

    ```json
    {
      "startOffset": 0,
      "length": 10737418240,
      "changedArea": [
        {
          "startOffset": 1048576,
          "length": 65536
        },
        {
          "startOffset": 5242880,
          "length": 131072
        }
      ]
    }
    ```

- **样例含义** 这表示查询从 0 字节开始，覆盖 10GB 范围，结果里返回了两段连续变化区域，备份程序只需要读取这些 extent，而不需要把整盘重新扫一遍。

- **查询边界的意义** 它输出的不是“文件级别”的差异，也不是“应用级别”的差异，而是最底层的磁盘区间差异。对于上层程序来说，这已经足够再往下映射成读取、同步、导入或写盘动作。


### 🧠 面向 VMware → FusionCompute 的推荐路线

- **第一阶段：全量基线** 先创建 CBT 并建快照，全量读盘，把初始数据同步到目标端，同时保存每块盘的 `changeId`。

- **第二阶段：多轮增量** 每轮都重新建快照，再调用 `QueryChangedDiskAreas` 拿 changed extents，只同步变化块，持续缩小两端差异。

- **第三阶段：最终切换** 停机后做最后一次增量，收口完成后在目标平台拉起虚机。

### CBT 只是块变化，不能拉数据

CBT 只告诉我们“哪些块变了”，但没有直接告诉我们“该从哪个已被官方验证的数据源，把这些块按快照时刻正确读出来”
