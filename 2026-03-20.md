## 接口性能排查的基本思路

当面临接口性能问题时，可以按照以下维度进行系统性排查与分析：

1. **分段测时**：把一次 CRUD 拆解为网关/接口层、业务层、DAO/SQL、数据库执行、响应序列化，确定具体的瓶颈在哪一段。
2. **分析库表设计**：查表这几个大字段的类型，是否有 JSON/TEXT 大列，单条平均大小、每个字段的数据有多大。
3. **审视更新方式**：Update 是否每次都整行覆盖大字段，哪怕只改名字也把整份拓扑/IP 规划重写一次；这会拖慢 DB 和序列化。
4. **梳理删除链路**：Delete 是否做了级联删除、软删后还有额外清理、审计、异步任务等待。
5. **优化接口返回设计**：Create/Update/Get/List 的返回体是否都把全部字段（包括大字段）都返回给前端。如果前端其实不需要那么多字段，尤其是大字段，返回裁剪后的轻量结构会立刻见效（例如曾实现从 1.33s 降到了 13ms）。数据库避免“全字段查询”，查某些不需要的列，尤其是很大的字段，改成“只查需要的列”。

## 真实慢样本的定位与网络 Timing 分析

具体的排查定位方法如下：
1. 选一个最慢的接口， 优先 ListProject，其次 GetProject，再看 UpdateProject。
2. 抓取真实慢样本：包括一次请求参数、返回耗时、数据规模。排查查了几条，单条数据的字段多大。
3. 在浏览器开发者工具的 Network 面板点击该接口，查看：
   * **Timing 标签页**：分析 `Waiting for server response` (TTFB 耗时) 和 `Content Download` (内容下载耗时)。
   * **Preview 标签页**：查看每条数据字段，点击 `Show more` 查看具体字段的字节大小。

## 详情接口大字段的拆分与延迟加载方案

详情接口（如 Get 接口）承担的是完整详情的职责，从职责上看，返回大字段本身不算设计错误，这里的“重”是有业务合理性的。

优化的核心不是“去掉字段”，而是“同样要这些字段，如何避免一次性传输过大”：
1. **拆分接口（延迟加载）**：如果大字段虽然页面会用到，但不是“首屏立刻需要”，就拆成延迟加载。从“单接口全返回”，拆成“基础字段 Get 接口”和“大字段单独按需 Get 接口”。
2. **启用压缩传输**：如果要传输大体积 of JSON，确保服务端开启了 `Gzip` 或 `Brotli` 压缩，通常能显著降低网络传输时间。

## 传输压缩（Gzip/Brotli）的验证与调优

### 如何判断是否开启了传输压缩？
1. **检查浏览器响应头**：
   * 请求头包含：`Accept-Encoding: gzip, deflate, br`
   * 响应头包含：`Content-Encoding: gzip` 或 `br`
   只有响应头中包含 `Content-Encoding` 才说明实际应用了压缩。
2. **查看 Network 面板的大小显示**：
   如果浏览器显示的“传输大小”（Transferred Size）明显小于“资源大小”（Resource Size），通常说明压缩生效了。

### 压缩效果的验证基线：
记录以下基线数据进行对比：
- `Time` (总耗时)
- `Waiting (TTFB)` (等待服务器响应时间)
- `Content Download` (内容下载时间)
- `Size` (包体积)

### 验证逻辑：
1. 若只有 `Content Download` 明显下降，说明网络压缩生效（例如数据从 7137KB 降到了 263KB，总时间从 1.37s 降到了 740ms），传输瓶颈被打掉，但服务端瓶颈仍在。
2. 若 `Waiting (TTFB)` 仍然很高，说明服务端序列化/查询仍然很重，这 636ms 基本在服务端处理阶段（数据库读取、大 JSON 从 MySQL 取出、Go 对象构造、序列化、Gzip 压缩等）。
3. 服务端重启后，重点验证以下响应头：
   * `Content-Encoding: gzip`
   * `Vary: Accept-Encoding`

## 服务端查询与序列化瓶颈排查（日志埋点）

若网络传输不是主瓶颈，主要时间耗在服务端处理阶段。应当进行代码打点日志：

```go
start := time.Now()
// 数据库执行或业务处理
r.log.WithContext(ctx).Infof("ProjectRepo.Get: project_uuid=%s db_query_ms=%d", uuid, time.Since(start).Milliseconds())
```

- 在接口入口和返回前打日志，以获取整个接口耗时。
- 在执行 SQL 的代码段两端打日志，以获取具体的数据库查询耗时。
- 通过开启 MySQL 慢查询日志、`EXPLAIN`，或者在客户端手动执行 SQL 确认耗时。
- 比对时间：
  * 若 `db_query_ms` 接近 `total_ms`，说明慢在数据库查询或 ORM 取数。
  * 若 `db_query_ms` 明显小于 `total_ms`，说明慢在应用层（如反序列化、CPU 密集处理）。

## 数据库慢 SQL 与整行读取瓶颈优化

例如，通过日志排查分析出慢 SQL 痕迹：
```log
2026/03/20 10:21:11 /var/jenkins_home/workspace/216-network-config-generator/internal/data/project.go:213 SLOW SQL >= 200ms
[580.087ms] [rows:1] SELECT * FROM `projects` WHERE uuid = '4b854540-6452-41d2-a311-f488e52b8bea' AND `projects`.`deleted_at` IS NULL ORDER BY `projects`.`id` LIMIT 1
INFO ... msg=ProjectRepo.Get: project_uuid=... db_query_ms=580
INFO ... msg=GetProject: project_uuid=... total_ms=580
INFO ... latency=0.580847965
```

这说明：
1. `db_query_ms`、`total_ms` 和 HTTP 接口总延迟高度接近（均为 580ms），说明瓶颈 100% 在数据库查询。
2. 根本原因是使用了 `SELECT *` 导致把 `projects` 表里的超大字段（如 `ipPlanningTable` JSON，大小达 6MB+）整行拉了出来，导致 MySQL 在读取超大记录本身时耗时巨大。
3. **优化方案**：
   * 将大字段（如 `ipPlanningTable`）从主表拆分出去，放入单独的扩展表。
   * 基础项目查询只读主表的瘦记录，需要大字段时再单独查扩展表。

## XHR 请求重放与前端请求压缩优化

1. **重新测试接口**：在浏览器 Network 面板找到对应的 XHR 请求，右键选择 `Replay XHR`，即可在保持相同上下文的情况下重新发送请求并快速测试接口表现。
2. **请求体积优化**：如果不仅返回包大，客户端发送的请求体也过大，可以在前端开启 `Gzip` 请求压缩，配合后端解压接收。
