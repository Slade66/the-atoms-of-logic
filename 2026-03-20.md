# 如何排查接口的性能问题？

排查思路：

  1. 先分段测时
     把一次 CRUD 拆成：网关/接口层、业务层、DAO/SQL、数据库执行、响应序列化，先确定慢在哪一段。
  2. 看库表设计
     查表这几个大字段的类型，是否有 JSON/TEXT 大列，单条平均大小、每个字段的数据有多大。
  3. 看更新方式
     Update 是否每次都整行覆盖大字段，哪怕只改名字也把整份拓扑/IP 规划重写一次；这会拖慢 DB 和序列化。
  4. 看删除链路
     Delete 是否做了级联删除、软删后还有额外清理、审计、异步任务等待。
  5. 看接口返回设计是否合理
     Create/Update/Get/List 的返回体是否都把全部字段（包括大字段）都返回给前端。如果前端其实不需要那么多字段，尤其是大字段，返回裁剪后的轻量结构会立刻见效。

     从 1.33s 降到了 13ms。

     数据库是否是“全字段查询”？查了某些不需要的列，尤其是很大的字段，改成“只查需要的列”

具体的排查方法：
1. 选一个最慢的接口， 优先 ListProject，其次 GetProject，再看 UpdateProject。
2. 真实慢样本：一次请求参数、返回耗时、数据规模。查了几条，单条数据的字段多大。
在浏览器，在network面板点击接口，在右侧的 Timing 标签里
把 wating for server response 的耗时
和 content download 的耗时
还有 Preview 标签里可以看到每条数据字段有个 Show more，会显示字段的大小。


### Get 接口返回大怎么办？
“详情接口”承担的是完整详情职责，从职责上看，这是合理的，它和 ListProject 不同，返回大字段本身不算设计错误，这里“重”是有业务合理性的。

不是“去掉字段”，而是“同样要这些字段，怎么别一次性传那么大？
1. 拆接口：很多时候是“页面最终会用到”，但不是“首屏立刻要”。如果不是首屏必需，就拆成延迟加载。不要一个 Get 扛所有大字段，从单接口全返回，拆成基础字段 Get 和 大字段单独按需 Get。
2. 启用压缩传输：如果要传大 JSON，至少确认服务端开了 gzip/br 压缩，往往立刻降很多传输时间。

#### 如何判断有没有开启传输压缩？

 看浏览器响应头，检查：
      - 请求头有没有 Accept-Encoding: gzip, br
      - 响应头有没有 Content-Encoding: gzip 或 br
     有 Content-Encoding 才说明实际压缩了。

 看 Network 的大小显示
     如果浏览器显示类似“传输大小明显小于资源大小”，通常就是压缩了。

是否起效？
记录当前基线
 记录这几个值
      - Time
      - Waiting (TTFB)
      - Content Download
      - Size
看效果
      - Content Download 是否明显下降
      - Size 是否明显下降
      - 总耗时是否下降
 1. 如果只有 Content Download 明显下降，说明压缩生效了
  2. 如果 Waiting (TTFB) 仍高，说明服务端序列化/查询仍重
  3. 如果总耗时下降有限，说明后续还得继续拆 GetProject
重启服务后，重点验证这两条接口的响应头：
  - Content-Encoding: gzip
  - Vary: Accept-Encoding


从 1.37s 降到了 740ms。
从 7137kb 降到了 263kb。
说明你已经把“传输瓶颈”打掉了，但“服务端瓶颈”还在。
- 原来慢 = 服务端处理 + 网络传输
- 现在你只优化掉了网络传输，网络传输不是主瓶颈了
- 服务端处理时间几乎还在， 主要耗时在服务端生成响应。

结论很明确：
 1. 压缩非常有效
     网络传输已经不是主要矛盾了。
  2. 现在主要慢在服务端
     这 636 ms 基本都在服务端处理阶段：
      - 数据库读取
      - 大 JSON 从 MySQL 取出
      - Go 侧对象构造
      - JSON 序列化
      - gzip 压缩本身也会占一点 CPU
  3. 所以总耗时不会按体积同比例下降
     因为原来那 1.37 s 里，不全是下载时间。

3. 真正慢的是这一条 SQL：

而且这条 SQL 的关键信号是：

优化方案： 
前端不是每次首屏都需要全部大字段
把超大字段拆出去，把大字段从主表拆去扩展表。
 接口改成分段加载：
- 查基础项目信息时只读瘦表
- 需要大字段时再单独查
“这条 UUID 对应的记录太胖，SELECT * 把 6MB+ 大对象整行拉出来了”。

当前瓶颈已坐实是 projects 表里超大 JSON，尤其 ipPlanningTable，被 GetProject 整行读取。
