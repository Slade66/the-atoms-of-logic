### 🧠 什么是规则（Rules）

**是什么**：规则是你事先写好的一套要求，告诉 Agent 在这个项目里（或所有项目里）该怎么做事、不该做什么。你可以在规则文件中写明具体的上下文背景和行为准则，Agent 会在所有对话中自动遵守它们。

**有什么限制**：规则以 Markdown 文本文件的形式保存，但每个文件的内容不能超过 `12000` 个字符。

一个典型的规则文件看起来是这样的：
```markdown
1. 我的构建系统是 Bazel
2. 我的测试框架是 pytest
3. 不要修改 /legacy 目录下的任何文件
4. 不要使用已废弃的 v1 API
```

### 🧠 规则的四种激活模式

每条规则都可以单独配置触发时机，而不是一股脑对所有对话生效。

| 模式 | 触发方式 | 适用场景 |
|------|----------|----------|
| **Manual（手动）** | 在输入框中用 `@规则名` 主动唤起 | 临时性、按需调用的参考规则 |
| **Always On（常驻）** | 对话全程自动加载，无需任何触发 | 语气、格式、语言等始终生效的基础行为准则 |
| **Model Decision（模型判断）** | 你写一段自然语言描述，由模型自行决定是否激活 | 语义范围模糊、需要上下文感知的条件性规则 |
| **Glob（文件匹配）** | 按你定义的 glob 模式（如 `*.js`、`src/**/*.ts`）自动匹配文件 | 针对特定文件类型或目录的专属编码规范 |

### 🧠 全局规则 (Global) 与 工作区规则 (Workspace)

**全局规则（Global Rules）：** 对你打开的**所有项目**都生效。存放在 `~/.gemini/GEMINI.md`。

**工作区规则（Workspace Rules）：** 只在**当前项目**内生效。存放在项目根目录下的 `.agent/rules/` 文件夹中。

❌ **错误做法：把只属于某个项目的规则，写进了全局配置**
```bash
# 危险：把某个项目专用的要求塞进全局配置文件
# 后果：以后打开其他任何项目，Agent 都会莫名其妙地遵守这条旧规则
cat "必须使用 React 16 类组件语法" > ~/.gemini/GEMINI.md
```

✅ **正确做法：全局只放通用规则，项目专属规则放进工作区**
```bash
# 全局只放对所有项目都适用的规则（比如回复的格式、语气）
~/.gemini/GEMINI.md

# 把只属于这个项目的规则，单独存放在工作区目录下
.agent/rules/using-typescript.md
```

### 🧠 编辑规则

1. **唤起面板**：点击智能体面板右上角的 `...` 按钮，选择 Customizations。
2. **进入配置**：在导航菜单中切换至 Rules 标签页。
3. **按需创建**：
   - 点击 + Global：建立适用于所有项目的通用准则。
   - 点击 + Workspace：建立仅在当前项目文件夹下生效的特定规则。

### 🧠 @ 文件动态引用机制 

在规则文件中，你可以使用 `@文件路径` 引用其他文件，类似编程中的 `import`，避免将内容大段复制粘贴。

1. **相对路径**：以当前规则文件所在位置为基准进行解析。
2. **绝对路径**：优先解析为硬盘上的真实路径；若文件不存在，则回退至以仓库根目录为起点重试。

❌ **灾难做法：散落在各处的祖传复制粘贴**
```markdown
（冗长地复制了一段长达100行的公司级底层安全审核指南）
此处必须做好SQL漏洞排查...（由于原始源变更，产生信息差灾难）
```

✅ **优雅做法：维持系统唯一可信源的零拷贝架构**
```markdown
进行当前逻辑操作时，你必须一并校验以下安全基线：
@./security-baseline.md
```
