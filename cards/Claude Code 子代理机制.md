# Claude Code 子代理机制

### 🧠 主对话的上下文污染困境

Claude Code 的主对话上下文是线性追加的，不会自动过期。当你让它跑测试、扫描日志、搜索代码库时，会产生大量中间输出——几百行测试日志、上千行搜索结果、复杂的错误堆栈。这些对"当下执行"必要、但对"后续决策"毫无价值的信息，会被当成长期记忆保留，最终把主对话淹没在细节里。

### 🧠 核心定义：什么是子代理

子代理是一种职责拆分机制——主代理是指挥官，子代理是专岗执行者。每个子代理带着明确的任务、受限的权限、独立的上下文窗口出去干活，完成后只把结论带回来。

用公司来类比：老板不会亲自跑测试（300 行日志）、亲自扫日志（500 行堆栈）、亲自搜代码（成百上千行 grep 输出），然后脑子里塞满细节，记不清最初的目标。他会安排专人去做，每个人只交一份结论报告。子代理就是这些专人。

它的三个核心特征：

1. **独立上下文窗口：** 每个子代理拥有自己的上下文，执行过程中的中间噪声不会回流到主对话。

2. **受限的工具权限：** 你可以精确控制子代理能用哪些工具，比如只给 `Read` 和 `Grep`，让它只能看、不能改。

3. **结论回传：** 子代理完成任务后，只把结果摘要带回主对话，中间过程即丢弃。

```text
主 Claude: 这个任务需要跑大量测试，让我创建一个子代理来处理。
    ↓
子代理（test-runner）: 执行测试，只把结果汇报给主 Claude
    ↓
主 Claude: 收到"3 个测试失败"，决定下一步行动（保持上下文纯净）
```

### 🧠 工作原理：上下文隔离 + 工具权限约束

子代理通过两种机制发挥作用：

1. **上下文隔离：** 子代理拥有独立的上下文窗口。执行过程中的高噪声信息（日志、搜索结果、中间推理）被隔离在子代理内部，执行完即丢弃，只把结论带回主对话。

2. **工具权限边界：** 子代理可以被精确控制工具权限。比如你让 Claude 审查代码，只希望它看、不希望它动手改。没有权限边界时，Claude 可能在审查过程中顺手"修"了一些它认为有问题的代码。有了子代理，你可以创建一个 `code-reviewer` 角色，明确规定它只有 `Read`、`Grep` 权限——再怎么想插手，也改不了任何东西。

### 🧠 配置字段说明

- **`description` — 触发条件：** 决定了 Claude 何时自动调用你的子代理。一个清晰的描述能让 Claude 在合适的场景下自主判断是否需要派出该子代理。

- **`tools` 与 `disallowedTools` — 白名单与黑名单：** 子代理只需少数几个工具时，用 `tools` 白名单（"只能用这些"）；需要大部分工具但想排除个别危险操作时，用 `disallowedTools` 黑名单（"继承所有，但排除这些"）更简洁。

  ```yaml
  tools: Read, Grep, Glob        # 白名单：只读审查场景
  disallowedTools: Write, Edit   # 黑名单：排除写操作
  ```

- **`skills` — 预加载知识：** 允许在子代理启动时，将指定 Skill 的完整内容注入到子代理的上下文中。子代理不需要在执行过程中发现和加载 Skill——知识已经在它的脑子里了。

### 🧠 三类最佳使用场景

1. **高噪声输出的任务：** 执行过程产生大量中间信息，但主对话真正关心的只有一个结论。比如跑测试只想知道通过还是失败，扫描日志只需要一两条关键错误，搜索代码库只需要知道相关文件在哪里。

2. **角色边界必须明确的任务：** 有些事情只希望 Claude "看"而不"动手"。比如代码审查场景，必须严格区分"只读审查"和"修改代码"两个角色。

3. **可以并行展开的任务：** 让多个子代理各自完成自己的探索，再把结果汇总回来。子代理在这里不只是隔离上下文，更是天然的并行加速器。

### 🧠 使用方式

1. **交互式创建：** 在 Claude Code 中输入 `/agents`，按照向导操作即可创建子代理。

2. **前台与后台执行：** Claude 会根据任务自动选择执行模式，也可以手动控制——对 Claude 说 `run this in the background` 让任务后台运行，或按 `Ctrl+B` 将正在运行的前台子代理切到后台。

### 🧠 使用限制

1. **扁平化编排：** 子代理不能生成子代理，所有编排必须由主对话完成。如果你需要"先审查再修复"的流程，必须由主对话依次调用两个子代理，而不是让第一个子代理去调用第二个。
