# OS 跨平台文件命名法则

## 抛出现象：跨系统与部署的碎链灾难
在跨平台同步知识库（如通过 Git 推送至 Linux 服务器，或使用 Hugo 构建博客发布）时，由于随意命名，经常会触发 404 找不到文件、URL 乱码甚至 Git 追踪丢失等诡异报错。

## 核心痛点：操作系统的异构机制冲突
真正的元凶在于：Windows (NTFS)、Linux (Ext4)、macOS (APFS) 底层文件系统的设计哲学存在不可调和的差异。一旦你在文件名中“随心所欲”地使用空格或标点，极易踩中另一系统的保留字或禁区，导致数据无法被完整迁移。

## 具象化灾难：三大操作系统的雷区剖析

### 1. Windows（限制最严苛）
Windows 的历史包袱最重，存在大量被系统底座强占的规则：
- **保留字符**：绝对禁用 `<` `>` `:` `"` `/` `\` `|` `?` `*`。
- **DOS 时代保留字**：禁用设备预留名（如 `CON`, `PRN`, `AUX`, `NUL`, `COM1~9` 等），即使加后缀（如 `NUL.txt`）也会立刻触发底层报错。
- **末尾陷阱**：文件名结尾绝不能留有空格或英文句点（`.`）。
- **大小写不敏感**：`Zap.md` 和 `zap.md` 视为同一物理文件。在双链语法中若滥写大小写，一旦迁移到 Linux，链接将集体失效。

### 2. Linux（宽容引发的 Shell 灾难）
Linux 的规则极度开放，但在工程运维侧反而是重灾区：
- **极其宽松**：底层仅明文禁用正斜杠（`/`）和空字符（`\0`）。
- **Shell 元字符暴雷**：若给文件随意加上空格及 `&`、`*`、`$`、`|` 等特殊符号，那么在终端执行 `cp`、`grep` 或日常批处理时，会引发切词错位或变量注入。排障时必须加上烦琐的转义符（`\`）。
- **大小写绝对敏感**：`a.md` 和 `A.md` 是完全不同的两个文件。

### 3. macOS（暗藏迷惑的兼容层）
macOS 基于 Unix 打造，但为了交互做了种种妥协：
- **Finder 图形界面封锁**：在图形界面由于需向前兼容 Classic Mac OS 的路径规范，强制禁止使用冒号（`:`）。
- **大小写玄学**：默认发行的 APFS 文件系统通常为大小写不敏感，但保留大小写原貌（Case-Insensitive, Case-Preserving），在团队 Git 跨端协同提交时，极易引发幽灵般的修改冲突。

## 解决方案：极简降级的 Kebab-Case
为了保障卡片在任何物理环境与 Web 路由下流通无阻，“降级至最小公约数”是跨端命名的唯一解。推导出的终极防线为短横线命名法（Kebab-Case）极限版。

✅ **核心防御铁律：**
1. **字符白名单**：一生仅限使用中文字符、小写字母 `a-z`、数字 `0-9` 以及连字符（`-`）。
2. **绞杀空格与标点**：意群之间一律用 `-` 断句；禁止任何英文及中文标点符号（如括号、惊叹号等）。
3. **强制全小写**：不管业务侧缩写叫得多响亮（如 API、JSON 等），文件名必须强降为全小写（`api`、`json`），从物理层面永绝异构系统的大小写死链问题。


❌ **灾难做法（含特殊标点、空格、大写，在 URL 中会被转义为恶心的 `%20` 乱码）：**
```text
为什么 Zap 不能使用 Fatalf？(2026 最新版).md
AWS S3 架构设计图 & 思考.md
```

✅ **优雅接法（具备纯正静态化 URL 感，在终端里按 Tab 键可光速补全匹配）：**
```text
zap-没有fatalf的原因-2026最新版.md
aws-s3-架构设计图与思考.md
```
