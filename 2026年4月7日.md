
### 异步长任务模式

为了彻底解耦API接收与后台执行，必须引入分布式任务队列组件 Redis Stream。

系统被划分为API网关层、任务编排微服务、VMware数据提取微服务、FusionCompute交互微服务以及Linux本地系统适配微服务。

当用户通过前端发起针对特定虚拟机的迁移请求时，API层将一个结构化的迁移任务负载（Payload）推入Redis支持的Stream队列中，并立即向客户端返回一个全局唯一的任务ID（Job ID）。

在后台，运行在Linux节点上的Stream Worker集群持续监听Redis队列。Worker提取任务后，严格按照状态机模型推进迁移生命周期，将任务状态从“等待中（PENDING）”逐步更新为“数据同步中（SYNCING）”、“离线适配中（ADAPTING）”直至“已完成（COMPLETED）”，并将实时进度和日志持久化至关系型数据库中，供前端轮询或通过WebSocket推送到用户界面。

### 任务容错机制与死信队列设计

在异构环境之间的数据海量传输过程中，不可避免地会遇到网络抖动（网络带宽拥塞导致的延迟突然变高 / 丢包）、存储I/O拥塞（磁盘 IOPS 上限导致写入变慢）或API限流（服务并发保护机制导致请求失败）等瞬态故障。且时间越长，遇到异常概率越大。这是不可避免的系统现象。

当某个块数据流同步失败时，Worker不会导致整个迁移任务崩溃，而是记录断点，并在指定的延迟后重新尝试执行。

若重试次数达到配置的上限，任务将被安全地转移至死信队列（Dead Letter Queue, DLQ）中，触发告警，等待运维人员的人工干预与分析。

### 下载机制

在获取到需要提取的磁盘区域列表后，平台并不通过缓慢的常规API接口读取数据，而是向ESXi主机申请网络文件拷贝（Network File Copy, NFC）租约。通过调用govmomi中的ExportSnapshot或相应的NFC API，主机返回一个HttpNfcLease对象。当租约进入准备就绪（Ready）状态时，VMware会提供一组直接指向底层VMDK文件的HTTP(S)下载链接。

Worker此时会实例化一个高性能的Go HTTP客户端并发池。根据DiskChangeInfo中指示的变更块偏移量，HTTP客户端在请求头中构造精确的Range: bytes=start-end参数，向NFC端点发起并行的部分内容下载请求。

这些并发的数据流在Go内存中被接收后，直接通过流式处理（Streaming）机制转发至目标端，避免了在Linux迁移节点上产生巨大的本地磁盘I/O和内存溢出瓶颈。

### 必须新建一个占位虚拟机

为承载即将到来的源端数据，平台必须预先在FusionCompute集群中创建一台“占位虚拟机（Placeholder VM）”。此过程涉及对VMware源端虚拟机硬件配置的精准映射。平台向<basic_uri>/sites/<site_id>/vms端点发送POST请求，将提取到的VMware vCPU核心数、内存容量以及网卡MAC地址精确映射到FusionCompute的JSON请求体中。

### 如何往 FC 写碎片数据

通过调用<site_uri>/<site_id>/volumes接口，平台为每个源端VMDK在目标存储上创建等容量的空卷（Volume）。

FusionCompute VRM并未提供类似于VMware NFC的公共HTTP接口用于外部系统直接向虚拟卷写入任意偏移量的数据块。

为突破这一架构限制，本方案在Linux迁移环境内部署了一种“代理直写（Proxy Ingestion）”机制。平台预先在FusionCompute集群中维护一台持续运行的轻量级Linux代理虚拟机。当占位卷创建完成后，Kratos Worker调用VRM API（POST /vm_uri/{vm_id}/action/attachvol），动态将这些尚未包含任何数据的目标卷以第二磁盘（/dev/vdb, /dev/vdc等）的形式挂载到代理虚拟机上。随后，Go Worker与代理虚拟机建立持久化的gRPC长连接。当从VMware端并发下载的变更数据块到达Go Worker时，这些数据块携带原始偏移量信息通过gRPC流式传输至代理虚拟机内部，由代理机内部的守护进程直接利用底层块设备接口（如Linux的pwrite或dd系统调用），将数据严格按照字节偏移量写入对应的目标裸设备中。这一创新性架构彻底绕过了文件系统层面的限制，实现了最高效的数据流转。

### 异构 Hypervisor 操作系统的离线自动化适配（驱动注入）

- 将承载在VMware底层虚拟化硬件上的系统磁盘直接转移至KVM环境并强制启动，必将导致灾难性的后果。

- 针对Windows Server 2022 DataCenter与Red Hat Enterprise Linux (RHEL) 7.9两类目标操作系统，实施底层的离线驱动注入与内核重构，以消除底层虚拟化硬件抽象层（从VMware虚拟主板到KVM虚拟主板）切换导致的蓝屏（BSOD）或内核崩溃（Kernel Panic）问题。

- VMware默认向客户机呈现LSI Logic SAS存储控制器与VMXNET3虚拟网卡；而华为FusionCompute优化后的KVM架构主要依赖于VirtIO半虚拟化接口。

- 如果不进行驱动干预，Windows Server 2022将因无法识别系统盘所在的存储控制器而蓝屏崩溃；而RHEL 7.9的启动内核则会卡死在dracut紧急求救终端，抛出无法挂载根文件系统的致命错误。

- 为了在Linux迁移平台上实现完全自动化、无需人工干预的驱动转换，深度集成基于libguestfs套件的virt-v2v。在割接完成且目标磁盘从代理机卸载后、正式挂载至目标占位虚拟机之前，Linux平台对这些裸盘镜像执行底层的离线修改。libguestfs通过启动一个微型的安全Linux Appliance，将目标磁盘作为块设备挂载，从而在不需要引导目标操作系统的前提下，直接读取和修改其内部的注册表、配置文件及文件系统树。

#### 如何迁移 Windows

针对Windows体系，操作的核心在于将Red Hat签名的virtio-win驱动族（特别是vioscsi.sys存储驱动、netkvm.sys网络驱动以及balloon.sys内存气球驱动）强行植入离线的系统盘，并重构Windows的早期启动逻辑。

适配步骤：
1. 驱动文件转移：从迁移平台内置的virtio-win.iso介质中，提取匹配Windows Server 2022内核架构（amd64\2k22）的驱动二进制文件，直接拷贝至虚拟磁盘的%SystemRoot%\System32\Drivers与DriverStore目录中。

2. 注册表离线修改：仅拷贝文件无法解决引导问题，因为Windows引导装载程序（Bootmgr）只会加载注册表中被标记为关键启动设备的服务。工具通过挂载\Windows\System32\config\SYSTEM配置单元，解析并修改离线注册表。

3. 关键设备注入：平台将VirtIO SCSI与Block控制器的PCI Vendor ID（例如VEN_10AF）写入注册表的CriticalDeviceDatabase键值树下。这欺骗了Windows即插即用（PnP）管理器，使其在下一次探测到KVM硬件时认为该设备是安全且已授权的。

4. 服务自启动设置：在ControlSet001\Services\vioscsi下创建对应的服务键值，并将Start属性强制设为0（Boot Start）。此举确保Windows在尝试装载C盘NTFS文件系统之前，优先将VirtIO SCSI驱动加载至内核空间，彻底规避0x0000007B蓝屏宕机。

若原VMware环境安装了VMware Tools服务，适配程序还将在注册表中注入预留的RunOnce清理脚本，确保虚拟机首次在FusionCompute上线后，自动静默卸载VMware遗留的驱动与代理守护进程，防止与华为的Guest Tools产生冲突。

#### 如何迁移 Linux

与Windows不同，Red Hat Enterprise Linux 7.9的发行版内核已经内建了对KVM的全面支持（包含virtio_blk、virtio_scsi及virtio_net等模块）。

然而，问题的核心在于RHEL的初始内存文件系统（initramfs）。如果该虚拟机是在VMware上原生安装的，系统的dracut生成器在构建initramfs时，为了精简体积，只会将当时必需的VMware PVSCSI或mptspi驱动打包进去。当该系统被强制转移至KVM平台时，Bootloader加载内核后，内核解压initramfs，却发现其中没有任何能够驱动VirtIO SCSI总线的模块，导致系统彻底失去对/dev/vda或/dev/sda的访问能力，从而进入紧急模式。

适配步骤：
1. 根环境挂载： libguestfs挂载目标磁盘的逻辑卷（LVM）或标准分区，定位到根目录（/）及引导分区（/boot），利用内部环境执行chroot进入RHEL 7.9系统的实际上下文空间。

2. 强制重构initramfs： 平台在此隔离环境中调用操作系统的原生内核管理工具，执行带有强制注入标志的命令：dracut -f --add-drivers "virtio virtio_pci virtio_block virtio_net virtio_scsi" /boot/initramfs-$(uname -r).img $(uname -r)。该指令迫使系统解压现有的引导镜像，将KVM所需的半虚拟化驱动强行编译并打包进新的initramfs中，确保Linux内核在启动的最早期阶段具备读取虚拟化SCSI控制器的能力。

3. 网络配置清理：由于底层网卡MAC地址的变更，Linux传统的udev设备管理器会将新检测到的VirtIO网卡重命名为eth1或乱序名称，导致原有绑定在eth0上的静态IP配置文件失效。适配程序通过直接删除/etc/udev/rules.d/70-persistent-net.rules等持久化网络规则文件，并利用正则表达式修改/etc/sysconfig/network-scripts/ifcfg-*脚本，剥离硬编码的MAC地址校验。这使得系统在新平台启动时，能够正确地将新生成的VirtIO网卡与原始网络配置文件相匹配，保障网络连通性的瞬间恢复。

### 在线迁移的设计方案

- 结合VMware CBT底层块级流式传输技术、再辅以Linux环境强大的脱机libguestfs内核重构能力的跨架构在线迁移解决方案。

- 第一阶段为全量基线同步。此时源端Windows或RHEL虚拟机处于完全正常的业务运行状态。平台后台利用CBT全量位图提取所有已分配数据，通过代理直写机制源源不断地写入FusionCompute的目标卷。尽管此过程耗时最长，但由于完全在后台进行，前端业务不会受到任何实质性阻断。

- 第二阶段为迭代增量同步。基线同步完成后，源端虚拟机在此期间必定产生新的业务数据写入。平台触发新一轮的VMware快照，并利用前一次快照的changeId执行QueryChangedDiskAreas。VMware仅返回在这段时间内发生修改的增量数据块。Go Worker迅速提取这些极小的数据增量并同步至目标端。随着增量周期的不断迭代，单次同步所需传输的数据量呈指数级下降，同步耗时逐渐从数小时缩减至几分钟甚至几十秒，平台据此进入高度同步的“就绪（Ready to Cutover）”状态。

- 第三阶段为最终割接（Cutover），也是整个生命周期中唯一需要业务停机的短暂窗口。在与业务方约定的维护时间点，平台通过govmomi下发指令，暂时挂起（Suspend）或关机（Power Off）源端的VMware虚拟机，确保所有内存中的脏数据刷入磁盘。随后，平台执行最后一次终极增量同步。因为此时虚拟机不再产生新数据，这次同步将彻底消除两端的数据差异，实现100%的字节级一致性。完成最终同步后，目标卷从代理虚拟机上剥离（Detach），准备进入最为关键的离线操作系统适配阶段。

- 给系统注入驱动、修改启动逻辑。

- 第五阶段：调用华为FusionCompute VRM的API，向目标占位虚拟机发出磁盘绑定请求（POST /vm_uri/{vm_id}/action/attachvol）。在此过程中，系统严格按照源端VMware的磁盘顺位，通过精确控制sequenceNum参数，将经过完美改造的系统卷与数据卷依次挂载到目标虚拟机对应的SCSI或VIRTIO总线上。

- 第六阶段：最后，API触发开机动作（POST /vm_uri/{vm_id}/action/start）。

- 即使面对最极端的适配失败场景，回滚机制也仅仅需要将源端处于关机（Power Off）或挂起状态的VMware虚拟机重新启动，业务便可瞬间回溯至安全状态。
