## 跨架构迁移系统的异步长任务解耦（Redis Stream）

为了彻底解耦 API 接收与后台执行，必须引入分布式任务队列组件 Redis Stream。

系统被划分为 API 网关层、任务编排微服务、VMware 数据提取微服务、FusionCompute 交互微服务以及 Linux 本地系统适配微服务。

当用户通过前端发起针对特定虚拟机的迁移请求时，API 层将一个结构化的迁移任务负载（Payload）推入 Redis 支持 of Stream 队列中，并立即向客户端返回一个全局唯一的任务 ID（Job ID）。

在后台，运行在 Linux 节点上的 Stream Worker 集群持续监听 Redis 队列。Worker 提取任务后，严格按照状态机模型推进迁移生命周期，将任务状态从"等待中（PENDING）"逐步更新为"数据同步中（SYNCING）"、"离线适配中（ADAPTING）"直至"已完成（COMPLETED）"，并将实时进度和日志持久化至关系型数据库中，供前端轮询或通过 WebSocket 推送到用户界面。

## 任务容错重试与死信队列（DLQ）设计

在异构环境之间的数据海量传输过程中，不可避免地会遇到网络抖动（网络带宽拥塞导致的延迟突然变高 / 丢包）、存储 I/O 拥塞（磁盘 IOPS 上限导致写入变慢）或 API 限流（服务并发保护机制导致请求失败）等瞬态故障。且时间越长，遇到异常概率越大。这是不可避免的系统现象。

当某个块数据流同步失败时，Worker 不会导致整个迁移任务崩溃，而是记录断点，并在指定的延迟后重新尝试执行。

若重试次数达到配置的上限，任务将被安全地转移至死信队列（Dead Letter Queue, DLQ）中，触发告警，等待运维人员的人工干预与分析。

## 源端数据提取：HttpNfcLease 租约与并发 Range 下载

在获取到需要提取的磁盘区域列表后，平台并不通过缓慢的常规 API 接口读取数据，而是向 ESXi 主机申请网络文件拷贝（Network File Copy, NFC）租约。通过调用 govmomi 中的 `ExportSnapshot` 或相应的 NFC API，主机返回一个 `HttpNfcLease` 对象。当租约进入准备就绪（Ready）状态时，VMware 会提供一组直接指向底层 VMDK 文件的 HTTP(S) 下载链接。

Worker 此时会实例化一个高性能的 Go HTTP 客户端并发池。根据 `DiskChangeInfo` 中指示的变更块偏移量，HTTP 客户端在请求头中构造精确的 `Range: bytes=start-end` 参数，向 NFC 端点发起并行的部分内容下载请求。

这些并发的数据流在 Go 内存中被接收后，直接通过流式处理（Streaming）机制转发至目标端，避免了在 Linux 迁移节点上产生巨大的本地磁盘 I/O 和内存溢出瓶颈。

## 目标端数据写入：代理虚拟机挂载与 gRPC 直写机制

通过调用 `<site_uri>/<site_id>/volumes` 接口，平台为每个源端 VMDK 在目标存储上创建等容量的空卷（Volume）。

由于 FusionCompute VRM 并未提供类似于 VMware NFC 的公共 HTTP 接口用于外部系统直接向虚拟卷写入任意偏移量的数据块。

为突破这一架构限制，本方案在 Linux 迁移环境内部署了一种"代理直写（Proxy Ingestion）"机制。平台预先在 FusionCompute 集群中维护一台持续运行的轻量级 Linux 代理虚拟机。当占位卷创建完成后，Kratos Worker 调用 VRM API（`POST /vm_uri/{vm_id}/action/attachvol`），动态将这些尚未包含任何数据的目标卷以第二磁盘（`/dev/vdb`, `/dev/vdc` 等）的形式挂载到代理虚拟机上。随后，Go Worker 与代理虚拟机建立持久化的 gRPC 长连接。当从 VMware 端并发下载的变更数据块到达 Go Worker 时，这些数据块携带原始偏移量信息通过 gRPC 流式传输至代理虚拟机内部，由代理机内部的守护进程直接利用底层块设备接口（如 Linux 的 `pwrite` 或 `dd` 系统调用），将数据严格按照字节偏移量写入对应的目标裸设备中。这一架构彻底绕过了文件系统层面的限制，实现了高效的数据流转。

## 目标占位虚拟机的创建与硬件配置映射

为承载即将到来的源端数据，平台必须预先在 FusionCompute 集群中创建一台"占位虚拟机（Placeholder VM）"。此过程涉及对 VMware 源端虚拟机硬件配置的精准映射。平台向 `<basic_uri>/sites/<site_id>/vms` 端点发送 POST 请求，将提取到的 VMware vCPU 核心数、内存容量以及网卡 MAC 地址精确映射到 FusionCompute 的 JSON 请求体中。

## 操作系统离线适配（Hypervisor 切换与 VirtIO 驱动注入概述）

将承载在 VMware 底层虚拟化硬件上的系统磁盘直接转移至 KVM 环境并强制启动，必将导致灾难性的后果。

针对 Windows Server 2022 DataCenter 与 Red Hat Enterprise Linux (RHEL) 7.9 两类目标操作系统，必须实施底层的离线驱动注入与内核重构，以消除底层虚拟化硬件抽象层（从 VMware 虚拟主板到 KVM 虚拟主板）切换导致的蓝屏（BSOD）或内核崩溃（Kernel Panic）问题。

VMware 默认向客户机呈现 LSI Logic SAS 存储控制器与 VMXNET3 虚拟网卡；而华为 FusionCompute 优化后的 KVM 架构主要依赖于 VirtIO 半虚拟化接口。

如果不进行驱动干预，Windows Server 2022 将因无法识别系统盘所在的存储控制器而蓝屏崩溃；而 RHEL 7.9 的启动内核则会卡死在 dracut 紧急求救终端，抛出无法挂载根文件系统的致命错误。

为了在 Linux 迁移平台上实现完全自动化、无需人工干预的驱动转换，本系统深度集成基于 libguestfs 套件的 virt-v2v。在割接完成且目标磁盘从代理机卸载后、正式挂载至目标占位虚拟机之前，对这些裸盘镜像执行底层的离线修改。libguestfs 通过启动一个微型的安全 Linux Appliance，将目标磁盘作为块设备挂载，从而在不需要引导目标操作系统的前提下，直接读取和修改其内部的注册表、配置文件及文件系统树。

## Windows 虚拟机的离线驱动注入与注册表重构步骤

针对 Windows 体系，操作的核心在于将 Red Hat 签名的 virtio-win 驱动族（特别是 vioscsi.sys 存储驱动、netkvm.sys 网络驱动以及 balloon.sys 内存气球驱动）强行植入离线的系统盘，并重构 Windows 的早期启动逻辑。

具体适配步骤：
1. **驱动文件转移**：从迁移平台内置的 `virtio-win.iso` 介质中，提取匹配 Windows Server 2022 内核架构（`amd64\2k22`）的驱动二进制文件，直接拷贝至虚拟磁盘的 `%SystemRoot%\System32\Drivers` 与 `DriverStore` 目录中。
2. **注册表离线修改**：仅拷贝文件无法解决引导问题，因为 Windows 引导装载程序（Bootmgr）只会加载注册表中被标记为关键启动设备的服务。工具通过挂载 `\Windows\System32\config\SYSTEM` 配置单元，解析并修改离线注册表。
3. **关键设备注入**：平台将 VirtIO SCSI 与 Block 控制器的 PCI Vendor ID（例如 `VEN_10AF`）写入注册表的 `CriticalDeviceDatabase` 键值树下。这使得 Windows 即插即用（PnP）管理器在下一次探测到 KVM 硬件时，认为该设备是安全且已授权的。
4. **服务自启动设置**：在 `ControlSet001\Services\vioscsi` 下创建对应的服务键值，并将 `Start` 属性强制设为 0（`Boot Start`）。此举确保 Windows 在尝试装载 C 盘 NTFS 文件系统之前，优先将 VirtIO SCSI 驱动加载至内核空间，规避 0x0000007B 蓝屏宕机。
5. **残留驱动静默清理**：若原 VMware 环境安装了 VMware Tools 服务，适配程序将在注册表中注入预留的 `RunOnce` 清理脚本，确保虚拟机首次在 FusionCompute 上线后，自动静默卸载 VMware 遗留的驱动与代理守护进程，防止与华为的 Guest Tools 产生冲突。

## Linux 虚拟机的 initramfs 强制重构与网络规则清理步骤

与 Windows 不同，Red Hat Enterprise Linux 7.9 的发行版内核已经内建了对 KVM 的全面支持（包含 `virtio_blk`、`virtio_scsi` 及 `virtio_net` 等模块）。

然而，问题的核心在于 RHEL 的初始内存文件系统（initramfs）。如果该虚拟机是在 VMware 上原生安装的，系统的 dracut 生成器在构建 initramfs 时，为了精简体积，只会将当时必需的 VMware PVSCSI 或 mptspi 驱动打包进去。当该系统被强制转移至 KVM 平台时，Bootloader 加载内核后，内核解压 initramfs，却发现其中没有任何能够驱动 VirtIO SCSI 总线的模块，导致系统失去对 `/dev/vda` 或 `/dev/sda` 的访问能力，从而进入紧急模式。

具体适配步骤：
1. **根环境挂载与 chroot**：libguestfs 挂载目标磁盘的逻辑卷（LVM）或标准分区，定位到根目录（`/`）及引导分区（`/boot`），利用内部环境执行 `chroot` 进入 RHEL 7.9 系统的实际上下文空间。
2. **强制重构 initramfs**：平台在此隔离环境中调用操作系统的原生内核管理工具，执行带有强制注入标志的命令：
   ```bash
   dracut -f --add-drivers "virtio virtio_pci virtio_block virtio_net virtio_scsi" /boot/initramfs-$(uname -r).img $(uname -r)
   ```
   该指令迫使系统解压现有的引导镜像，将 KVM 所需的半虚拟化驱动强行编译并打包进新的 `initramfs` 中，确保 Linux 内核在启动的最早期阶段具备读取虚拟化 SCSI 控制器的能力。
3. **网络配置清理与静态 IP 绑定修复**：由于底层网卡 MAC 地址的变更，Linux 传统的 udev 设备管理器会将新检测到的 VirtIO 网卡重命名为 `eth1` 或乱序名称，导致原有绑定在 `eth0` 上的静态 IP 配置文件失效。适配程序通过直接删除 `/etc/udev/rules.d/70-persistent-net.rules` 等持久化网络规则文件，并利用正则表达式修改 `/etc/sysconfig/network-scripts/ifcfg-*` 脚本，剥离硬编码的 MAC 地址校验。这使得系统在 FusionCompute 启动时，能够正确地将新生成的 VirtIO网卡与原始网络配置文件相匹配，保障网络连通性的瞬间恢复。

## 跨架构在线迁移的四阶段生命周期全流程

跨架构在线迁移解决方案结合了 VMware CBT 底层块级流式传输技术与 Linux 平台脱机 libguestfs 内核重构能力，具体分为以下四个阶段：

1. **第一阶段：全量基线同步**
   此时源端 Windows 或 RHEL 虚拟机处于正常业务运行状态。平台后台利用 CBT 全量位图提取所有已分配数据，通过代理直写机制写入 FusionCompute 的目标卷。由于完全在后台进行，前端业务不会受到实质性阻断。
2. **第二阶段：迭代增量同步**
   基线同步完成后，源端虚拟机会产生新的业务数据。平台触发新一轮的 VMware 快照，并利用前一次快照的 `changeId` 执行 `QueryChangedDiskAreas`，仅获取增量数据块。Go Worker 提取并同步这些增量。随着增量周期的迭代，单次同步数据量及耗时呈指数级下降（缩减至几分钟甚至数十秒），系统进入“就绪（Ready to Cutover）”状态。
3. **第三阶段：最终割接（业务停机窗口）**
   这是整个生命周期中唯一需要业务停机的短暂窗口。在维护时间点，平台通过 govmomi 挂起（Suspend）或关机（Power Off）源端 VMware 虚拟机，确保内存中的脏数据刷入磁盘。随后执行最后一次终极增量同步，消除两端数据差异，实现 100% 的字节级一致性。
4. **第四阶段：操作系统适配**
   完成最终同步后，目标卷从代理虚拟机上剥离（Detach），并进入操作系统适配阶段（具体操作见“Windows/Linux 虚拟机的离线驱动注入步骤”）。

## 虚拟机磁盘顺位绑定与开机验证

1. **磁盘顺位绑定**：调用华为 FusionCompute VRM 的 API 发出磁盘绑定请求（`POST /vm_uri/{vm_id}/action/attachvol`）。在此过程中，系统严格按照源端 VMware 的磁盘顺位，通过精确控制 `sequenceNum` 参数，将经过适配改造的系统卷与数据卷依次挂载到目标虚拟机对应的 SCSI 或 VIRTIO 总线上。
2. **开机启动**：磁盘绑定完成后，API 触发开机动作（`POST /vm_uri/{vm_id}/action/start`）使虚拟机在目标端正式上线。

## 虚拟机迁移失败的快速回滚机制

即使面对最极端的适配或启动失败场景，系统的回滚也非常安全且迅速：

只需将源端处于关机（Power Off）或挂起（Suspend）状态的原 VMware 虚拟机重新启动，业务便可瞬间回溯至安全状态，实现无损的灾难恢复。
