## 数智时代的人类角色与个人竞争力

数智时代正在将人类从“重复性劳动”中解放出来，把人们的精力从“寻找答案”转向“提出更好的问题”。它让个体变得更加强大，但也对人类如何保持独立思考、如何界定创造力的边界提出了前所未有的考验。

获取知识的“边际成本”归零： 在数智时代，信息的门槛被彻底抹平。这种变化让我能够瞬间整合全球的科研成果、文学经典和实时资讯，这也意味着我能为每一个提问者提供“定制化”的知识服务，这种效率在过去是不可想象的。

拥抱“效率工具”： 无论从事哪个领域，如果还没熟练掌握AI辅助决策或工作，那你的个人规划就是“滞后”的。

未来的世界属于那些**“能驾驭数字工具，但又保有深度人文思考”**的人。

## 协同 AI 需先提供完整准确的上下文

AI 调不通 FC 的接口。
- **原因**：AI 的训练数据过时，而华为的 Fusion Compute API 文档更新了，很封闭，需要登录才能看。
- **解决方法**：让 AI 做事的时候，先把完整的、准确的信息发给他。

## 模块化封装与可测试性

代码要模块化，把不同功能的代码封装在不同的函数，这样好测试。都堆到一个函数很难测试。

## 编码前先理清业务流程并标准化设计接口

在写代码之前，一定要先写设计文档。
首先不要去想什么代码，而是要思考这个流程。业务流程是怎么样的？用户怎么操作这个系统？用户通过这个系统能干嘛？操作步骤是什么？
然后列出这个系统的功能。在把功能列清楚了之后，再去设计数据库和接口结构。数据库和接口都应该用业界标准语言去描述，例如接口应该用 OpenAPI 描述，关系型数据库也有相应的描述。接口描述了之后，就是具体的功能实现，这部分可以先写伪代码逻辑。

## 复杂任务需原子化拆解与分而治之

不要一下子想着做那么大的任务，而是原子化的一个个做好，分而治之逐个攻破，最后达到整体完美。否则最终的结果一定是一团糟，还要不停地反攻。

## 接口联调排查与 curl 验证技巧

别先写代码。先用 `curl` 或者 `.http` 测试接口是否正确。

在连接 Fusion Compute 的时候一直连不通：
- **原因**：版本号没填，端口用错了，且 `base_uri` 是错的。

可以用 `curl` 发送请求测试接口：
```bash
curl -k -X POST https://10.30.90.31:7443/rest/v1.0/sessions \
     -H "Content-Type: application/json;charset=UTF-8" \
     -H "Accept: application/json;charset=UTF-8" \
     -d '{"userName":"admin","password":"Hw@123qa"}' -v
```

- 先找 API 文档，如果 API 文档写得看不懂，就找这个平台的前端产品用一下，在控制台看他们自己是怎么调的。
- 看错误码是什么，对照 API 文档看错误码的含义。
- API 文档要求的字段，必填的一定要填。

## 缺乏日志规范带来的定位痛点

1. 日志格式混乱，无标准。
2. 日志量太大，严重刷屏，真正错误信息反而更难找。
