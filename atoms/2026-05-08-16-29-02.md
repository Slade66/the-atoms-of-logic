# 一个数据包进入 Docker 容器的全过程

### 一、核心概括

- **整体流程** 外部请求先到达宿主机网卡，再被 `iptables` 的 DNAT 规则改写目标地址，然后经过路由判断进入 Docker 网桥，最后通过 `veth pair` 到达容器内部网卡。

### 二、数据包进入容器的步骤

- **步骤一：到达宿主机** 外部世界或宿主机上的进程发起请求，数据包先到达宿主机物理网卡，例如 `eth0` 上暴露的宿主机端口；此时目标地址仍然是宿主机 IP 和宿主机端口。

    ```text
    dst = 10.30.90.194:443
    ```

- **步骤二：DNAT 改写** 宿主机内核匹配到 Docker 创建的 `iptables` DNAT 规则后，会把数据包的目标地址从宿主机地址改成容器地址。

    ```text
    修改前：dst = 10.30.90.194:443
    修改后：dst = 172.18.0.3:443
    ```

- **步骤三：记录连接** 内核会把这次地址改写记录到 `conntrack` 连接跟踪表，后续回包才能自动还原成宿主机地址。

    ```text
    1.2.3.4:51000 -> 10.30.90.194:443 <=> 1.2.3.4:51000 -> 172.18.0.3:443
    ```

- **步骤四：重新路由** 目标地址改成容器 IP 后，内核会重新“查地图”：`172.18.0.3` 属于哪张网、应该从哪个网络设备出去；路由表显示 `172.18.0.0/16` 这段容器网络要走 Docker 网桥，所以数据包会进入 Docker 虚拟网络。

    ```text
    172.18.0.0/16 dev br-1b748c70f17b
    ```

- **步骤五：网桥转发** Docker 网桥像虚拟交换机一样工作，会根据 MAC 地址表找到容器对应的 `veth` 设备；`veth pair` 可以理解成一根虚拟网线，一端连在宿主机网桥上，另一端连在容器内部的 `eth0` 上。

    ```text
    br-1b748c70f17b (172.18.0.1)
        |
        +-- vethda98f16 <-> 容器内 eth0 (172.18.0.3)
    ```

- **步骤六：到达服务** 数据包通过 `veth pair` 进入容器网络命名空间，最后到达容器内部服务监听的端口。

- **步骤七：回包还原** 容器内服务返回响应时，回包会从容器的 `eth0` 出来，经过 `veth pair` 回到 Docker 网桥，再回到宿主机网络栈；宿主机内核查询之前记录在 `conntrack` 里的连接信息，把回包源地址从容器地址还原成宿主机地址。

    ```text
    修改前：src = 172.18.0.3:443
    修改后：src = 10.30.90.194:443
    ```

- **步骤八：发回客户端** 地址还原完成后，宿主机通过物理网卡把响应发回外部客户端。
