# Git Rebase

## 一、核心作用

- **重新接线：** `git rebase` 的核心作用，是把当前分支上的提交重新“挪到”另一个分支的最新提交之后，让历史变成一条直线。

- **好处：** 历史线性、清爽，方便阅读和排查问题，没有一堆 “Merge branch...“ 的噪音提交。

## 二、原始问题

### 场景一：本地分支落后于主线

- **功能分支：** 多人开发时，先从 `main` 拉出功能分支 `feature`。

    ```text
    A---B---C  main
         \
          D---E  feature
    ```

- **主线更新：** 后来 `main` 又产生了新的提交，`feature` 仍然基于旧的提交继续开发。

    ```text
    A---B---C---F---G  main
         \
          D---E        feature
    ```

- **直接合并：** 这时直接合并会产生额外的 `merge commit`，这个提交不是新的业务代码，而是把 `main` 和 `feature` 两条线合到一起。

    ```text
    A---B---C---F---G---M  main
         \             /
          D---E-------
    ```

- **合并特点：** `merge` 会保留真实分叉历史，也会多出 `merge commit`，提交图会更复杂。

### 场景二：本地与远程提交分叉

- **共同起点：** 你和同事都从提交 `C` 开始分头干活。

    ```text
    A --- B --- C        ← 共同的起点
    ```

- **同事进展：** 同事提交了 `D`、`E` 并推到了远程 `origin/main`。

    ```text
    A --- B --- C --- D --- E      (origin/main，同事的)
    ```

- **本地进展：** 你在本地基于 `C` 提交了 `X`、`Y`。

    ```text
    A --- B --- C --- X --- Y      (你本地的)
    ```

- **历史分叉：** 两边都在 `C` 之后产生了新提交，历史在 `C` 处分叉。

    ```text
                  X --- Y          ← 你的本地提交
                 /
    A --- B --- C
                 \
                  D --- E          ← 同事的提交（已在远程）
    ```

- **Rebase 效果：** 执行 `git rebase origin/main` 后，`git` 先把 `X`、`Y` 临时拿走，让分支指向 `E`，再把 `X`、`Y` 依次“重放”到 `E` 后面。

    ```text
    A --- B --- C --- D --- E --- X' --- Y'    (rebase 之后)
    ```

    - `X'`、`Y'` 是全新的提交，内容跟 `X`、`Y` 一样，但因为父提交变了，哈希值是新的。
    - 历史变成一条干净的直线，看不出曾经分叉过。

- **对比 Merge：** 如果用 `git merge` 而不是 `rebase`，`git` 会造一个额外的合并提交 `M` 把两条线缝起来。

    ```text
    A --- B --- C --- D --- E --- M
                 \               /
                  X --- Y -------
    ```

    - 历史里会留下分叉和合并的痕迹，而 `rebase` 把它拉直了。

## 三、Rebase 解决方式

- **执行命令：** 在 `feature` 分支上执行 `git rebase main`，可以把 `feature` 上的提交重新接到 `main` 最新提交之后。

    ```bash
    git checkout feature
    git rebase main
    ```

- **执行效果：** `rebase` 会把 `D`、`E` 重新接到 `G` 后面。

    ```text
    A---B---C---F---G  main
                     \
                      D'---E'  feature
    ```

- **提交变化：** `rebase` 后提交内容还在，但提交 ID 会变，原来的 `D`、`E` 会变成新的 `D'`、`E'`。

## 四、核心区别

- **Merge 含义：** `merge` 是把两条开发线合起来。

- **Rebase 含义：** `rebase` 是把当前分支的提交搬到最新主线后面。历史更直、更容易看。

## 五、冲突处理

- **重放机制：** `rebase` 把你的提交逐个“重放”到新基底上，像一条流水线依次搬运。

    ```text
    要重放的: [X] [Y]   →  接到 E 后面

    1. A--B--C--D--E--X'        ✓ X 重放成功
    2. A--B--C--D--E--X'--???   ✗ Y 改的地方同事在 D/E 也改了 → 冲突！
                            ↑
                        rebase 在这里停住
    ```

- **冲突选项：** 出现冲突后你处在“rebase 进行中”的状态，处理流程可三选一。

    ```text
              ┌─ 解决冲突 ─→ git add <文件> ─→ git rebase --continue ─→ 继续重放 Y'
    停在冲突 ──┤
              ├─ 想直接跳过这个提交 ─────────→ git rebase --skip
              │
              └─ 不想搞了，全部撤销 ─────────→ git rebase --abort ─→ 回到 rebase 前的样子
    ```

- **解决冲突：** 打开冲突文件，手动修改并通知 Git 继续重放。

    1. 打开冲突文件，定位 `<<<<<<<`、`=======`、`>>>>>>>` 冲突标记。

    2. 手动将代码改成想要的最终样子，并删掉冲突标记。

    3. 执行 `git add <file>` 告诉 Git 冲突已解决。

    4. 执行 `git rebase --continue` 让其接着重放下一个提交。

- **特别提醒：** 解决冲突后千万别用 `git commit`。

    - **自动生成：** 只需要 `git add` 并配合 `git rebase --continue`，Git 会自动使用原来的提交信息生成新的重放提交 `Y'`。

    - **安全撤销：** 如果一时搞乱了，`git rebase --abort` 是永远的后悔药，会把分支恢复到 rebase 开始前的状态，`X`、`Y` 原封不动。

- **跳过提交：** `git rebase --skip` 的意思是“这个正在重放的提交，我整个不要了”。

    - 重放到 Y 时冲突 → `git rebase --skip`
    
    - **结果：** `C --- D --- E --- X'`        ← Y 被整个丢掉，Y 改的内容不会出现在结果里

    - **适用场景：** `Y` 的改动同事在 `D` 或 `E` 里已经做过了，重放时发现“没什么可改的”，这种空提交跳掉正合适。

- **完整四步流程：**

    1. **第 0 步** `git fetch`（或 `pull` 的前半段）：先把同事的 `D`、`E` 拉到本地

          拉取后:  `C --- D --- E`       (origin/main)
                    \
                     `X --- Y`           (你的，还没动)

    2. **第 1 步** rebase 把你的 `X`、`Y` “摘下来”暂存到一边

          `C --- D --- E`                (分支指针先退回，X、Y 被收进暂存区)
          暂存: `[X] [Y]`

    3. **第 2 步** 把你的分支指针移到新基底 `E` 上

    4. **第 3 步** 把暂存的 `X`、`Y` 依次重放，生成 `X'`、`Y'`

          `C --- D --- E --- X' --- Y'`

    - 原来的 `X`、`Y` 并没有被真正“删除”，它们暂时脱离了主线，但还躺在 git 的 `reflog` 里，所以 `git rebase --abort` 能随时把它们找回来。
