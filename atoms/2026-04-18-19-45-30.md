# 驱动注入




Ubuntu Server 的 initramfs.conf 默认 MODULES=most，开箱就已经带 virtio。先 lsinitramfs | grep virtio 探测

### dracut 
dracut -f --add-drivers "virtio_blk virtio_net virtio_scsi virtio_pci"
dracut -f 只重写磁盘上的 initramfs 文件，不重载运行中的内核。正在跑的系统毫无感知，下次重启才生效。

VMware 用 pvscsi / lsilogic_sas / nvme，不用 virtio。initramfs 启动时按 PCI 硬件枚举加载驱动，多带几个 virtio 模块在那儿闲着，VMware 路径会直接跳过它（--add-drivers 是"追加模块可用"，不是"强制这些模块必须接管"）。




#### Win

可行但显著更复杂
不能简单"拷过去就行"，Windows 要求启动关键驱动必须在注册表里被标记为 Start=0（boot-start），否则 boot.wim 会直接 BSOD 进 WinRE。这正是 virt-v2v 在 Windows 路径上干的活。在线注入步骤：

拷 viostor.inf/sys/cat 到 %SystemRoot%\inf\ 和 %SystemRoot%\System32\drivers\
写注册表 HKLM\SYSTEM\CurrentControlSet\Services\viostor：Start=0, Type=1, Group=SCSI miniport
写 HKLM\...\CriticalDeviceDatabase：PCI\VEN_1AF4&DEV_1001 → viostor
全程通过 GuestOps 跑 reg.exe / PowerShell 完成

#### 怎么切？

把驱动注入留在停机窗口：切换窗口里启动一个 transformation VM，把刚传完的盘挂给它 → chroot → dracut --add-drivers 注入 virtio → shut down → 再正式在 AHV 上建 VM 挂盘开机

碰"在源 VM 运行时改它的 initramfs"这种侵入性操作？
保守路线：先数据，等停机了在干净的 transformation VM 里操作，保证不碰源

预检 (pre-flight)：源兼容性、目标容量、网段映射自动检查

Transformation VM 在停机窗口里吃掉整整几分钟：启 VM（30s）+ 挂盘（10s）+ chroot + dracut（30s~2min）+ 关 VM（10s）+ 建正式 VM（30s） = 2~4 分钟纯串行

我们的 GuestOps 预烤把这步整个挪到 prep 阶段，切换窗口零成本 —— 这是我们 vs Move 最关键的优势

Move 是"安全但慢一点"，我们是"激进但快"。我们的 GuestOps 在线预烤对 Move 的 transformation VM 是严格改进（strict improvement），同样的事提前做，停机窗口少 2~4 分钟。这正好是 < 5 min 停机窗口的 KPI 关键一环。

驱动预烤在 prep 阶段 —— 停机窗口比 Nutanix Move 短 2~4 分钟（决定性）

### VMware Converter Standalone（历史教训）
VMware 自己的工具，2022 年 EOL。P2V + V2V 进 VMware。重点是：它也走"在线预注入 VMware Tools 驱动再切换"路线，跟我们的 GuestOps 预烤路径一致。

证明了："源还在运行时改 initramfs/注册表 → 然后切换"这条路 VMware 自己用了 15 年，工业可靠。我们不是在冒险走无人区。

### 我们这条路的缺点

OS 覆盖面是 MVP 瓶颈 —— 第一版 Ubuntu 18/20/22 + CentOS 7/8 + RHEL 7/8，Windows 要第二版
边缘 case 积累为 0 —— UEFI+LVM+LUKS 的组合炸了只能现场查，没有十年工单库
只管 VMware 源 —— 裸机、Hyper-V、公有云源，一概不支持（战略上也不需要）


#### virt-v2v

停机后 libguestfs 离线读盘（一次性全量） chroot 进去 dracut（需要停机窗口），停机窗口 小时级

源 VM 停机后启动一个 libguestfs appliance（tiny KVM），把源盘挂进去 chroot，在 chroot 里：

dracut --add-drivers virtio* 注入 virtio
改 /etc/fstab、/boot/grub2/grub.cfg 修 UUID 和 bootloader
卸 VMware Tools、装 qemu-guest-agent
Windows：改注册表 CriticalDeviceDatabase + 拷 virtio-win 的 viostor.sys
输出 qcow2/raw，再 upload 到目标（libvirt/oVirt/OpenStack）

为什么是这样
libguestfs 这套工具链就是为离线对磁盘动手术设计的。RH 的 POV 是 "一次性转换，转完就扔"，不关心停机窗口。

优点：
OS 覆盖面爆表：RHEL 49、CentOS、SLES、Ubuntu、Debian、Windows XPServer 2022、AIX、Solaris……几十年积累的 libosinfo + 转换脚本
细节处理无敌：SELinux relabel flag、fstab UUID 重写、cloud-init 重置、网卡 MAC 清理、hostname 保留、kernel cmdline 清 console=ttyS0、Windows 激活重置……我们根本不知道的坑它都填过
Windows 驱动注入已经工业化：打包了 virtio-win 驱动，有完整的 INF → 注册表自动化逻辑
纯离线，不碰源网络：连 VMware Tools 都不需要


缺点：
必须停机才能开始 —— 500 GB 生产库停机窗口 = 传输时间 + 转换时间 + 目标启动时间 = 几小时。V2 架构存在的全部理由就是打掉这个。
没有 CBT，跑一次就是一次全量，中途失败从头来
必须中间落 qcow2/raw 文件，要大临时存储
单进程单流，没有 proxy 并发

virt-v2v 是"牺牲停机换正确性"的瑞士军刀。我们是"牺牲部分 OS 覆盖换零停机"。优化目标完全不同，不是同一维度的竞品，而是互补：V1（我们旧路线）直接调 virt-v2v 是合理的；V2 架构抛弃 virt-v2v 是 KPI 决定的必然。

借 virt-v2v 的转换细节 checklist —— 它的 convert_linux.ml 是金矿，列了所有要改的地方（fstab、grub、cloud-init、SELinux、NIC mac……），我们 dracut 脚本抄一份正确性检查表

###  VMware Guest Operations API

"源 VM 没 SSH"的坑：走 VMware Tools 内通道，不依赖 VM 的网络/SSH，只需要 vCenter/ESXi 的 443，无需访问源 VM
这套 API 就是为"在 VM 里跑脚本不开 SSH"设计的。

走 VMware Tools 这条 ESXi 和 Guest OS 之间的"内嵌管道"。只要 VM 里装了 Tools 且运行着，ESXi 宿主就能给 Guest OS 下命令：上传文件、执行程序、拿退出码。

借用 VMware Tools 的 Guest Operations API，从外面让 OS 跑一个临时脚本（不留常驻进程）
没有自己装 Agent 的成本。

### 风险

源 VM 内 VMware Tools 心跳掉（装了但没 running）	中	编排器开工前先 GuestAuth 探测一次，不通直接 fail fast

RHEL 8 偶见 dracut-install: ERROR installing virtio_pci（内核 module 包缺）	中	脚本里先 modinfo virtio_pci 探测模块存在，不存在就先 yum install kernel-modules

用户脚本权限错误（非 root 跑 dracut）	低	GuestOps 用 NamePasswordAuthentication 走 root 账号

Pro（相对我们）
真正跨 hypervisor：我们绑死 VMware（VDDK + HotAdd + CBT）
能读 Windows VSS：应用一致性比 VMware quiesce 更准
能精细控制迁移策略（排除文件、选择性迁移），我们只能整盘
不依赖 VDDK license（VDDK 免费下载但商用有条款）

Con（相对我们）
Agent 安装是最大的落地阻力：企业客户"在 500 台生产 VM 上装未知来源的 Agent"要过安全评审、变更评审，周期几周
Agent 在 guest 内读盘 = 跟应用抢 I/O；我们 HotAdd 在 hypervisor 层读，源 VM 几乎无感

我们为什么不用：客户买 FC 的时候没有"必须先在 100 台生产 VM 装个不知道哪来的程序"的心理预算。这一关过不了，单子就丢了。

⚠️ 战略关键：我们的目标市场（FC 客户 = 华为自家客户）大量是"VMware → FusionCompute"场景，同厂换平台。这种场景下客户对"装 Agent"的抗拒远大于对"OS 覆盖面"的要求 —— agentless 是最核心的竞争力。

### HyperBDR / CloudEndure / PlateSpin / Carbonite Migrate（商业 CDP）

CDP:  Continuous Data Protection（持续数据保护）
字面意思：每写一次磁盘，立刻同步到对端。不是定时备份（每小时/每天），是写一个字节立刻复制一个字节。

对比：
方式	频率	丢数据上限（RPO）
每天全备	1 天	24 小时
每小时增备	1 小时	1 小时
CBT 增量（我们）	1 轮几分钟	几分钟
CDP（HyperBDR）	实时	几秒

普通驱动：操作系统跟硬件沟通的翻译官（鼠标驱动、显卡驱动）。
内核 Filter Driver（过滤驱动）：插在 OS 写磁盘的必经之路上的拦截器。

OS 写磁盘的正常流程：
应用程序 write() → 文件系统 → 块设备层 → 真实磁盘
装了 filter driver 之后：
应用程序 write() → 文件系统 → 块设备层 → [filter driver 抄一份] → 真实磁盘
                                                 ↓
                                            发到对端 appliance

为什么叫"内核级"：这个监听器必须装在 OS 内核里（不是用户态程序）才能拦截到所有写入 —— 用户态拦不住其他用户态程序的 IO。

怎么做
在 guest 里装 内核级 filter driver（Linux: blk 层 hook；Windows: storage stack filter）。Driver 截获每一次写，记 journal → 流到对端 appliance → 对端按写入顺序 replay 到目标卷。切换时 stop-writes → flush journal → failover。

本命是 DR（灾备），迁移是顺带。DR 要 RPO < 1 min，polling CBT 做不到，只有 filter driver 能实时抓每个写。

优点：
RPO 秒级，我们 CBT 最多做到分钟级
不用 VMware snapshot：VMware 快照会拖累源性能 + 数据存储膨胀（redo log），我们 Round 0→Round N 的快照链会越拖越重
跨 hypervisor



缺点：
内核驱动是企业运维最恨的侵入性改动：
Windows 装驱动要重启
内核版本对不上直接 BSOD / kernel panic
驱动签名、SecureBoot、HIDS（EDR）冲突一堆麻烦
跑在内核里 = 写错一行 → 整机蓝屏/panic，不像普通程序崩了重启就行
Windows 装内核驱动要重启 + 走数字签名，企业改个东西要走变更流程
内核版本升级要重新编译/适配：CentOS 7 升小版本可能就 panic

他们优化 RPO 秒级 DR，我们优化规划性迁移分钟级停机。对"规划内迁移"这个场景，RPO 秒级属于过度设计，为此装内核驱动不划算。

#### 在线 vs 离线注入 —— 生产级抉择


本质区别（不是"时机"，是 10 个连锁后果）
维度	在线注入	离线注入
盘的状态	运行中 OS 正在读写	盘已冷，独占访问
对源 VM 影响	initramfs 文件被替换，运行中内核毫无感知，下次重启才生效	源已停机，零影响
需要凭证	✅ root 密码 / sudoer	❌ 只要访问盘文件
能处理损坏源 VM	❌ 源起不来就没办法	✅ 只要盘文件在
工具可用性	受限于源 OS 装了什么（老 CentOS 可能没 dracut）	救援环境里塞齐所有工具
Windows boot-start 驱动	能干但诡异（注册表部分键受保护）	干净利落（reg load 直接改 hive）
崩溃恢复	dracut 自身原子写，但 grub 改错次启动就炸	盘已卸载，重试无副作用
一致性	依赖应用 quiesce	天然一致（盘已冷）
切换窗口成本	0（prep 阶段已经做完）	2~4 min（boot 救援 VM + chroot + 关机）
回归风险	动了源 VM，理论上有"改完源在 VMware 上起不来"的灾难	零 —— 源从头到尾没碰

真正决定性的一条：在线注入的最大 Pro = 切换窗口不吃驱动时间；离线注入的最大 Pro = 源 VM 从头到尾不被碰，出问题也能回滚

### Nutanix Move 的架构

三种 Preparation 模式（Move 的精华设计）
Move UI 提供三个选项：
模式	谁来干	怎么干
Automatic	Move 自己	GuestOps 推脚本 → 远程跑 dracut / PowerShell
Manual	运维自己	Move 打印出完整脚本 → 运维在源 VM 里手工跑 → 回 Move UI 打勾
Mixed	混合	迁移计划里每台 VM 单选模式

Move 没有"离线 rescue VM 兜底"，当 Automatic 失败时，唯一的 fallback 是切到 Manual 让人工处理。这解释了为什么社区里大家的"迁移经验贴"普遍建议：老运维都绕过 Move 的 Automatic 模式，自己先装好驱动再用 Move —— 因为 Automatic 太脆

Move 的 Windows 双段式：

Stage 1（源 VM 在 VMware 上跑时，走 GuestOps）：PowerShell 拷 virtio .inf/.sys 到 %SystemRoot%\inf\、注册一个 Windows Scheduled Task
Stage 2（VM 在 AHV 上第一次开机时）：那个 Scheduled Task 自动跑，用 pnputil /install 正式激活驱动 + 改注册表 Start=0 + 清 VMware Tools
为什么要两段：Windows 把 virtio 标记为 boot-start 必须用 pnputil，而 pnputil 在"还跑在 VMware 的源 VM 里"执行会立刻触发 PnP 枚举 virtio 硬件 —— 但源 VM 上根本没 virtio 硬件（它在 VMware 上），会注册失败或留下脏状态。Move 绕开这个矛盾：在源 VM 里只拷文件 + 埋任务；真正激活推迟到 AHV 环境跑。

这是 AWS MGN "launch template" 思路的轻量版 —— 一部分工作（复制驱动文件）前移到 prep，一部分工作（激活驱动）推后到首次启动。值得我们 Windows 版参考。

失败模式（Move 的真实痛点）
从社区故障贴扒出来的清单：

失败	根因	Move 的处理
dracut: installkernel failed in module kernel-modules-extra	RHEL 没装 kernel-modules-extra 包	报错，让用户手工装
Ubuntu 24.04 virtio_scsi could not be installed for kernel 6.8.0-35-generic	Move 脚本硬编码 kernel 版本适配表，新 kernel 不在里面	报错，建议切 Manual
Rocky Linux 8.8 Could not authenticate	Guest credentials 传入有问题 / 源 VM sudoers 配置	报错在 Prepare 阶段
CentOS 7 dracut-initqueue timeout（迁完在 AHV 上起不来）	initramfs 里加了 virtio_blk 但没加 virtio_pci，或者 grub 没加对应 rd.modules	事后补救：挂 rescue ISO 重建 initramfs
核心洞察：Move 的 Automatic 模式在主流场景（RHEL/CentOS/Ubuntu + 标准 kernel）可靠率 ~90%。剩下 10% 客户要么切 Manual，要么事后救援。这跟我们 MVP 规划的可靠率目标一致。

维度	Nutanix Move	我们 bifrost
形态	VM appliance（产品化）	Windows 本地编排器（开发期）
数据面	VDDK + CBT + HotAdd	VDDK + CBT + HotAdd（一致）
Linux 驱动注入	GuestOps 推脚本跑 dracut（在线）	GuestOps 推脚本跑 dracut（在线，一致）
Windows 驱动注入	GuestOps 拷文件 + 埋 Scheduled Task，首次 AHV 开机激活	未规划（V3 再做，可抄 Move 思路）
失败 fallback	Manual 模式（用户手工）	未规划（V2 可加 rescue VM）
OS 支持矩阵	主流 Linux + Windows 2008~2022 + 几十个变种	Ubuntu 22 / CentOS 7（MVP）
预检 pre-flight	有（扫 kernel 版本、RDM、快照链）	没有（V1.5 必须加）
UI / Job 管理	完整 Web UI	纯 CLI 日志
成熟度	7 年 + 上万客户反馈	MVP

启示：
驱动注入脚本要按 CPU 架构分 AMD/Intel —— Move 这么分是有原因的，某些 virtio 模块（virtio_input、virtio_balloon）在 AMD SEV 环境下有特殊处理。我们 MVP 先不管，但脚本接口留好（脚本参数里传 --cpu-vendor），以后补。
脚本要处理 kernel-modules-extra / linux-modules-extra-* 缺失：RHEL/CentOS 常见；Ubuntu 叫 linux-modules-extra-$(uname -r)。注入脚本开头先 yum install -y kernel-modules-extra-$(uname -r) / apt install -y linux-modules-extra-$(uname -r)，装不上就报错退出（别硬着头皮跑 dracut）。
Windows 两段式：我们 V3 Windows 路径要学 Move 的"拷文件 + 埋 Scheduled Task + 首次启动激活"。这也是 Plan B 之外的第二条 Windows 路径。
Manual 模式不能少：V1 就应该支持"用户说我自己改过 initramfs 了，你别动"的跳过开关。至少 CLI 加个 --skip-driver-injection flag。
预检要早做：Move 的 Pre-flight 里扫 kernel 版本、检查 virtio 模块是否已存在、验证 guest credentials 能不能登。这些我们在里程碑 5.1 通道探测阶段就能顺手做了。


我们能做得比 Move 更好的地方
切换窗口更短：Move 的切换窗口里还要等 VADP snapshot 创建 + 卷切换 + AHV 建 VM，实测大 VM 能到 5~15 分钟（rack2cloud 文章里有数据）。我们因为目标卷是提前 pre-provisioned 的，切换时只要卸卷+FC 建 VM 挂卷，能压到 < 5 min。
GuestOps 失败的体验更友好：Move 失败就甩一串 dracut 错误让用户干瞪眼。我们可以在编排器里把 stderr 结构化，失败时明确告诉用户"缺 xx 包，运行 yy 命令"。
预检能做得更激进：Move 的预检是黑盒的（vCenter 权限决定了能看什么）。我们 VDDK 能直接看 VMDK 元数据，能发现更多问题（比如 VMDK 有 snapshot 链太长）。
### Linux 内核的"驱动"有两种身份
编译内核时，每个驱动（比如 virtio_pci）在 .config 里可以取三种值：

.config 里的值	编译产物	形象比方
CONFIG_VIRTIO_PCI=y	驱动代码编译进 vmlinuz 本体，不存在单独文件	驱动"长在内核身上"
CONFIG_VIRTIO_PCI=m	驱动代码编译成独立的 virtio_pci.ko 文件，放在 /lib/modules/<kver>/ 下	驱动是"外挂配件"，要用时才 load
CONFIG_VIRTIO_PCI 未设 / =n	根本不编译这个驱动	这个内核不支持 virtio_pci
=y = built-in / 内置
=m = module / 可加载模块
启动时发生什么（关键在这里）
BIOS/UEFI  →  bootloader (grub)  →  把 vmlinuz + initramfs 读进内存  →  vmlinuz 开始跑
                                                                              ↓
                                                            内核初始化，要 mount 根盘（rootfs）
                                                                              ↓
                                                            根盘在哪？—— 是块 virtio_blk 设备
                                                                              ↓
                                     【分岔】vmlinuz 现在有没有驱动这块盘的能力？
                    ┌─────────────────────────────────────────┴─────────────────────────┐
                    ↓                                                                   ↓
        virtio_blk=y (built-in)                                          virtio_blk=m (module)
                    ↓                                                                   ↓
        vmlinuz 本身就能识别 virtio_blk                   vmlinuz 自己不会，要从 initramfs 里
        设备，直接 mount 根盘 ✓                           加载 virtio_blk.ko，再 mount 根盘
                    ↓                                                                   ↓
        initramfs 里 **根本不需要** virtio                initramfs 里 **必须有**
        相关 .ko 文件                                     virtio_blk.ko 这个模块文件
模块是编译进 vmlinuz 的，它没有独立的 .ko 文件，你要是想去 /lib/modules/ 里找 virtio_pci.ko，你找不到的，不是因为它丢了，是它根本不该以那种形式存在。

 VM 的内核把 virtio 驱动编进 vmlinuz 本体了。这意味着：

✅ vmlinuz 本身就认识 virtio 设备
✅ 启动时不需要 initramfs 配合加载 virtio 驱动
✅ 这个 vmlinuz 文件原封不动搬到 FC 上，FC 上的盘挂成 virtio_blk 设备，它直接就能认、直接就能启动到登录页
结果：源 VM 不需要任何预烤、不需要装包、不需要改 initramfs、不需要重启 —— 开箱即迁移。
