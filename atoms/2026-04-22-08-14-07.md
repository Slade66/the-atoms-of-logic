# Kratos Proto 生成机制

### Makefile 解析

先装齐 6 个工具（make init）
Makefile:20-26
init:
    go install google.golang.org/protobuf/cmd/protoc-gen-go@latest
    go install google.golang.org/grpc/cmd/protoc-gen-go-grpc@latest
    go install github.com/go-kratos/kratos/cmd/kratos/v2@latest
    go install github.com/go-kratos/kratos/cmd/protoc-gen-go-http/v2@latest
    go install github.com/google/gnostic/cmd/protoc-gen-openapi@latest
    go install github.com/google/wire/cmd/wire@latest
各 plugin 的职责：

protoc-gen-go：生成 *.pb.go（message 结构体 + 序列化）
protoc-gen-go-grpc：生成 *_grpc.pb.go（gRPC server/client stub）
protoc-gen-go-http：Kratos 特有，读取 google.api.http 注解，生成 *_http.pb.go（HTTP server/client，基于 Kratos transport）
protoc-gen-openapi：生成 openapi.yaml（Swagger 文档）
kratos：脚手架 CLI（kratos proto add/client 等）
wire：依赖注入（不属于 proto，但常一起生成）

这些二进制必须在 $GOPATH/bin 里，且该目录在 PATH 中，protoc 才能通过 --go_out / --go-http_out 这样的参数名找到对应 plugin（protoc 约定 --xxx_out 会去找 protoc-gen-xxx）。


两类 proto 的分工
Kratos 模板刻意把 proto 分成两堆，目录不同、生成参数也不同：

api/**/*.proto：对外 API（service 定义 + request/response message），要生成 pb + grpc + http + openapi
internal/**/*.proto：内部配置（如 @a:\Users\liyuz\Desktop\bifrost\internal\conf\conf.proto），只生成 pb，不需要 service
Makefile 里通过 find 动态收集：

Makefile:11-15
INTERNAL_PROTO_FILES=$(shell $(Git_Bash) -c "find internal -name *.proto")
API_PROTO_FILES=$(shell $(Git_Bash) -c "find api -name *.proto")
Windows 下用 Git Bash 的 find.exe，避免撞上系统 find。



config:
	protoc --proto_path=./internal \
	       --proto_path=./third_party \
	       --go_out=paths=source_relative:./internal \
	       $(INTERNAL_PROTO_FILES)

--proto_path=./internal：import 搜索根（第一个）
--proto_path=./third_party：第二个根，里面放 google/api/annotations.proto、validate/validate.proto 等第三方 proto，import 才能解析到
--go_out=paths=source_relative:./internal：把 .pb.go 输出到 a:\Users\liyuz\Desktop\bifrost/./internal，paths=source_relative 表示 生成文件与源 .proto 保持相同相对路径（而不是按 go_package 重建目录），这也是为什么 conf.proto 旁边直接出现 conf.pb.go


api:
	protoc --proto_path=./api \
	       --proto_path=./third_party \
	       --go_out=paths=source_relative:./api \
	       --go-http_out=paths=source_relative:./api \
	       --go-grpc_out=paths=source_relative:./api \
	       --openapi_out=fq_schema_naming=true,default_response=false:. \
	       $(API_PROTO_FILES)


--go_out → greeter.pb.go（message）
--go-grpc_out → greeter_grpc.pb.go（gRPC GreeterServer 接口）
--go-http_out → greeter_http.pb.go（Kratos HTTP GreeterHTTPServer 接口，按 google.api.http 注解生成路由）
--openapi_out → 根目录 openapi.yaml
fq_schema_naming=true：schema 用全限定名（helloworld.v1.HelloRequest）避免撞名
default_response=false：不自动塞一个默认 default response

### HTTP 注解怎么驱动生成

rpc SayHello (HelloRequest) returns (HelloReply) {
  option (google.api.http) = {
    get: "/helloworld/{name}"
  };
}

这个 option (google.api.http) 就是来自/annotations.proto。protoc-gen-go-http 扫到后会：

生成 GreeterHTTPServer 接口
生成 RegisterGreeterHTTPServer(srv, impl)，内部把 GET /helloworld/{name} 绑到 SayHello
自动把 URL path 里的 {name} 填到 HelloRequest.Name


### 典型工作流
平时改 API 只需：

bash
# 1. 编辑 api/xxx/v1/xxx.proto
# 2. 重新生成
make api
# 3. 实现（或修改）service
#    Kratos 服务端实现通常在 internal/service/xxx.go，要满足新的 XxxHTTPServer / XxxServer 接口
# 4. 如果改了 internal/conf/*.proto
make config
# 5. 如果动了 wire.go 注入
make generate
或者一键：make all（= api + config + generate）。

### 用 kratos CLI 加速（可选）
kratos proto add api/xxx/v1/xxx.proto：创建一个带标准头部的 proto 骨架
kratos proto client api/xxx/v1/xxx.proto：等价于 make api 单文件
kratos proto server api/xxx/v1/xxx.proto -t internal/service：根据 proto 里的 service 反向生成 internal/service/xxx.go 实现骨架（方法签名、TODO 占位）


## 底层驱动翻译

gorm.ErrDuplicatedKey 分支是死代码，永远不会命中
esxi_host.go:67-72
if err := r.data.DB.WithContext(ctx).Create(&po).Error; err != nil {
    if errors.Is(err, gorm.ErrDuplicatedKey) {
        return biz.ErrEsxiHostNameConflict
    }
    return err
}
GORM 官方文档 Error Handling 白纸黑字写了：这类 dialect-translated error 必须显式打开 TranslateError: true 才会把底层驱动的错误翻译成 gorm.ErrDuplicatedKey；没开的情况下，errors.Is(..., gorm.ErrDuplicatedKey) 永远是 false（https://gorm.io/docs/error_handling.html → Dialect Translated Errors）。

看你的 gorm.Config：

data.go:58-60
return gorm.Open(dialector, &gorm.Config{
    Logger: gormlogger.Default.LogMode(gormlogger.Info),
})
没开 TranslateError。结论：目前 name 冲突会以 SQLite 原始错误穿透到 HTTP 层，返回 500 UNIQUE constraint failed: esxi_host_pos.name，根本不是你设计的 ESXI_HOST_NAME_CONFLICT / 409。

修法一行：

go
return gorm.Open(dialector, &gorm.Config{
    TranslateError: true,
    Logger:         gormlogger.Default.LogMode(gormlogger.Info),
})
Update 里那个同样的分支也是一起复活的。
