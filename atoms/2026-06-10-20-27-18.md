## 结构化错误明细与元数据设计

### 痛点缺陷
逐台设备的错误明细在 service 层被“拍扁”成了长字符串。虽然 biz 层已经收集了结构化的错误列表 `[]DeviceConfigError{DeviceName, Error}`，但 `service/plan.go` 却用分号拼接成一坨字符串（如 `"SW1: 未选择主模板; SW2: 渲染失败; ..."`）塞入 `message` 并直接返回。这导致：
1. **前端展示困难**：前端拿到这坨字符串后只能整段弹窗，无法渲染成结构化的设备列表或跳转链接。
2. **契约脆弱**：如果前端尝试使用 `"; "` 强行切分，一旦某个设备的错误文案中本身包含分号，解析就会彻底失效。
3. **元数据缺失**：手搓的 `DEVICE_CONFIG_ERRORS` 隐式契约在 API 文档中无法体现。

### 改法原则（各司其职）
- **`message` 给人看**（短中文摘要，代表大方向）。
- **`metadata` 给程序看**（结构化明细，以 JSON 格式传递数据）。

### Kratos 元数据设计实践
Kratos 的 `*errors.Error.Metadata` 在 Proto 中被定义为 `map[string]string`，不支持嵌套对象。要传递结构化列表，必须将 JSON 数组序列化为字符串。

**后端错误构建与翻译示例**：
```go
// ExportDeviceConfigs 导出规划下各设备的命令配置，打包成 zip
func (s *PlanService) ExportDeviceConfigs(ctx context.Context, planUUID string) (string, []byte, error) {
    files, err := s.uc.ExportDeviceConfigs(ctx, planUUID)
    if err != nil {
        return "", nil, toExportAPIError(err) // 统一翻译领域错误，不直接裸透传
    }
    zipBytes, err := zipConfigFiles(files)
    if err != nil {
        return "", nil, pb.ErrorPlanExportConfigFailed("打包 zip 失败：%v", err)
    }
    return "configs_" + planUUID + ".zip", zipBytes, nil
}

// toExportAPIError 把导出用例的领域错误翻译成 API 错误
func toExportAPIError(err error) error {
    var devErrs *biz.ErrDeviceConfigErrors
    if errors.As(err, &devErrs) {
        detail, _ := json.Marshal(devErrs.Errors)
        return pb.ErrorPlanDeviceConfigErrors("%s", devErrs.Error()). // 「N 台设备配置生成失败」
            WithMetadata(map[string]string{
                "device_errors": string(detail),
                "failed_count":  strconv.Itoa(len(devErrs.Errors)),
            })
    }
    
    if errors.Is(err, biz.ErrPlanNotFound) {
        return pb.ErrorPlanNotFound("%s", err.Error())
    }
    
    return pb.ErrorPlanExportConfigFailed("%s", err.Error())
}
```

**响应的 JSON 结构**：
```json
{
  "code": 400,
  "reason": "PLAN_DEVICE_CONFIG_ERRORS",
  "message": "3 台设备配置生成失败",
  "metadata": {
    "device_errors": "[{\"device_name\":\"SW1\",\"error\":\"未选择主模板\"},{\"device_name\":\"SW7\",\"error\":\"渲染失败：模板缺字段 .vrfs\"}]",
    "failed_count": "3"
  }
}
```

## 接口错误的分类与状态码分级机制

### 痛点缺陷
业务错误与系统错误界限模糊。例如，当非设备类错误发生时直接裸透传（如规划不存在的 `ErrPlanNotFound` 本应报 400，但实际抛出了 500）；或者手搓字符串定义状态，导致 proto 的 `ErrorReason` 枚举中查无此码。

### 分级设计的意义与告警过滤
1. **责任归属明确**：前端通过 `reason` 可以清晰判定是“用户配置问题需在界面修改（4xx）”，还是“系统服务崩溃需要联系运维排查（5xx）”。
2. **监控指标降噪**：运维监控系统通常只对 5xx 响应进行告警。如果把“用户没选主模板（4xx）”误报为 500，不仅会在半夜触发无意义的运维值班电话，还会导致真实的 5xx 系统级故障由于告警泛滥而被淹没。
3. **网关与客户端自动重试**：根据契约，网关或客户端仅应当自动重试 5xx 和 429 错误。4xx 错误由于请求本身就是非法的，重试一万次也无济于事，应当立刻拦截。

## 软件工程中的状态码与责任归属契约

HTTP 状态码是客户端与服务端之间明确的「责任归属契约」：

| 状态码 | 含义 | 一句话本质 |
| :--- | :--- | :--- |
| **2xx** | 成功 | 服务端成功办妥了请求 |
| **3xx** | 重定向 | 请求需要前往另一个地址处理 |
| **4xx** | 客户端错误 | 请求体/参数本身有错误，重试无意义，必须修改参数后再发起 |
| **5xx** | 服务端错误 | 服务端内部异常，请求没问题。过会儿自动重试可能会成功 |

### 常见状态码场景对应表

| 调用方期望与行为 | 状态码 | 典型使用场景 |
| :--- | :--- | :--- |
| **修改请求参数/请求体** | 400 Bad Request | 数据校验失败、缺少必填字段、配置项冲突（如设备没选主模板） |
| **先去登录系统** | 401 Unauthorized | 认证 Token 缺失或已过期（未认证） |
| **更换有权限的账号** | 403 Forbidden | 已登录但当前角色无操作权限（已认证但无权） |
| **换个存在的资源 ID** | 404 Not Found | 规划的 UUID 在数据库中查无此项 |
| **解决状态冲突后再试** | 409 Conflict | 重复提交表单、并发修改引起冲突撞车 |
| **慢点请求** | 429 Too Many Requests | 触发平台并发限流机制 |
| **无需修改，等待排查** | 500 Internal Error | 后端代码 Bug、运行时 Panic、数据库或基础服务宕机 |
| **等待系统服务恢复** | 503 Service Unavailable | 依赖的微服务挂了、系统正在停机维护发布中 |
| **请求超时，请重试** | 504 Gateway Timeout | 后端接口耗时超出门限（通常由网关代理层发出） |

## 服务端日志的分级规范与上下文埋点

### 痛点缺陷
1. **设备细节缺失**：仅打印汇总行（如 `Warn: 3/32 台生成失败`），日志中没有包含“哪一台”及“具体什么错误”，导致运维只能找前端用户要截图。
2. **Panic 堆栈丢失**：模板渲染的 panic 异常被 `recover` 拦截后，仅返回“渲染异常”，没有打印现场调用堆栈，导致代码 bug 无法追踪。

### 日志级别与上下文规范
- **Error**：系统级故障，需要运维或后端介入排查（如数据库断连、Bug、Panic 等）。
- **Warn**：用户输入数据异常，或者已被系统妥善降级/兜底处理的非致命问题。
- **Info**：记录正常的业务流。
- **单请求一条结局日志原则**：避免多层级重复打印同一条错误日志，减少刷屏；日志必须带上唯一可检索的上下文（如 `plan_uuid` ；设备级报错需带上 `device_name`）。
- **Panic 处理**：所有被 `recover` 拦截的 Panic，日志输出时级别必须为 `Error` 并且必须附带 `debug.Stack()` 以保留完整的堆栈信息。

**优化后的日志埋点规范**：
- 当部分设备配置生成失败时，在输出汇总 Warn 之外，为每台失败的设备逐行打印一条简明 Warn：
  ```text
  Warn 设备配置生成失败：plan=plan-fb6, device=SW1, 原因=未选择主模板
  ```
- 异常捕获改用 Error 输出：
  ```go
  if r := recover(); r != nil {
      log.Errorf("渲染引擎发生 panic：%v\n%s", r, debug.Stack())
  }
  ```

## 第一性原理指导下的技术文档与计划编写规范

### 1. 第一性原理的实践
第一性原理要求我们抛弃已有的惯例思维，将复杂系统拆解到最基础、无法再拆分的基本事实，以此为基石重新推导构建逻辑。

### 2. 计划与技术文档的编写原则
- **避免过细细节**：不要将具体的函数签名、入参出参以及循环控制逻辑等内容直接写入计划文档中。这些技术实现细节在代码评审（PR）时极易发生漂移，容易导致文档编写完即过时。
- **明确“改哪”与“达成效果”**：文档中应将关注点放在架构影响范围（落在哪些文件、哪个模块）和改动意图上，将底层的执行逻辑留给代码本身去表达。
- **提供清晰的验收标准 (Acceptance Criteria)**：每个步骤应当拥有“做完的判据”，最好使用可执行的断言表示（例如：“使用 ACS RDB 规划导出后，查看 `LF14_A.cfg` 文件的 vlan batch 行必须包含 `vlan batch 1200`”）。
- **明确 Non-goals（非目标范围）**：单独列出一节，明确指出本次改动“不做什么”（例如保持模板语法不变、仅维护回退机制等），防止计划实施范围不受控蔓延。
- **详略与改动风险成正比**：遵循 Google、RFC、ADR 等业界标准，计划文档的详细程度应当与本次改动的技术风险呈正相关。
