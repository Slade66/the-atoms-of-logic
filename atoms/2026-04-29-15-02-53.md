# firewalld

- **一句话定义** `firewalld` 是 Linux 系统上的防火墙管理工具，常见于 RHEL、CentOS、Fedora、AlmaLinux 等发行版。

- **核心特点**

    1. 它支持动态更新规则，也就是修改防火墙策略时不需要重启服务，不会导致服务断连。

    2. 它引入了 Zone 的概念，让同一台机器可以根据网络环境切换不同的防火墙策略。

### firewalld 有什么用

- **阻止未授权访问** 默认情况下，firewalld 会阻断大部分外部主动连接。

- **显式放行流量** 只有明确允许的端口、服务或规则，外部流量才可以进入。

- **降低服务暴露面** 即使机器上运行了某个有漏洞的后台服务，只要对应端口没有被放行，外部攻击者也很难直接探测或利用它。

- **适配不同环境** 同一套系统可以在家庭、公司、公网服务器等环境中切换不同策略，不需要每次从零配置。

### Zone 是什么

- **基本含义** Zone 可以理解为一套预设的网络信任等级。

- **核心问题** 一台电脑可能出现在不同网络环境里，而不同环境需要不同的安全策略。

- **在家场景** 家里的设备通常比较可信，可以使用 `trusted` 这类更宽松的区域。

- **办公室场景** 办公网属于半信任环境，可以使用 `work` 或 `internal`，只开放必要的办公服务。

- **公共网络场景** 咖啡厅、酒店、机场这类陌生网络更适合使用 `public`，只允许非常有限的入站连接。

- **价值：快速切换** 有了 Zone 之后，只需要把网卡切换到对应区域，预设规则就会整体生效，不必手动重写一堆端口规则。

### 常见 Zone

- **预设区域** firewalld 提供了一组常见 Zone，每个 Zone 都代表一种默认安全姿态。

    | Zone | 默认行为 | 适用场景 |
    | --- | --- | --- |
    | `drop` | 丢弃所有进来的包，不回应对方 | 遭受攻击时的应急姿态 |
    | `block` | 拒绝所有连接，并返回拒绝消息 | 基础防御姿态 |
    | `public` | 默认选项，只允许明确放行的服务或端口 | 暴露在互联网或陌生网络中的机器 |
    | `work` | 默认拒绝大多数入站请求，但适合办公网络 | 公司办公环境 |
    | `internal` | 信任程度较高，常用于内部网络 | 公司内网或受控局域网 |
    | `trusted` | 接受所有流量 | 极可信的私有环境或测试环境 |

- **选择原则** 网络越陌生，Zone 越应该保守；网络越可信，Zone 才可以更宽松。

- **work 区域**

    - 它会拒绝大部分入站请求，但相比 `public`，更适合开放办公环境里常用的服务。

    - 它解决的是“半信任环境”的平衡：不能像公网一样极度封闭，也不能像 `trusted` 一样完全放开。

### 底层原理

- **整体链路** firewalld 的底层关系可以理解为：`firewall-cmd` 发命令，`firewalld` 管理规则，`nftables` 或 `iptables` 表达规则，内核的 Netfilter 真正拦截数据包。

- **命令与管理层** `firewall-cmd` 是用户入口，firewalld 接收命令后，负责维护配置、切换 Zone，并把配置下发给底层后端。

- **规则后端** nftables 和 iptables 都是在管理 Netfilter 规则；现代系统通常使用 nftables，旧系统可能仍然使用 iptables。

- **内核执行层** Netfilter 是 Linux 内核中的数据包过滤框架，最终拦截、放行、丢弃数据包的动作都发生在这一层。

### 常用操作

- **启动服务** 使用 `systemctl` 启动 firewalld，并设置开机自启。

    ```bash
    systemctl start firewalld
    systemctl enable firewalld
    firewall-cmd --state
    ```

- **查看当前规则** 使用 `--list-all` 查看当前区域放行了哪些端口和服务。

    ```bash
    firewall-cmd --list-all
    ```

- **临时开放端口** 不带 `--permanent` 的规则会立即生效，但重启后失效。

    ```bash
    firewall-cmd --add-port=8080/tcp
    ```

- **永久开放端口** 带 `--permanent` 的规则会写入永久配置，但需要 `reload` 后才会应用到当前运行状态。

    ```bash
    firewall-cmd --permanent --add-port=80/tcp
    firewall-cmd --reload
    ```

- **永久开放端口范围** 可以一次开放连续端口段。

    ```bash
    firewall-cmd --permanent --add-port=3000-4000/tcp
    firewall-cmd --reload
    ```

- **关闭端口** 使用 `--remove-port` 移除已经放行的端口。

    ```bash
    firewall-cmd --permanent --remove-port=8080/tcp
    firewall-cmd --reload
    ```

- **开放服务** 服务是预定义的端口集合，例如 `http` 通常对应 80 端口，`ssh` 通常对应 22 端口。

    ```bash
    firewall-cmd --permanent --add-service=http
    firewall-cmd --reload
    ```

- **关闭服务** 使用 `--remove-service` 移除已经放行的服务。

    ```bash
    firewall-cmd --permanent --remove-service=http
    firewall-cmd --reload
    ```

### 富规则

- **使用场景** 富规则用于更精细的访问控制，例如“开放数据库端口，但只允许指定 IP 访问”。

- **安全收益** 相比直接向全网开放端口，基于来源 IP 的限制可以显著降低暴露面。

- **示例命令** 只允许 `192.168.1.50` 访问本机 `3306/tcp`。

    ```bash
    firewall-cmd --permanent --add-rich-rule='rule family="ipv4" source address="192.168.1.50" port protocol="tcp" port="3306" accept'
    firewall-cmd --reload
    ```

### Runtime 与 Permanent

- **Runtime 配置** 不带 `--permanent` 的配置属于运行时配置，立即生效，但重启 firewalld 或系统后会丢失。

- **Runtime 用途** 它适合测试规则，尤其是在远程服务器上修改防火墙时，可以避免错误规则永久把自己锁在外面。

- **Permanent 配置** 带 `--permanent` 的配置会写入永久配置，但不会立刻改变当前运行规则。

- **Permanent 用途** 它适合已经验证过的生产配置。

- **标准流程** 先用 Runtime 规则测试，确认连接正常后，再写入 Permanent 配置并执行 `reload`。

    ```bash
    firewall-cmd --add-port=8080/tcp
    firewall-cmd --permanent --add-port=8080/tcp
    firewall-cmd --reload
    ```

### 注意事项

- **远程服务器风险** 在云服务器上修改防火墙前，必须确认 SSH 端口已经放行，否则可能会把自己关在服务器外。

- **记得 reload** 如果使用了 `--permanent`，修改后必须执行 `firewall-cmd --reload`，否则永久配置不会应用到当前运行状态。
