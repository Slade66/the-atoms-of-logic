# 为什么覆盖不生效


根因是：**Nginx 配置文件是单文件 bind mount，CI/CD 更新了宿主机文件，但没有重启 Nginx 容器，只做了 reload，导致容器里仍然读旧配置。**

具体过程是这样：

1. 上次代码里已经把 `deploy/nginx/default.conf` 的 rewrite 删除了。
2. CI/CD 把新的 `default.conf` 上传到了服务器。
3. 部署脚本用 `install` 覆盖了宿主机上的：

```text
/opt/docker-fleet/nginx/default.conf
```

4. 但是 Nginx 容器挂载的是单个文件：

```text
/opt/docker-fleet/nginx/default.conf -> /etc/nginx/conf.d/default.conf
```

5. `install` 覆盖文件时会产生新 inode，而已经运行的 Docker 容器仍然挂着旧 inode。
6. CI/CD 后面只是执行：

```bash
nginx -s reload
```

7. 所以 Nginx reload 的还是容器里旧的配置文件，里面还保留：

```nginx
rewrite ^/api/(.*)$ /$1 break;
```

于是 `/api/auth/login` 被改成 `/auth/login`，后端没有这个路由，就 404。

所以不是 GitHub Actions 没跑，也不是后端没启动，而是**Nginx 单文件挂载 + 只 reload 不 restart** 这个部署细节坑住了。

我已经改了两处，避免下次再发生：

- `deploy-ec2.sh`：已有 Nginx 容器时改成 `docker restart docker-fleet-nginx`
- `.github/workflows/release.yml`：发布结束后也改成重启 Nginx 容器

这样下次 CI/CD 会让容器重新挂载最新配置。

根因
根因是 Nginx 容器使用了单文件 bind mount：

/opt/docker-fleet/nginx/default.conf -> /etc/nginx/conf.d/default.conf
上次 CI/CD 虽然已经把新的 Nginx 配置上传到服务器，也删除了错误的 rewrite 规则，但部署流程只执行了：

nginx -s reload
没有重启 Nginx 容器。

由于宿主机配置文件被 install 覆盖后 inode 发生变化，正在运行的容器仍然挂载旧 inode，所以容器内实际配置还是旧版本，仍保留：

rewrite ^/api/(.*)$ /$1 break;
最终导致 /api/auth/login 被错误转发为 /auth/login。


以后建议
以后更新单文件挂载的 Nginx 配置时，不要只 reload，应直接重启 Nginx 容器。

更稳的方案是挂载整个配置目录，而不是单个配置文件，减少 inode 替换导致的挂载旧文件问题。

CI/CD 发布后应增加最小健康检查，例如：

GET /
GET /login
POST /api/auth/login
只要登录接口不是 200/400 这类后端响应，而是 404/502，就应该让部署失败并打印 Nginx 当前配置。
