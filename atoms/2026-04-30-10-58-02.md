# HTTP 状态码 vs 接口业务码

### 这两类码的层次

- HTTP 状态码：告诉你「请求在 HTTP 层面是否被正确处理」。

- 业务码：再告诉你「这次业务本身是不是成功」。

- 用 HTTP 状态码做**通用分流**（如鉴权失败、请求格式错误、服务端异常）。

- 用业务码做**细分业务逻辑**：鉴权失败用`10004`（token 无效/过期）和`10005`（未登录）。

### HTTP 状态码（协议层）

- 先看 HTTP 状态码，确认服务端是否给出了有效的 HTTP 响应，以及属于哪一类（2xx/4xx/5xx）。

- HTTP 状态码只说明请求在 HTTP 层面的结果（是否被正确处理并返回），是否业务成功要继续看业务码。

- `200`：请求成功到达并处理

  - 常见场景：接口返回成功响应报文（不代表业务一定成功）  
  - 处理方式：继续看业务码确认业务结果

- `400`：请求本身有问题（Bad Request）
  
  - 常见触发场景：参数没填、类型错、JSON 语法错  
  - 处理方式：修正请求参数后重试

- `401`：未登录或凭证失效（Unauthorized）
  
  - 常见触发场景：未登录、Token 无效或过期  
  - 处理方式：提示登录或刷新 token

- `403`：已登录但没有权限（Forbidden）
  
  - 常见触发场景：账号权限不够  
  - 处理方式：提示无权限或引导更换账号/申请

- `500`：服务端出错（Internal Server Error）
  
  - 常见触发场景：后端代码异常、依赖挂了  
  - 处理方式：查看服务端日志，必要时重试

### 业务码（业务语义层）

- `0`：成功。

- `10001`：用户不存在。

- `10002`：密码错误。

- `10003`：账号已禁用。

- `10004`：token 无效/过期。

- `10005`：未登录。

- `10010`：参数校验失败。

- `10011`：学号已存在。

### 同时出现时如何解读

- HTTP 状态码和业务码常会一起返回：一个说明“网络请求是否拿到响应”，一个说明“业务结果是否成功”。

- 解读顺序是：

  - 先看 HTTP 状态码：先判断这次请求在 HTTP 层是否通了。
  - 再看业务码：再判断具体业务是否成功，以及失败具体原因。

- `HTTP 200` + `code=10002`
  
  - HTTP：请求正常经过 HTTP 层并拿到响应。  
  - 业务：业务没有成功，原因是“密码错误”。

  - 处理思路：提示用户重试输入账号密码。

- `HTTP 400` + `code=10010`
  
  - HTTP：请求本身不合法（如参数缺失、格式不对），服务器直接拦截。  
  - 业务：参数校验失败。

  - 处理思路：修正请求参数后重试。

- `HTTP 401` + `code=10005`
  
  - HTTP：未认证成功（未登录或凭证无效）。  
  - 业务：未登录状态。

  - 处理思路：提示登录或刷新 token。
