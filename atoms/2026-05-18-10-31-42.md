# 软删除场景下的唯一索引冲突解决方案

## 问题日志

- **原始数据** 一开始数据库里有一条未删除的 `System` 菜单记录。

    | uid | route_name | deleted_at |
    | --- | --- | --- |
    | menu-1 | System | NULL |

- **软删除行为** 删除菜单时，代码并不会真正执行 `DELETE FROM menus`，而是执行软删除更新。

    ```sql
    UPDATE menus
    SET deleted_at = NOW()
    WHERE uid = 'menu-1' AND deleted_at IS NULL;
    ```

- **占用唯一值** 软删除后的 `menu-1` 还在表里，因此仍然占着 `route_name = 'System'` 这个唯一值。

    | uid | route_name | deleted_at |
    | --- | --- | --- |
    | menu-1 | System | 2026-05-18 10:00:00 |

- **冲突结果** 再创建一个新的 `route_name = 'System'` 菜单时，唯一索引看到旧行还在，就会报唯一键冲突。

    ```sql
    INSERT INTO menus (uid, route_name, deleted_at)
    VALUES ('menu-2', 'System', NULL);
    -- 报错：Duplicate entry 'System'
    ```

## 根因分析

- **根本原因** 软删除只是把 `deleted_at` 填上，数据库行还在，唯一索引依然会检查这行。

- **等价流程** 整个问题可以等价成下面的 SQL 流程。

    ```sql
    INSERT INTO menus (uid, route_name, deleted_at)
    VALUES ('menu-1', 'System', NULL);

    UPDATE menus
    SET deleted_at = NOW()
    WHERE uid = 'menu-1';

    INSERT INTO menus (uid, route_name, deleted_at)
    VALUES ('menu-2', 'System', NULL);
    -- 报错：Duplicate entry 'System'
    ```

## 解决方案

### A

- **推荐方案** 跨数据库最稳的做法是使用普通字段加删除字段的联合唯一索引。

- **字段规则** `deleted_uid` 用来区分未删除记录和已删除记录。

    - 未删除：`deleted_uid = ''`

    - 软删除：`deleted_uid = 当前记录自己的 uid`

- **唯一索引** 对 `route_name` 和 `deleted_uid` 建立联合唯一索引。

    ```sql
    UNIQUE KEY uk_menu_route_alive (route_name, deleted_uid)
    ```

- **实际效果** 同一个 `route_name` 只能有一条未删除记录，但可以存在多条已删除历史记录。

    | uid | route_name | deleted_uid | 含义 |
    | --- | --- | --- | --- |
    | A | System |  | 当前有效，占用 `System` |
    | A | System | A | 已删除，释放 `System` |
    | B | System |  | 新建成功 |
    | B | System | B | 再次删除也成功 |
