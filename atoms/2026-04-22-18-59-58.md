# 调用第三方 API 的设计模式

### 一、Repo / Gateway 命名

- **Repo 定位** `Repo` 是 Repository 的缩写，专门指本地数据库访问层，例如 `FcHostRepo` 负责在本地数据库里增删改查主机信息；它像本地“存折”，里面的数据是系统自己持久化进去的，不负责实时访问远端系统。

- **Gateway 定位** `Gateway` 适合表达第三方系统 API 的访问入口，是 Martin Fowler 在《企业应用架构模式》中定义的经典模式；它强调“翻译”和“过关”，把外部接口协议，例如 FC REST API，翻译成系统内部能理解的业务语言。

- **命名结论** 去 ESXi / FC 这类远端系统实时拿数据的接口，不应该叫 `Repo`，因为它不是本地落库查询，而是对远端 API 的透传或封装；用 `Gateway` 比用 `Repo` 更准确，例如 `PaymentGateway`、`CloudGateway`。

### 二、Dialer 抽象模式

- **核心原则** 把“会话 / 连接”和“资源查询”解耦：“怎么连 ESXi”只写在 `Dialer` 里，“查什么资源”各写各的；以后要加 session 池、连接复用、证书 pin、vCenter 切换，也只改这一层。

- **反面设计** 每个资源接口自己登录 ESXi，会导致登录逻辑在 `vm.go`、`datastore.go`、`cluster.go` 等文件里重复出现；后续加证书验证、改 URL 格式、换连接方式，都要改多个文件，漏改一处就是 bug。

    ```go
    // 文件 1：vm.go
    func ListVMs(host *EsxiHost) ([]*VM, error) {
        u, _ := url.Parse("https://" + host.Host + "/sdk")
        u.User = url.UserPassword(host.Username, host.Password)
        c, err := govmomi.NewClient(ctx, u, true)
        if err != nil {
            return nil, err
        }
        defer c.Logout(ctx)

        return queryVMs(c)
    }

    // 文件 2：datastore.go
    func ListDatastores(host *EsxiHost) ([]*Datastore, error) {
        u, _ := url.Parse("https://" + host.Host + "/sdk")
        u.User = url.UserPassword(host.Username, host.Password)
        c, err := govmomi.NewClient(ctx, u, true)
        if err != nil {
            return nil, err
        }
        defer c.Logout(ctx)

        return queryDatastores(c)
    }
    ```

- **正面设计** 把登录逻辑抽到 `Dial`，资源查询函数只负责查询自己的资源；未来要加 session 缓存、换 vCenter、改超时，只改 `Dial` 函数，`vm.go` 和 `datastore.go` 不需要动。

    ```go
    // 文件 1：dialer.go
    func Dial(host *EsxiHost) (*govmomi.Client, error) {
        u, _ := url.Parse("https://" + host.Host + "/sdk")
        u.User = url.UserPassword(host.Username, host.Password)
        return govmomi.NewClient(ctx, u, true)
    }

    // 文件 2：vm.go
    func ListVMs(host *EsxiHost) ([]*VM, error) {
        c, err := Dial(host)
        if err != nil {
            return nil, err
        }
        defer c.Logout(ctx)

        return queryVMs(c)
    }

    // 文件 3：datastore.go
    func ListDatastores(host *EsxiHost) ([]*Datastore, error) {
        c, err := Dial(host)
        if err != nil {
            return nil, err
        }
        defer c.Logout(ctx)

        return queryDatastores(c)
    }
    ```

### 三、小接口原则

- **接口边界** 不要在 `biz` 层放一个超大的 `EsxiInventory` 接口，把 `ListVMs`、`ListDatastores`、`ListClusters` 全塞进去；这种大接口会让 service 层测试变重，也会让文件越来越难维护，只想测 VM 逻辑时也被迫 mock 所有方法，多人同时修改时更容易产生 merge conflict。

- **推荐做法** 按资源或用例拆小接口，每个接口只保留调用方真正需要的方法，写起来清楚，测起来简单。

- **反面设计** 业务逻辑只需要查 VM，却依赖了一个包含所有资源查询能力的大接口；测试时为了实现这个接口，被迫补齐一堆当前用不到的方法。

    ```go
    type EsxiGateway interface {
        ListVMs(ctx context.Context, hostID string) ([]*VM, error)
        ListDatastores(ctx context.Context, hostID string) ([]*Datastore, error)
        ListHosts(ctx context.Context) ([]*Host, error)
    }
    ```

- **正面设计** 按资源定义最小依赖面，把 VM、Datastore、Host 拆成三个小接口；调用方只依赖自己需要的接口，测试时也只 mock 当前用到的能力。

    ```go
    type VMGateway interface {
        ListVMs(ctx context.Context, hostID string) ([]*VM, error)
    }

    type DatastoreGateway interface {
        ListDatastores(ctx context.Context, hostID string) ([]*Datastore, error)
    }

    type HostGateway interface {
        ListHosts(ctx context.Context) ([]*Host, error)
    }
    ```
