# 如何将 Windows 从 VMware 迁移到 KVM（FusionCompute）

### 迁移后蓝屏的原因与关键条件

- **迁移风险** 当通过逐字节克隆磁盘的方式，将 Windows Server 2012 R2 虚拟机从 VMware ESXi 迁移到基于 KVM 的虚拟机管理程序（例如华为 FusionCompute）时，生成的虚拟机很可能会在启动时失败，并出现 `STOP 0x0000007B INACCESSIBLE_BOOT_DEVICE` 错误。

- **根本原因** Windows 默认没有内置 KVM VirtIO 半虚拟化存储控制器所需的驱动。启动早期，内核必须先读取自己的启动卷；如果它无法识别承载启动卷的磁盘控制器，就无法挂载系统盘，也就会触发 0x7B 蓝屏。

- **关键条件** 迁移成功的关键不是网络、气球驱动或其他增强功能，而是启动阶段能否加载正确的 VirtIO 存储驱动。

### 启动关键驱动的判定与分类

- **加载时机** 在 NT 内核初始化的极早期阶段（即 `NTLDR/winload.exe → ntoskrnl.exe` 的流程中），内核必须在挂载任何文件系统之前，加载所有服务条目中 `Start` 字段设置为 `0`（即 `SERVICE_BOOT_START`）的驱动程序。

- **失败机制** 读取启动卷本身的操作依赖这些驱动。如果磁盘控制器驱动程序缺失，或者没有被标记为 boot-start，内核将无法枚举启动磁盘，并立即触发 `STOP 0x0000007B INACCESSIBLE_BOOT_DEVICE`。

- **关键类型** 在 VMware 迁移到 KVM 的场景中，真正属于“启动关键”的只有一类驱动：存储 HBA（主机总线适配器）驱动。

- **控制器差异** VMware 环境中，磁盘通常通过 LSI_SAS 或 pvscsi（VMware 半虚拟化 SCSI）驱动呈现给系统；KVM 环境中，磁盘则通过 `viostor`（VirtIO-Block）或 `vioscsi`（VirtIO-SCSI）驱动呈现。

- **非关键驱动** 网络驱动（`netkvm`）、Balloon 驱动、串口驱动、RNG 驱动以及 qemu-pci-serial 驱动都不是启动关键驱动。它们通常在操作系统启动完成后才加载，只影响系统功能性，不决定系统能否启动。

- **驱动对照** 不同 VirtIO 驱动对启动过程的影响不同，存储类驱动必须优先处理。

    | 驱动 | 功能 | 是否启动关键 | 是否建议迁移前注入 |
    | --- | --- | --- | --- |
    | `vioscsi` | VirtIO-SCSI 存储 HBA | 如果磁盘挂在 virtio-scsi 上则是 | 是 |
    | `viostor` | VirtIO-Block 存储驱动 | 如果磁盘挂在 virtio-blk 上则是 | 是 |
    | `netkvm` | VirtIO 网卡驱动 | 否 | 建议，否则首次启动后可能没有网络 |
    | `Balloon` / `vioballoon` | 内存气球驱动 | 否 | 建议 |
    | `vioserial` | VirtIO serial / qemu-ga 通道 | 否 | 可选 |
    | `viorng` | VirtIO RNG | 否 | 可选 |
    | `qemupciserial` | QEMU PCI serial | 否 | 可选 |

### KVM 首次启动前必须满足的三个条件

- **启动前提** 为了确保 Windows 能在 KVM 环境下完成首次启动，VirtIO 存储驱动必须同时满足三个条件。

- **驱动文件** `viostor.sys` 和/或 `vioscsi.sys` 必须实际存在于 `C:\Windows\System32\drivers\` 目录下。若缺少二进制文件，SCM 将无法在启动时找到对应的 `ImagePath`。

- **服务条目** 驱动必须注册为 Windows 内核服务。注册表路径 `HKLM\SYSTEM\CurrentControlSet\Services\<driver>` 下必须存在对应服务定义。

- **服务定义** 对于 `vioscsi`，上游 INF 文件及社区脚本生成的服务定义大致如下：

    ```reg
    [HKLM\SYSTEM\CurrentControlSet\Services\vioscsi]
    "Type"=dword:00000001
    "Start"=dword:00000000
    "ErrorControl"=dword:00000001
    "ImagePath"=hex(2):"System32\drivers\vioscsi.sys"
    "Group"="SCSI miniport"
    "Tag"=dword:00000040
    ```

- **关键字段** `viostor` 也是同样的结构。其中最关键的值是 `Start = 0`。正是这个设置确保 Windows 能在磁盘子系统上线之前加载该驱动。如果遗漏这一项，即使驱动文件已经存在于磁盘上，Windows 也不会在启动阶段加载它。

- **PnP 绑定** 驱动必须完成 PnP 绑定。内核需要知道某个启动驱动应该绑定到哪一个具体设备。

- **旧版机制** 在 Windows XP/2003/2008/2008 R2/Vista/7 中，这类绑定主要通过 `HKLM\SYSTEM\CurrentControlSet\Control\CriticalDeviceDatabase` 实现。

- **新版机制** 从 Windows 8 / Server 2012 起，微软将这类映射关系从 `CriticalDeviceDatabase` 移出，改为放入 `HKLM\SYSTEM\DriverDatabase`，也就是与 `C:\Windows\System32\DriverStore\FileRepository\` 对应的离线驱动程序存储体系。

- **可靠方案** 对于 Windows Server 2012 R2，更可靠的安装方式是使用 `pnputil` 或 `DISM /Add-Driver`，而不是手动编辑 `.reg` 文件。前者能够同时填充 DriverStore、DriverDatabase，并创建属性正确的 `Services` 注册表项。

### VirtIO 存储驱动的选择与注入策略

- **驱动范围** `virtio-win` 提供了两个存储驱动程序：`viostor.sys` 和 `vioscsi.sys`。

- **VirtIO Block** `viostor.sys`（VirtIO Block）表现为一个 "Red Hat VirtIO SCSI 控制器" 磁盘节点，但本质上是块设备驱动。这是较旧、较简单的实现；不支持 SCSI 命令透传，旧版本中也不支持 TRIM。

- **VirtIO SCSI** `vioscsi.sys`（VirtIO SCSI）作为真正的 SCSI HBA 运行，支持多个 LUN、SCSI 命令透传、热插拔以及 TRIM/UNMAP。对于 Windows Server 2012 R2 及更新版本，更推荐使用 `vioscsi`。

- **注入策略** 实际需要注入哪个驱动，取决于 FusionCompute / KVM 平台最终通过哪种总线类型呈现磁盘设备。由于迁移前往往无法预知管理员会将磁盘挂载到 virtio-blk 还是 virtio-scsi 控制器上，最稳妥的做法是在磁盘脱离 ESXi 环境前，同时预安装 `viostor` 和 `vioscsi`。

- **辅助驱动** 同时，也可以一并安装 `netkvm`（网络驱动）和 `Balloon`（内存气球驱动），以保证首次启动后的可用性。

### virtio-win 驱动程序包的获取与分发

- **项目来源** virtio-win 是上游开源项目，由 Red Hat 维护，构建地址为 `https://github.com/virtio-win/kvm-guest-drivers-windows`，负责生成经 WHQL / Red Hat 签名的驱动程序包。

- **分发形式** virtio-win 程序包主要以下列形式分发。

    - 一个 ISO 镜像文件 `virtio-win.iso`，其目录结构呈树状，每个驱动程序对应一个文件夹，每个文件夹下又按 Windows 版本（`2k12R2/`、`2k16/`、`w10/` 等）和架构（`amd64/`、`x86/`）划分出相应的子文件夹。
    - 一个 MSI 安装包 `virtio-win-gt-x64.msi`，用于安装所有驱动程序以及 `qemu-ga`。
    - 一个"访客工具"（Guest Tools）的 `.exe` 可执行文件。

- **目录选择** Windows Server 2012 R2 应选用 `2k12R2/amd64` 子目录。

### 驱动安装方法对比（Pnputil / DISM / 手工注册表）

- **首选方法** 针对 Server 2012 R2，在虚拟机仍运行于 ESXi 之上时执行在线 `pnputil`，是最可靠的方法。

- **Pnputil 效果** 在运行中的 Windows 上执行 `pnputil /add-driver <inf> /install`，会以原子操作的方式完成所有工作：复制驱动文件到 DriverStore，填充 DriverDatabase，创建包含正确 `Start`、`Type`、`ErrorControl`、`Group`、`ImagePath` 值的 `Services\<drv>` 注册表项，并注册 PnP 绑定。

- **DISM 替代** `DISM /Online /Add-Driver` 也能实现等效效果，是常用的替代方案。

- **验证方式** 在正式启动磁盘迁移操作之前，可通过"设备管理器"或 PowerShell 命令 `Get-PnpDevice` 即时验证驱动是否成功注册。

- **常见误区** 仅复制 `.sys` 文件而不通过 `pnputil` 执行规范安装是无效的。驱动注册必须走完整的服务创建与 PnP 绑定流程。

- **手工风险** 使用 `reg load` 配合手写 `.reg` 补丁虽然可能奏效，但在 Windows Server 2012 R2 上更脆弱：关键设备映射已从 `CriticalDeviceDatabase` 迁移至 `DriverDatabase`，且系统枚举设备时还会校验 `.cat` 文件和 Authenticode 签名链。

- **手动补丁** 通过 `regedit /s` 导入注册表补丁属于旧式 virt-v2v 技术，主要针对 Server 2012 之前的系统设计，在 2012 R2 及后续版本中稳定性差且易出错。仅当 pnputil / DISM 均不可用时，才考虑此手段。

- **自动处理** 使用 `DISM` 或 `pnputil` 时，Windows 会自动处理驱动存储、签名验证、`.cat` 文件以及 PnP 绑定，无需人工干预。

### 推荐预安装与 PowerShell 脚本

- **推荐做法** Red Hat、Proxmox、Platform9、Nutanix、Microsoft Q&A 以及华为云的建议基本一致：在关闭源虚拟机电源之前，先在运行中的 Windows 虚拟机内安装完整的 `virtio-win` 驱动包。

- **有效原因** 即使当前系统中还没有匹配的 VirtIO 硬件设备，Windows 仍然可以把 VirtIO 驱动注册进驱动存储库，并创建相应的 `vioscsi` / `viostor` 服务条目。

- **脚本用途** 以下脚本在源 VMware 虚拟机仍处于运行状态时，预安装 VirtIO 驱动、确认启动关键服务、调整 SAN 策略，并安排首次启动后的清理任务。

    ```pwsh
    # 1. Stage virtio-win — either copy the ISO via VMware Tools FileTransfer
    #    or download in-guest (admin chooses). Mount the ISO.
    $iso = "C:\stage\virtio-win.iso"
    Mount-DiskImage -ImagePath $iso
    $drv = (Get-DiskImage $iso | Get-Volume).DriveLetter + ":"

    # 2. Pre-install the boot-critical storage driver (BOTH variants, for safety)
    pnputil /add-driver "$drv\vioscsi\2k12R2\amd64\vioscsi.inf" /install
    pnputil /add-driver "$drv\viostor\2k12R2\amd64\viostor.inf" /install

    # 3. Pre-install network + balloon + serial (non-boot-critical but useful)
    pnputil /add-driver "$drv\NetKVM\2k12R2\amd64\netkvm.inf"  /install
    pnputil /add-driver "$drv\Balloon\2k12R2\amd64\balloon.inf" /install
    pnputil /add-driver "$drv\vioserial\2k12R2\amd64\vioser.inf" /install
    pnputil /add-driver "$drv\viorng\2k12R2\amd64\viorng.inf"  /install

    # 4. Belt-and-braces: force boot-start on the storage services
    #    (pnputil should already have done this, but verify)
    $paths = @(
      "HKLM:\SYSTEM\CurrentControlSet\Services\vioscsi",
      "HKLM:\SYSTEM\CurrentControlSet\Services\viostor"
    )
    foreach ($p in $paths) {
      if (Test-Path $p) {
        Set-ItemProperty $p -Name Start        -Type DWord -Value 0
        Set-ItemProperty $p -Name Type         -Type DWord -Value 1
        Set-ItemProperty $p -Name ErrorControl -Type DWord -Value 1
        Set-ItemProperty $p -Name Group        -Type String -Value "SCSI miniport"
      }
    }

    # 5. Disable SAN policy so secondary disks come online automatically on KVM
    "san policy=OnlineAll" | diskpart

    # 6. Optional: install QEMU Guest Agent so FusionCompute can do quiesced
    #    snapshots / graceful shutdown after migration
    Start-Process msiexec -Wait -ArgumentList "/i `"$drv\guest-agent\qemu-ga-x86_64.msi`" /qn"

    # 7. (recommended) Schedule a first-boot cleanup task that uninstalls VMware
    #    Tools and re-runs PnP enumeration once we are on KVM
    $cmd = 'cmd /c "MsiExec.exe /x {VMware-Tools-MSI-GUID} /qn /norestart & pnputil /scan-devices"'
    schtasks /Create /SC ONSTART /RU SYSTEM /TN "PostMigrate" /TR $cmd /F

    Dismount-DiskImage -ImagePath $iso
    shutdown /s /t 0
    ```

### 离线 DISM 驱动注入（适用于无法在线安装的场景）

- **适用场景** 如果无法将脚本推送到正在运行的客户机中，而只能从 Linux 或 Windows 暂存主机上挂载克隆的 VHD/qcow2 镜像，那么相应的操作就是"离线 DISM"。

- **驱动注入** 离线镜像可以通过以下 DISM 命令注入 VirtIO 驱动。

    ```cmd
    dism /Image:E:\ /Add-Driver /Driver:F:\vioscsi\2k12R2\amd64\vioscsi.inf
    dism /Image:E:\ /Add-Driver /Driver:F:\viostor\2k12R2\amd64\viostor.inf
    dism /Image:E:\ /Add-Driver /Driver:F:\NetKVM\2k12R2\amd64\netkvm.inf
    dism /Image:E:\ /Add-Driver /Driver:F:\Balloon\2k12R2\amd64\balloon.inf
    ```

- **执行效果** 针对离线镜像运行 `DISM /Add-Driver` 命令，不仅会将驱动程序填充至 DriverStore，还会创建相应的服务注册表项，并将其 `Start` 值正确设置为 `0`；其效果与在线状态下运行 `pnputil` 工具完全一致。

- **救援场景** 如果在此之后镜像仍无法启动，恢复步骤是在 WinPE 环境中手动加载 SYSTEM 配置单元，并移除其中相互冲突的旧驱动程序引用。

    ```cmd
    reg load HKLM\Temp E:\Windows\System32\config\SYSTEM
    reg delete HKLM\Temp\ControlSet001\Services\viostor /f
    reg delete HKLM\Temp\ControlSet001\Services\vioscsi /f
    reg unload HKLM\Temp
    dism /Image:E:\ /Add-Driver /Driver:F:\amd64\viostor.inf
    dism /Image:E:\ /Add-Driver /Driver:F:\amd64\vioscsi.inf
    ```

### Boot Loader 与固件配置 

- **BIOS 模式** BIOS/MBR 模式下，`\boot\BCD` 文件位于"系统保留分区"中。迁移完成后，BIOS 仍能找到 MBR，而 NTLDR/bootmgr 也仍会从原路径加载 `winload.exe`。只要 KVM 配置使用的是 BIOS（这是 FusionCompute 中非 UEFI 虚拟机的默认设置），就不需要对 BCD 进行任何修改；此时唯一可能出现的故障代码是 0x7B，这是由于缺少存储驱动程序所致。

- **UEFI 模式** UEFI/GPT 模式下，EFI 系统分区中包含 `\EFI\Microsoft\Boot\bootmgfw.efi` 文件。KVM/QEMU 必须通过 OVMF（即 `pve-edk2-firmware`，对应 FusionCompute 中的 UEFI 选项）启动，且 NVRAM 变量必须正确指向该 EFI 二进制文件。在转换过程中，`virt-v2v` 工具会针对 UEFI Windows 虚拟机明确执行"修复 EFI 启动配置"的操作。如果目标虚拟机在创建时未配置 UEFI 固件，而源磁盘却是 GPT/UEFI 格式，那么你会收到"无启动设备"（no bootable device）的错误提示，而非 0x7B 错误；此时的解决方法是为 FusionCompute 虚拟机挂载 EFI/OVMF 固件，而不是尝试注入更多的驱动程序。

- **BCD 重建** 通常情况下，如果仅更换了磁盘控制器，无需使用 `bcdboot` 命令重建 BCD。这是因为 BCD 是通过 GPT-GUID 或 MBR 签名来记录操作系统分区的，而非依据总线类型；因此，只要该分区依然可读（即存储驱动程序能够正常加载），Windows 就能自动识别并找到它。仅当在迁移过程中 BCD 本身发生损坏时，执行 `bcdboot C:\Windows` 才作为一种恢复手段被启用。

### VMware 组件的清理策略

- **移除时机** VMware Tools 的移除工作不应在迁移之前进行。在 ESXi 环境下运行中的虚拟机上直接移除 VMware Tools，可能导致该虚拟机丧失控制台/键盘功能。

- **保留影响** 如果在迁移后未移除 VMware Tools，虽可能引发错误并导致性能下降，但不会阻碍虚拟机正常启动。

- **组件范围** 从 ESXi 导出的 Server 2012 R2 镜像通常包含 VMware 存储驱动、Microsoft 内置存储驱动，以及 VMware Tools 相关的用户态服务和辅助驱动。

- **pvscsi 驱动** pvscsi（VMware 半虚拟化 SCSI 驱动程序）是一个 `Start=0` 的服务，在迁移过程中应保留原位。它在 KVM 环境下是无害的，因为没有任何 PCI 设备与它的即插即用（PnP）ID 匹配，因此该驱动程序不会被加载。若在迁移前将其移除，则存在风险：如果在注入 `vioscsi` 驱动之前移除了唯一一个 `Start=0` 的存储服务，该虚拟机（VM）将无法在任何一种虚拟机管理程序（Hypervisor）上启动。

- **内置驱动** LSI_SAS、storahci、intelide 是 Microsoft 系统内置（inbox）的驱动程序，保留它们是无害的。

- **VMware Tools** VMTools、vmci、vsock、vmhgfs、vmxnet3 属于 VMware Tools 的用户模式服务及辅助驱动程序。不应在虚拟机仍运行于 ESXi 环境时就卸载 VMware Tools。若在未安装 Tools 的状态下运行，你将失去用于推送迁移脚本的 VMware GuestOps / RPC 通道。因此，更推荐的做法是安排一个"首次启动卸载"任务，使其在虚拟机切换至 KVM 环境并首次启动时自动执行。

- **卸载时机** 如果确实选择在虚拟机仍处于 ESXi 环境时卸载 Tools，务必将其作为关机前的"最后一步"来执行，以确保你始终保有远程执行操作的通道。

### FusionCompute 平台适配

- **平台架构** 华为 FusionCompute 架构基于华为定制的 SUSE/KVM 堆栈（即"UVP"）构建，并通过标准的上游 VirtIO 接口向虚拟机（Guest OS）提供半虚拟化设备。

- **官方要求** 华为官方的镜像制作文档明确要求：在将 Windows 镜像转换为 FusionCompute 或华为云的私有镜像之前，必须预先在虚拟机内部安装 VirtIO 驱动程序；文档中也指引管理员使用标准的 virtio-win MSI 安装包进行操作。

- **磁盘控制器** 在实际运行中，FusionCompute 默认将系统磁盘挂载至 VirtIO-SCSI 控制器（适用于较新版本的 FusionCompute），但也同样支持 VirtIO-Block 模式；正因如此，在制作镜像时同时注入 `vioscsi` 和 `viostor` 这两类驱动，是最稳妥且通用的默认配置方案。

- **驱动来源** FusionCompute 自身并不提供专用的磁盘驱动程序，其采用的是标准的上游 VirtIO 驱动。GitHub 上托管的旧版华为"UVP-Tools"工具集，是专为基于 Xen 架构的早期 FusionCompute 版本设计的，其中提供的是 Xen 前端半虚拟化驱动（如 xen-vbd、xen-vnif 等），而非 VirtIO 驱动。对于当前基于 KVM 架构的 FusionCompute 版本，应选用 virtio-win 驱动包，而非 UVP-Tools。

### 整体迁移流程

- **源端状态** 在 VMware 环境中，虚拟机应保持运行状态，以便通过 VMware Tools GuestOps / RPC 通道推送文件和执行脚本。

- **文件推送** 通过 VMware Tools 的文件传输功能（FileTransfer），将 `virtio-win.iso` 镜像（或其解压后的文件树）推送到客户机中。

- **驱动安装** PowerShell 脚本负载针对 `vioscsi`、`viostor`、`netkvm`、`balloon`、`vioserial` 和 `viorng` 驱动执行 `pnputil /add-driver` 命令。

- **注册表验证** 在注册表中执行 `Get-ItemProperty HKLM:\SYSTEM\CurrentControlSet\Services\vioscsi` 命令，确认其显示 `Start=0`、`Type=1`、`ErrorControl=1`、`Group="SCSI miniport"` 以及 `ImagePath="System32\drivers\vioscsi.sys"`。对 `viostor` 驱动也执行同样的验证。

- **SAN 策略** 运行 `diskpart` 命令，并执行 `san policy=OnlineAll` 设置。

- **首次任务** 在"任务计划程序"中创建一个仅在首次启动时运行的任务，用于卸载 VMware Tools 并执行 `pnputil /scan-devices` 命令。

- **可选代理** 可选安装 `qemu-ga.msi` 代理程序，以便 FusionCompute 平台能够执行优雅关机（Graceful Shutdown）或静默快照（Quiesced Snapshots）操作。

- **磁盘迁移** 执行块级磁盘数据迁移，将磁盘数据传输至 FusionCompute 平台的目标卷中。

- **目标建机** 在 FusionCompute 平台上创建虚拟机，并确保其固件类型与源虚拟机保持一致，即同为 BIOS/SeaBIOS 或 UEFI/OVMF；将迁移后的磁盘挂载至 VirtIO-SCSI 控制器下，如果标准配置是 VirtIO-Block，则挂载至该控制器下；最后为虚拟机配置一块 VirtIO-Net 网卡。

- **首次启动** 首次启动虚拟机。由于驱动程序已预先部署，Windows 内核在初始化阶段即可识别并加载用于启动磁盘的 `vioscsi` 驱动，随后挂载系统卷并完成启动流程；紧接着，系统将通过即插即用（PnP）机制自动安装 `netkvm`、`Balloon`、`serial` 和 `RNG` 驱动。此时，此前预设的计划任务将自动运行，并执行卸载 VMware Tools 的操作。

- **最终验证** 打开"设备管理器"进行确认：确保没有任何带有黄色感叹号的设备；同时确认"Red Hat VirtIO SCSI controller"、"Red Hat VirtIO Ethernet Adapter"以及"VirtIO Balloon Driver"均已正常显示。再次重启虚拟机以确认系统状态完全正常且无残留问题，随后即可基于该虚拟机创建镜像或快照。

- **流程本质** 这一流程的本质是在运行中的源系统上预置 VirtIO 驱动程序作为启动服务，使即插即用（PnP）机制能够在目标系统首次启动时自动将其绑定，并将 VMware Tools 的移除及收尾工作推迟至首次启动脚本中执行。这正是 virt-v2v 工具在生产环境中沿用了十五年之久的算法。

### virt-v2v 的参考工作流

- **参考价值** 通过研读 virt-v2v 的源代码，可以提炼出完成此类迁移所需的"最少且正确"的必要步骤。

- **镜像检查** 对于 Windows 访客机，virt-v2v 会先利用 `libguestfs` / `virt-inspector` 检查磁盘镜像，定位 Windows 安装目录、SYSTEM 注册表配置单元（hive）以及固件类型（BIOS 或 UEFI）。

- **驱动复制** virt-v2v 会读取 `virtio-win` 镜像文件（通常位于 `/usr/share/virtio-win/virtio-win.iso`），并将匹配的 `viostor.sys`、`vioscsi.sys`、`netkvm.sys` 和 `Balloon.sys` 文件复制到 `Windows/System32/drivers/` 目录下。

- **服务注册** virt-v2v 会利用 `hivex`（也就是 `virt-win-reg` 使用的同一库）以离线模式加载 SYSTEM 注册表配置单元，并在 `Services\vioscsi` 和/或 `Services\viostor` 键下写入 `Start=0`、`Type=1`、`ErrorControl=1`、`Group="SCSI miniport"`、`ImagePath=System32\drivers\<drv>.sys` 等配置值。

- **设备绑定** 为了兼容旧版本 Windows，它还会写入匹配的 `CriticalDeviceDatabase` PCI 绑定条目；同时，它也会在 `DriverDatabase` 下添加记录，以确保 Windows 2012、2012 R2、2016 及更高版本在执行即插即用（PnP）检测时，能够自动识别并加载该驱动程序。

- **驱动存储** virt-v2v 会将 VirtIO INF 和 CAB 文件放置到 `C:\Windows\INF\` 目录下，并更新 `DriverDatabase`，使 Windows 首次启动时能够完成驱动程序存储库（Driver Store）的安装过程。

- **工具卸载** 对于 VMware Tools、Xen PV 驱动程序、Parallels Tools 等原平台工具，virt-v2v 只会检测并准备卸载，不会在离线状态下直接卸载。实际卸载动作会被放入 firstboot script，因为这类 MSI 安装程序通常要求操作系统处于运行状态。

- **首次启动** virt-v2v 会安装一个"首次启动服务"。当 Windows 第一次在 KVM 环境中启动时，该服务会执行一次性脚本，完成驱动程序的最终安装、移除原宿主机工具、让新增磁盘设备联机；如果客户机使用 UEFI 启动，还会修复 BCD / EFI 启动配置。

- **手动等价** 如果不使用 virt-v2v，就需要通过 VMware Tools GuestOps 手动执行上述驱动复制、驱动注册、PnP 绑定、firstboot 清理等步骤；如果源虚拟机采用 UEFI 模式，还需要额外处理 BCD / EFI 启动配置。

### 开源自动化迁移工具

- **工具范围** 目前已有几款成熟工具能够封装并自动化执行上述操作。这些工具均不依赖 VMware Tools GuestOps（即访客操作系统内部操作），而是针对离线磁盘进行操作。

- **virt-v2v** virt-v2v（libguestfs 项目；Red Hat 出品）是该领域的标准工具。它能够读取 VMware 虚拟机（通过 vCenter API、SSH 传输的 `.vmx` 文件，或本地 OVA 文件），检测 Windows 访客操作系统环境，注入 `vioscsi` / `viostor` / `netkvm` / `Balloon` 驱动，修复注册表，将 VMware Tools 的卸载操作配置为"首次启动脚本"（firstboot script），修复 UEFI BCD 引导配置，并将处理结果写入 libvirt、RHV、OpenStack 环境，或保存为本地 raw/qcow2 格式镜像。该工具已收录于 RHEL、CentOS、Ubuntu 的软件仓库中，源码地址为 `https://github.com/libguestfs/virt-v2v`。

- **libguestfs 模块** `virt-customize` / `virt-win-reg` / `guestfish` 同样属于 libguestfs 项目，是更底层的构建模块，允许用户通过脚本实现上述相同操作，例如上传 `viostor.sys` 驱动文件、将 `.reg` 注册表片段合并到离线状态的 SYSTEM 注册表配置单元（hive）中，或安装首次启动脚本。

- **MTV Forklift** MTV / Forklift（Migration Toolkit for Virtualization，虚拟化迁移工具包）是 Red Hat 与 KubeVirt 社区合作的项目，它将 virt-v2v 工具封装为一个 Kubernetes Operator，专用于实现 VMware 虚拟机向 OpenShift Virtualization 环境的大规模批量迁移。

- **在线方案** 对于基于 GuestOps、在线且通过脚本驱动的方案，目前尚无广为人知的"开箱即用"型开源工具。virt-v2v 之所以针对处于关机状态的虚拟机（Guest）进行操作，是因为它最初专为"冷迁移"场景设计。内部开发的 PowerShell 封装脚本（Wrapper）在设计思路上是正确的；本质上是在重新实现 virt-v2v 工具中负责处理"虚拟机内部（in-guest）"事务的那一部分功能。

### 常见故障模式与预防措施

- **0x7B 蓝屏** 在 KVM 虚拟机上首次启动时出现 `STOP 0x7B (INACCESSIBLE_BOOT_DEVICE)` 蓝屏错误，通常涉及 `vioscsi` / `viostor` 驱动。常见原因是驱动文件未存入磁盘、服务启动类型 `Start` 未设为 `0`，或 PnP 未成功绑定到 `VEN_1AF4` PCI ID。预防方法是使用 `pnputil` / `DISM` 注入驱动程序，验证注册表路径 `Services\vioscsi\Start` 的值是否为 `0`，并确认 `DriverDatabase`（Server 2012 及更高版本）或 `CriticalDeviceDatabase`（≤ Server 2008 R2）中已将 PCI ID `PCI\VEN_1AF4&DEV_1004/1048` 正确映射至 `vioscsi` 驱动。

- **加载过晚** 系统启动后卡在"旋转圆圈"界面，且仅当磁盘以 IDE/SATA 模式挂载时才能正常启动，通常说明 VirtIO 驱动加载过晚，或其启动类型 `Start` 被设为 `3 (DEMAND_START)`。预防方法是强制将存储服务（`vioscsi` / `viostor`）的启动类型 `Start` 设为 `0`，并通过 `pnputil` 重新安装驱动程序，以便 PnP 机制重建 `DriverDatabase`。

- **数据盘脱机** 首次启动正常，但在"磁盘管理"中数据盘显示为"脱机"（Offline）状态，通常是因为 Windows 的 SAN 策略将连接在"不同总线"上的非系统磁盘默认视为脱机。当存储控制器从 VMware 半虚拟化 SCSI 切换为 virtio-scsi 后，辅助数据盘可能显示为脱机或只读。预防方法是关机前运行 `diskpart` 执行 `san policy=OnlineAll`；这正是 virt-v2v 在"磁盘配置"步骤中所做的操作。

- **网络不可用** 系统启动正常但无法访问网络，通常是因为 `netkvm` 驱动未预装，或虚拟机仅配置了 e1000 类型的网卡。处理方式是安装 `netkvm` 驱动程序；也可以暂时挂载一块 e1000 类型的网卡，利用 Server 2012 R2 系统内置的 e1000 驱动先恢复网络连接，随后再添加 VirtIO 网卡并安装 `netkvm` 驱动。

- **黄色感叹号** 驱动安装程序提示成功，但在 ESXi 平台虚拟机的"设备管理器"中，VirtIO 设备旁显示黄色感叹号，这是正常现象。原因是在 ESXi 平台上并不存在真实的 VirtIO 硬件，驱动程序在此仅用于注册相应的服务；这也确认了预部署（pre-stage）阶段已成功完成。

- **UEFI 配置** 即使已安装 `vioscsi` 驱动，若目标 UEFI 虚拟机未配置 UEFI/OVMF 固件，或 EFI 系统分区 ID 发生变更，仍可能出现 0x7B 错误。解决方法是重新创建配置了 UEFI 固件的 FusionCompute 虚拟机；随后在 WinPE 环境下执行 `bcdboot C:\Windows /s S: /f UEFI` 重建 BCD 引导配置。

- **签名证书** 若使用较新版本的 virtio-win（≥0.1.225），可能会遭遇 0x7B 错误。原因是该驱动程序由 Red Hat CA 证书链签名，而离线状态下的 Windows Server 2012 R2 信任存储无法对其进行验证。建议将驱动版本固定至已知可顺利安装的旧版本（例如 0.1.215 或 0.1.189），或先在源系统上运行 Windows Update，以更新其受信任根证书列表。

- **安全启动** 启动后不久即发生蓝屏（Bug-check），且仅在启用"安全启动"（Secure Boot）时出现，通常是因为特定版本的 virtio-win 驱动链未被 UEFI 安全启动数据库（db）信任。解决方法是禁用"安全启动"，或使用具有正确 SHA-256 签名的特定 0.1.x 版本。
