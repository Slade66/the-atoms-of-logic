## 二、根因分析

**直接原因**：[go.mod](cci:7://file:///a:/Users/liyuz/Desktop/bifrost/go.mod:0:0-0:0) 第 3 行声明 `go 1.24.13`，Dockerfile 第 2 行 `FROM golang:1.24` 解析为 `golang:1.24.3`，版本不满足。

**为什么 `GOTOOLCHAIN=local` 阻止了自动升级**：Go 1.21+ 引入工具链自动下载机制，当 [go.mod](cci:7://file:///a:/Users/liyuz/Desktop/bifrost/go.mod:0:0-0:0) 要求更高版本时，默认行为是自动下载对应工具链。但 Docker 容器内 `GOTOOLCHAIN` 被设为 `local`（只使用本地已安装的 Go），导致自动下载被禁用，直接报错退出。

**为什么 `golang:1.24` 不是 1.24.13**：Docker Hub 的 `golang:1.24` tag 指向的是 1.24 分支的最新 patch release，但镜像构建时 1.24.13 尚未发布或 tag 未更新，实际指向 `1.24.3`。

**修复**：Dockerfile 基础镜像从 `golang:1.24` → `golang:1.24.13`，锁定精确版本。

---

## 三、以后如何操作

1. **[go.mod](cci:7://file:///a:/Users/liyuz/Desktop/bifrost/go.mod:0:0-0:0) 和 Dockerfile 版本必须对齐** — 每次 `go mod tidy` 升级 `go` 指令后，同步更新 Dockerfile 的 `FROM golang:X.Y.Z`，不要用模糊 tag（`1.24`），用精确版本（`1.24.13`）。

2. **可选：让 Docker 构建支持自动工具链下载** — 在 Dockerfile 的 `RUN go mod download` 前加：
   ```dockerfile
   ENV GOTOOLCHAIN=auto
   ```
   这样即使基础镜像版本略低，Go 也会自动下载 [go.mod](cci:7://file:///a:/Users/liyuz/Desktop/bifrost/go.mod:0:0-0:0) 要求的版本。代价是首次构建稍慢（需下载工具链），但后续有缓存。

3. **CI 前本地验证** — 推送前本地跑一次 `docker build -t bifrost:latest .`，能在 30 秒内发现此类版本不匹配问题，避免占 Jenkins 队列。

### 延伸

两个问题叠加：

node149 无法访问 Docker Hub — lookup registry-1.docker.io: i/o timeout，DNS 解析超时，国内网络直连 Docker Hub 受限
golang:1.24.13 需要拉新镜像 — 之前 golang:1.24 本地有缓存所以能跑，现在换 tag 需要重新拉取，拉不到就失败了
修复方案：回退到本地已缓存的 golang:1.24，加 GOTOOLCHAIN=auto 让 Go 自动下载 1.24.13 工具链。

原理：golang:1.24 本地有缓存不需要联网拉取；GOTOOLCHAIN=auto 让 Go 在 go mod download 时自动从 goproxy.cn 下载 1.24.13 工具链（走 Go module proxy，不走 Docker Hub）。
