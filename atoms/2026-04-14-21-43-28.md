# stderr 是什么？

## 一、基本定义

- **标准流** 操作系统为进程提供 `stdin`、`stdout`、`stderr` 三个标准数据流。

    | 流名称 | 文件描述符 | 核心用途 |
    | --- | --- | --- |
    | `stdin` | `0` | 接收输入 |
    | `stdout` | `1` | 输出正常结果，也就是数据通道 |
    | `stderr` | `2` | 输出错误、日志和诊断信息，也就是诊断通道 |

- **通用做法** `grep`、`curl`、`rsync` 这类命令行工具通常都会把正常输出写到 `stdout`，把错误、日志和诊断信息写到 `stderr`。

## 二、为什么要区分

- **职责分离** `stdout` 用于输出程序的正常结果，`stderr` 用于输出日志、错误和诊断信息。

    | 输出位置 | 用途 |
    | --- | --- |
    | `stdout` | 程序输出结果，也就是数据 |
    | `stderr` | 日志、错误、诊断信息 |

- **管道安全** 区分 `stdout` 和 `stderr` 后，错误信息不会混入正常数据流，后续程序读取管道时不会收到脏数据。

## 三、管道示例

- **管道规则** 管道 `|` 默认只接收 `stdout`，不会接收 `stderr`。

- **正常情况** 如果程序把数据写到 `stdout`，把日志写到 `stderr`，那么直接使用管道时，`consumer` 只会收到 `stdout` 数据，`stderr` 会继续显示在当前终端。

    ```bash
    ./vddk-service | ./consumer
    ```

- **污染情况** 真正的问题不是 `|` 会接收日志，而是程序错误地把日志也写进 `stdout`；这时管道会把“数据 + 日志”一起传给下游程序。

    ```bash
    # 假设 vddk-service 把日志也写到了 stdout
    ./vddk-service | ./consumer
    ```

- **丢弃日志** 如果日志正确走 `stderr`，可以用 `2>/dev/null` 丢弃诊断信息；这不会影响管道里的 `stdout` 数据。

    ```bash
    ./vddk-service 2>/dev/null | ./consumer
    ```

- **保存日志** 也可以用 `2>>app.log` 把 `stderr` 追加写入日志文件，同时继续把 `stdout` 传给 `consumer`。

    ```bash
    ./vddk-service 2>>app.log | ./consumer
    ```
