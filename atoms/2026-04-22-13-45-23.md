# 为什么要用 Canonical JSON 来做 License

### 一、核心论断：签名签的是字节，不是数据

- **验签函数只会比较字节** 数字签名签的是"字节"，不是"数据"。验签端不懂数据语义，字节不一致签名就对不上。

- **canonical JSON 的唯一目的** 给定同样的数据，永远产出同样的字节，让签发端和验签端得到一致的 `bytes` 去参与签名/验签。

### 二、同一份数据为什么会产出不同字节

- **五种都合法的写法** `{customer: "foo", quota: 50}` 在 JSON 规范下至少能写成这几种：

    ```json
    {"customer":"foo","quota":50}
    {"quota":50,"customer":"foo"}
    { "customer": "foo", "quota": 50 }
    {"customer":"foo","quota":50.0}
    {"customer":"foo","quota":50,}
    ```

- **根因一：key 顺序不固定** Go 里 `map[string]any` 遍历顺序是随机的，不同运行甚至不同版本可能输出不同顺序。

- **根因二：空白位置随意** JSON 规范允许 `:` / `,` 前后有任意空白，紧凑与带空格的写法字节完全不同。

- **后果** `bytes_A ≠ bytes_B` 即便语义相同，验签也必然失败。

### 三、canonical JSON 的规则

- **对象 key 按字典序排序** 永远按字典序顺序，不受输入顺序影响。

- **最小化空白** `:` / `,` 前后不留空格，等价于"紧凑模式"。

- **业界标准** RFC 8785（JCS, JSON Canonicalization Scheme）。

### 四、Go 里 "struct → `json.Marshal`" 为什么天然 canonical

- **输出顺序 = 结构体定义顺序** `encoding/json` 按字段在 struct 里的定义顺序输出，虽然不是字典序，但只要两端用同一份 struct 就有确定性。

- **默认紧凑输出** `json.Marshal` 默认无多余空格；只有 `json.MarshalIndent` 才加缩进。

- **bifrost 项目的做法** 签发端和验签端共享 `pkg/license/` 里的同一份 struct 定义，`json.Marshal(payload)` 永远产出相同字节，不用自己实现 key 排序。

### 五、为什么 YAML 不能直接签

- **YAML 语法比 JSON 还宽松**
    - 字符串加不加引号都合法：`customer: foo` 与 `customer: "foo"`。
    - 布尔 `yes` / `true` / `on` 都会被解析为 `true`，但原始字节不同。
    - 多行字符串有 `|` / `>` / `|-` / `>+` 四种风格。
    - 缩进 2 空格还是 4 空格都无所谓。

- **缺乏广泛实现的 canonical form** 不像 JSON 有 RFC 8785 这样的标准。

- **正确路径** `parse 到 struct → marshal 成 canonical JSON → 对 JSON 字节做签名`。

### 六、不做 canonical 会怎样

- **典型症状** 签发机器上签出来的 license 拿到客户机器上验签，随机成功或失败，表现为"玄学 bug"。

- **常见踩坑现场**
    - 签发用 Python `json.dumps()`（默认不排序 key），客户端用 Go `json.Marshal`（对 map 会排序）→ 永远验不过。
    - 签发端 Go 用 `struct`，客户端用 `map[string]any`→ 字段顺序不同 → 永远验不过。
    - payload 里嵌套了 `map`，Go 版本升级后 map 遍历顺序变化 → 某次升级后全体失败。

- **排查为什么难** 所有日志里"数据看起来一样"，但字节层面已经不一致，肉眼对不出来。
