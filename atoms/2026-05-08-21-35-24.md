# 为什么 SCP 失败

### 事件日志

迁移任务在 `Seed` 阶段失败，失败点为向 Helper VM 下发 `bifrost-receiver` 文件。

关键日志如下：

```text
[Seed] scp receiver: exit status 255
stderr:
Authorized users only. All activities may be monitored and reported.
subsystem request failed on channel 0
scp: Connection closed
```

现场在 Bifrost 容器内验证：

```bash
ssh root@99.30.7.133
```

可以正常登录 Helper VM，说明网络连通、SSH 账号和免密认证正常。

但在同一容器内执行：

```bash
scp bifrost-receiver root@99.30.7.133:/tmp/
```

失败并返回：

```text
subsystem request failed on channel 0
scp: Connection closed
```

### 根因分析

本次故障不是网络问题，也不是 SSH 免密问题。

根因是：Helper VM 的 SSH 服务未正确启用 SFTP 子系统，导致容器内执行 `scp` 时无法建立文件传输通道。

原因说明：

```text
1. Bifrost 程序部署 receiver/sender 文件时使用 scp 命令。
2. 当前容器内的 OpenSSH scp 默认使用 SFTP 子系统进行文件传输。
3. Helper VM 可以接受普通 SSH 登录，但无法响应 SFTP subsystem 请求。
4. 因此 ssh 命令可以成功，scp 命令失败。
```

所以现象表现为：

```text
ssh root@99.30.7.133 成功
scp root@99.30.7.133:/tmp 失败
```

### 解决方案

在 Helper VM `99.30.7.133` 上启用 SFTP 子系统。

麒麟 10 推荐使用 `internal-sftp`：

```bash
grep -n 'Subsystem.*sftp' /etc/ssh/sshd_config
```

如果没有配置，追加：

```bash
echo 'Subsystem sftp internal-sftp' >> /etc/ssh/sshd_config
```

如果已有旧配置，将其改为：

```text
Subsystem sftp internal-sftp
```

检查 SSH 配置语法：

```bash
sshd -t
```

无输出表示配置正确。

重启 SSH 服务：

```bash
systemctl restart sshd
```

最后在 Bifrost 容器内验证：

```bash
scp /app/build/bifrost-receiver root@99.30.7.133:/tmp/
```

验证成功后，重新发起迁移任务。
