# 虚拟机迁移难点：加密磁盘（LUKS / BitLocker）的兼容性挑战

- 在进行跨平台迁移（如 VMware 迁移至 FusionCompute）时，开启了磁盘加密（Windows BitLocker 或 Linux LUKS）的虚拟机是极为棘手的场景。

### 1. 核心瓶颈：无代理方案的“密文锁死”

- 若采用基于虚拟化层 API（如 vCenter API）的无代理（Agentless）迁移方式，整个流程会陷入以下技术封锁：

- **数据层密文**：迁移引擎从底层拉取的 `.vmdk` 镜像在物理块层面是完全加密的。

- **重构失败**：由于数据未解密，离线转换工具（如 `libguestfs`）无法挂载磁盘分区。

- **驱动注入受阻**：无法挂载意味着无法向系统盘注入必要的 VirtIO 驱动，导致迁移后的虚机因硬件变更且缺少驱动而无法启动。

### 2. 迁移策略建议

- 针对此类场景，技术选型必须做出明确调整：

- 首先明确不支持通过 API 的无代理方式迁移加密磁盘。

- **首选方案：Agent 代理模式**
    
    - 在源端虚拟机内部安装 Agent 代理程序。
    
    - **原理**：Agent 在操作系统内部运行，读取的是已经过逻辑解密的“明文块”，并将明文数据同步至目标端，从而完美绕过底层加密障碍。

- **规避方案：预处理解密**
    
    - 要求用户在迁移前暂时关闭并彻底解密 BitLocker 或 LUKS 系统盘。
    
    - 待迁移完成后，在目标环境再行开启加密。
