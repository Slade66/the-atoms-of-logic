# 重构代码的目的是什么？什么是好代码？

- **核心目的：** 改善代码本身，使其更好理解、更好维护。

- **关键前提：** 不改变外部行为，输入输出、对外接口都保持一致。

    - **不夹带功能变更：** 一边改结构边加功能，出了问题就分不清是哪一步引入的。

    - **测试兜底：** 用测试作为保护网，确保改完之后行为没变。

## 提升可读性

- **可读性（readability）：** 能不能看懂这段代码。它说的是阅读时的理解成本，靠的是好名字、短函数、清晰的结构。

- **一句话原则：** 代码是写给人看的，机器执行只是顺带。

- **为什么重要：** 代码被阅读的次数远多于被编写的次数，团队里别人读你代码的成本是真实存在的。好代码能让你和接手的同事在几个月后还能快速看懂逻辑，节省时间、精力和金钱（工资和 token 成本）。

### 做法一：单一职责（拆分大函数）

- **原则：** 一个函数只做一件事，只有一个“引起它变化的原因”——把几百行、什么都干的大函数拆成职责清晰的小函数。

- **Before：** `GenerateReport` 把统计、格式化、存储三件事揉在一起，这一个函数有至少三种变化原因。

    ```go
    func GenerateReport(orders []Order) (string, error) {
        // 变化原因 1：统计口径变了要改（比如以后要算中位数）
        var total float64
        for _, o := range orders {
            total += o.Amount
        }
        avg := total / float64(len(orders))

        // 变化原因 2：输出格式变了要改（比如以后要出 JSON 而不是文本）
        content := fmt.Sprintf("Total: %.2f\nAvg: %.2f", total, avg)

        // 变化原因 3：存储方式变了要改（比如以后要传 S3 而不是写本地）
        err := os.WriteFile("report.txt", []byte(content), 0644)
        return content, err
    }
    ```

- **After：** 一个函数一个变化原因，三件事解耦，`GenerateReport` 只剩编排。

    ```go
    // 只管算：统计口径变了只动这
    func summarize(orders []Order) Summary {
        var total float64
        for _, o := range orders {
            total += o.Amount
        }
        return Summary{Total: total, Avg: total / float64(len(orders))}
    }

    // 只管格式化：输出格式变了只动这
    func formatText(s Summary) string {
        return fmt.Sprintf("Total: %.2f\nAvg: %.2f", s.Total, s.Avg)
    }

    // 只管存：存储方式变了只动这
    func save(path, content string) error {
        return os.WriteFile(path, []byte(content), 0644)
    }

    // 编排：把三步串起来，自己不掺杂任何细节
    func GenerateReport(orders []Order) (string, error) {
        content := formatText(summarize(orders))
        return content, save("report.txt", content)
    }
    ```

### 做法二：命名见义

- **思路：** 把含糊的变量名改成有意义的名字，让名字自己把逻辑讲清楚。

- **Before：** `o.T == 1` 里的 `1` 是什么意思？鬼才知道。

    ```go
    func calc(o *Order) float64 {
        p := o.Amount
        if o.T == 1 {
            p = p * 0.9
        } else if o.T == 2 {
            p = p * 0.8
        }
        return p
    }
    ```

- **After：** 常量加类型枚举，名字本身就解释了业务。

    ```go
    const (
        DiscountVIP     = 0.9
        DiscountPartner = 0.8
    )

    func finalPrice(order *Order) float64 {
        switch order.CustomerType {
        case CustomerVIP:
            return order.Amount * DiscountVIP
        case CustomerPartner:
            return order.Amount * DiscountPartner
        default:
            return order.Amount
        }
    }
    ```

### 做法三：写注释

- **思路：** 代码能表达“做了什么”，但表达不了“为什么这么做”。决策的背景信息必须写进注释。

- **示例：** 一段反直觉的串行重试，注释挡住后人“顺手改成并发”的误操作。

    ```go
    // 这里必须串行重试，因为下游限流网关对并发请求会直接拒，
    // 不是性能没优化，别改成并发。
    for i := 0; i < maxRetry; i++ {
        callDownstream()
    }
    ```

## 提升可维护性

- **可维护性（maintainability）：** 指未来的修改成本——以后能不能安全地改这段代码（加功能、改逻辑）而不搞坏别的地方。良好的代码结构让改动面更小、影响更可控。

- **可读性 vs 可维护性：** 可读性是“看懂”，可维护性是“敢改”。

    - 看懂是敢改的前提：连看都看不懂的代码一定改不动，所以可读性差则可维护性一定差。

    - 反过来不成立：代码读起来顺，不代表就好改。

- **可维护性的构成：** 可维护性除了可读性，还包括测试覆盖、低耦合、良好的抽象。

- **四个支撑原则：** 把“低耦合 + 良好抽象”落到实处的设计原则。

    - **依赖倒置：** 高层不依赖低层，双方都依赖接口；“我需要什么能力”由高层说了算，低层反过来实现高层提出的接口。

    - **面向接口编程：** 持有接口，而不是具体类型。

    - **依赖注入：** 实例从外部塞进来，而不是在内部自己 `new`。

    - **关注点分离：** 不相干的事拆开，各自关起门来变化。

### 做法一：配置与代码分离

- **思路：** 把会变的东西（数据库地址、超时时间、各种阈值、开关）从代码里挪到配置文件 / 环境变量。改个超时不用重新编译发版；不同环境（dev/prod）用不同配置，代码完全一样。

- **Before：** 配置写死在代码里，换环境要改代码、重新编译发版。

    ```go
    func NewData() (*Data, error) {
        dsn := "root:dev@tcp(127.0.0.1:3306)/dev" // 写死，换 prod 得改这行再编译
        db, err := gorm.Open(mysql.Open(dsn))
        if err != nil {
            return nil, err
        }
        return &Data{db: db}, nil
    }
    ```

- **After（Kratos 做法）：** 值挪进 yaml，加载后注入 data 层，代码只认配置。

    - **配置文件 `configs/config.yaml`：** 换环境只改这个文件。

        ```yaml
        data:
          database:
            dsn: "root:dev@tcp(127.0.0.1:3306)/dev"
        ```

    - **读配置：** `config.New` 加载，`Scan` 进 proto 生成的 `conf.Bootstrap`。

        ```go
        c := config.New(config.WithSource(
            file.NewSource("configs/config.yaml"),
        ))
        defer c.Close()
        if err := c.Load(); err != nil {
            panic(err)
        }

        var bc conf.Bootstrap // 由 internal/conf/conf.proto 生成
        if err := c.Scan(&bc); err != nil {
            panic(err)
        }
        ```

    - **用配置：** data 层接收 `*conf.Data`，读 `c.Database.Dsn`，不写死。

        ```go
        func NewData(c *conf.Data) (*Data, error) {
            db, err := gorm.Open(mysql.Open(c.Database.Dsn))
            if err != nil {
                return nil, err
            }
            return &Data{db: db}, nil
        }
        ```

### 做法二：依赖倒置

- 让 biz 层只认接口、不认具体数据库；以后换实现、加缓存只动 data 层，上层一行都不用碰。

- **Before：** biz 跟 GORM/MySQL 绑死。

    ```go
    func (uc *UserUsecase) GetUser(ctx context.Context, id int64) (*User, error) {
        var u User
        uc.db.WithContext(ctx).First(&u, id)
        return &u, nil
    }
    ```

- **After：** biz 只依赖接口，data 去实现它。

    ```go
    // biz 层：只声明“我需要一个能按 ID 查用户的东西”
    type UserRepo interface {
        FindByID(ctx context.Context, id int64) (*User, error)
    }

    type UserUsecase struct {
        repo UserRepo
    }

    func (uc *UserUsecase) GetUser(ctx context.Context, id int64) (*User, error) {
        return uc.repo.FindByID(ctx, id)
    }

    // data 层：具体怎么查是我的事，哪天加 Redis 缓存也只动这里
    type userRepo struct {
        db *gorm.DB
    }

    func (r *userRepo) FindByID(ctx context.Context, id int64) (*User, error) {
        var u User
        if err := r.db.WithContext(ctx).First(&u, id).Error; err != nil {
            return nil, err
        }
        return &u, nil
    }
    ```

- **这例子不是为了可读性：** biz 直接操作 `*gorm.DB` 其实读起来一点都不难，`First(&u, id)` 一目了然。改成接口后函数体反而多绕一层（要先看 `UserRepo` 接口、再看 `userRepo` 实现），单论“看懂这一个函数”，After 甚至略微更难。

- **它优化的是“以后好不好改”，即可维护性：** Before 里 biz 和 GORM 焊死，将来换 ORM、加缓存、换数据源都得动 biz 的业务代码；After 把“怎么存取”隔离到接口背后，变化锁在 data 层，上层一行不用碰。让变化只发生在一个地方，正是可维护性的本质。

    - **解耦：** biz 不再依赖 GORM 这个具体实现，只依赖自己定义的 `UserRepo` 接口。

    - **可测试性：** 接口一抽出来，测 biz 时就能塞 mock 进去，不用连真 MySQL。

- **有时要牺牲一点局部可读性：** 为了可维护性，这里拿“多一层抽象、单个函数读起来稍绕”换“未来改动被隔离”。

### 做法三：消除重复（DRY）

- **核心收益：** 散落在多处的相同逻辑一旦要改，很容易漏改某处而留下 bug。收敛到一处后，只需动一个地方。

- **错误转换收敛（Kratos service 层）：** 把每个方法都抄一遍的「err → gRPC 错误码」逻辑抽成一个函数。

- **Before：** 错误转换逻辑每个方法抄一遍。

    ```go
    func (s *Service) GetUser(ctx context.Context, ...) (..., error) {
        u, err := s.uc.GetUser(ctx, id)
        if err != nil {
            if errors.Is(err, gorm.ErrRecordNotFound) {
                return nil, status.Error(codes.NotFound, "user not found")
            }
            return nil, status.Error(codes.Internal, err.Error())
        }
        // ...
    }

    func (s *Service) GetOrder(ctx context.Context, ...) (..., error) {
        o, err := s.uc.GetOrder(ctx, id)
        if err != nil {
            if errors.Is(err, gorm.ErrRecordNotFound) {        // 一模一样又抄一遍
                return nil, status.Error(codes.NotFound, "order not found")
            }
            return nil, status.Error(codes.Internal, err.Error())
        }
        // ...
    }
    ```

- **After：** 收敛成一个函数，要改错误码规则只改一处。

    ```go
    func toGRPCError(err error, notFoundMsg string) error {
        switch {
        case errors.Is(err, gorm.ErrRecordNotFound):
            return status.Error(codes.NotFound, notFoundMsg)
        default:
            return status.Error(codes.Internal, err.Error())
        }
    }

    // 调用处
    if err != nil {
        return nil, toGRPCError(err, "user not found")
    }
    ```

### 做法四：面向接口，依赖注入，消除耦合

- **核心做法：** 把依赖通过接口注入进来，而不是在函数里直接 new。测试时塞个 mock 进去，不连真的数据库也能验证业务规则，测试又快又稳。

- **示例·订单能否退款：** 把 `*gorm.DB` 换成 `OrderRepo` 接口，业务逻辑只依赖接口，测试里注入假实现。

- **Before：** 依赖真实数据库，要测就得连真库、灌数据，又慢又麻烦。

    ```go
    type OrderUsecase struct {
        db *gorm.DB
    }

    func (uc *OrderUsecase) CanRefund(ctx context.Context, id int64) (bool, error) {
        var o Order
        uc.db.WithContext(ctx).First(&o, id)
        return o.Status == "paid" && time.Since(o.PaidAt) < 7*24*time.Hour, nil
    }
    ```

- **After：** 依赖接口，测试里注入 mock。

    ```go
    type OrderRepo interface {
        FindByID(ctx context.Context, id int64) (*Order, error)
    }

    type OrderUsecase struct {
        repo OrderRepo
    }

    func (uc *OrderUsecase) CanRefund(ctx context.Context, id int64) (bool, error) {
        o, err := uc.repo.FindByID(ctx, id)
        if err != nil {
            return false, err
        }
        return o.Status == "paid" && time.Since(o.PaidAt) < 7*24*time.Hour, nil
    }
    ```

- **测试：** 这么写完全不碰数据库，注入 mock 即可。

    ```go
    type mockRepo struct{ order *Order }

    func (m *mockRepo) FindByID(ctx context.Context, id int64) (*Order, error) {
        return m.order, nil
    }

    func TestCanRefund_超过7天不能退(t *testing.T) {
        uc := &OrderUsecase{repo: &mockRepo{
            order: &Order{Status: "paid", PaidAt: time.Now().Add(-8 * 24 * time.Hour)},
        }}
        ok, _ := uc.CanRefund(context.Background(), 1)
        if ok {
            t.Fatal("已超过 7 天，不应允许退款")
        }
    }
    ```

### 做法五：开闭原则

- **思路：** 开闭原则（OCP）回答一个问题——新需求来了，该「加代码」还是「改代码」？答案是“对扩展开放、对修改关闭”：加功能时尽量新增代码，而不是去改已经在跑的老代码（可能弄坏老逻辑）。

- **Before：** 每多一种支付方式，就得回来改 `Pay`，往 switch 里塞一个分支。

    ```go
    func Pay(method string, amount float64) error {
        switch method {
        case "alipay":
            // 调支付宝...
            return callAlipay(amount)
        case "wechat":
            // 调微信...
            return callWechat(amount)
        // 以后接银联，得回来改这个 switch ↓
        // case "unionpay":
        //     return callUnionPay(amount)
        default:
            return errors.New("unsupported method")
        }
    }
    ```

- **After：** 把「一种支付方式」抽象成接口，每种实现是独立类型；加银联 = 新写一个文件实现接口，`Pay` 这段调度代码一个字都不用动。

    ```go
    // 抽象：所有支付方式都满足这个接口
    type PayMethod interface {
        Pay(amount float64) error
    }

    // 各自独立实现，互不干扰
    type Alipay struct{}
    func (Alipay) Pay(amount float64) error { return callAlipay(amount) }

    type Wechat struct{}
    func (Wechat) Pay(amount float64) error { return callWechat(amount) }

    // 注册表：加新方式就往这加一行，不碰下面的 Pay
    var methods = map[string]PayMethod{
        "alipay": Alipay{},
        "wechat": Wechat{},
    }

    // 调度逻辑：对修改“关闭”——以后永远不用动它
    func Pay(method string, amount float64) error {
        m, ok := methods[method]
        if !ok {
            return errors.New("unsupported method")
        }
        return m.Pay(amount)
    }
    ```

- **扩展：** 将来接银联，只新增一个文件，老代码全程不动。

    ```go
    type UnionPay struct{}
    func (UnionPay) Pay(amount float64) error { return callUnionPay(amount) }

    func init() {
        methods["unionpay"] = UnionPay{} // 注册，仅此而已
    }
    ```

- **本质是策略模式：** 可以互相替换的算法抽成接口，调用方依赖接口，新增策略不影响调度方。

- **代价与判断：** OCP 的代价是引入抽象（多了接口、多了类型），不是所有 switch 都值得这么改。判断标准是这个地方将来会不会频繁加新分支——只有两种、几乎不会再变的状态判断，硬套接口就是过度设计；变化频繁的才值得用 OCP 包起来，有些地方留着 switch 挺好。

## 控制技术债

- **什么是技术债：** 为了赶进度而提交的未被充分理解、测试且结构差劲的代码。

- **重构就是还债：** 避免债务累积到代码彻底改不动（架构垃圾、谁也看不懂）只能推倒重写。
