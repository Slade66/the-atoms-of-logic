# Bifrost 的系统设计

### 🧠 传统 HTTP 同步模型的局限

- **同步请求响应的机制** 传统 HTTP 是典型的同步请求-响应模型。它的工作流程是客户端发请求，然后服务端处理，接着返回结果，最后连接关闭。

- **资源强绑定特性** 在传统模型中，一个请求严格对应一个底层连接即 Socket。并且该连接在整个业务处理期间必须持续保持开启状态。

- **处理长任务的根本矛盾** 迁移平台需要处理大量耗时极长的数据同步任务，这往往需要数十分钟到数小时不等。用这种典型的短连接模型去处理长时间任务，网络连接一直不被释放，导致 CPU 和 IO 持续工作，最终拖垮系统。

### 🧠 长期占用连接引发的系统崩溃

- **负载均衡器强制断连** 负载均衡器通常有严格的超时机制。如果后端的请求太久没有返回，LB 会认为该请求已经卡住，从而强制断掉连接，这就导致业务任务还没有执行完成就被迫中断。任何超过几十秒的任务，都不应该用同步 HTTP 挂在 LB 后面执行。

- **LB 的工程折中策略** LB 需要在资源回收和用户体验之间进行平衡，连接既不能无限占用，也不能太快断开。在工程实践中往往将超时时间设置为 30 到 60 秒的最佳平衡点。

- **套接字资源枯竭瘫痪** 每个尚未完成的 HTTP 请求都会持续占用一个 Socket 资源。但系统的文件描述符、TCP 连接数以及可用内存都是有限的。大量耗时任务导致 Socket 长时间不释放，新请求完全无法建立连接，最终导致系统拒绝服务。

### 🧠 正确思路：解耦与异步任务模型

- **解耦请求与任务执行** 将轻量级的请求操作与重量级的任务执行过程彻底分离，采用异步任务模型来重构架构规划。

- **请求端的单一职责** 在这里请求的唯一目的仅仅是提交任务。服务端在接收到请求并入队后，立即向客户端返回一个唯一的标志符 taskId。

- **后台执行与状态感知** 实际的数据同步任务交由后端的 Worker 进程在独立后台环境中默默执行。客户端后续只需要通过建立轮询机制查询 taskId 的最新状态，或者等待服务端完成后的回调通知即可。
