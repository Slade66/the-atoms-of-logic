# 从 MBR 到 Bootloader 完整启动

### GRUB 是什么？三段拼起来的一个整体

- **GRUB 不是一个文件，而是被切成三块塞在硬盘不同位置的组合：**

    ```
    ┌──────────────────────────────────────────────────┐
    │ 位置 1：LBA 0         └─ 446 字节 (叫 stage 1)   │  ← MBR 里
    │ 位置 2：LBA 1 ~ 2047  └─ 约 30 KB (叫 stage 1.5) │  ← "MBR 后隙"
    │ 位置 3：/boot/grub/   └─ 几 MB (叫 stage 2)      │  ← 文件系统里
    └──────────────────────────────────────────────────┘
    ```

- **为啥要切三块？逐步升级能力：** 每一段的唯一使命就是"把下一段搞进内存"。

    | 段 | 大小 | 能力 |
    | --- | --- | --- |
    | stage 1 | 446 B | 只会按扇区号读盘 |
    | stage 1.5 | ~30 KB | 能认文件系统（ext4、ntfs...） |
    | stage 2 | 几 MB | 能显示菜单、读内核、跟用户交互 |

### 第 1 步：stage 1.5 自引导头读自己剩下的部分

- **起点衔接：** MBR 交棒时读进 `0x7C00` 的那个扇区，其实就是 stage 1.5 的第一个扇区——从这一刻起进入 GRUB 的地盘。

- **CPU 现在跑的是 stage 1.5 的"自引导头"**（第一个扇区，512 字节）。它干的第一件事：把自己剩下的部分也读进来。

    - stage 1.5 总共约 30 KB = 约 60 个扇区
    - 现在只读进来了 1 个（第一个）
    - 还差 59 个 ← 自己把它们读进来

- **怎么知道剩下的扇区在哪？硬编码写死在代码里：** 安装 GRUB 时，安装程序把"我剩下的 59 个扇区住在 LBA 2 ~ LBA 60"这个地址直接烧进了 stage 1.5 的代码。

- **读完后的内存现状（T=1）：**

    ```
    ┌─────────────────────────────────────────┐
    │ 0x7C00  ← stage 1.5 第 1 扇区            │
    │ 0x7E00  ← stage 1.5 第 2 扇区（新读进来）│
    │ 0x8000  ← stage 1.5 第 3 扇区            │
    │ ...     ← stage 1.5 第 4~60 扇区         │
    │ 约 0xF000 结束                           │
    └─────────────────────────────────────────┘
    ```

### 第 2 步：stage 1.5 全员到齐 → 加载文件系统驱动

- **stage 1.5 这 30 KB 里装了啥：**

    ```
    ├── 一小段自引导代码（前面那 512 字节）
    ├── 文件系统驱动（比如 ext4 驱动）
    ├── 读盘的封装
    └── "去找 stage 2" 的逻辑
    ```

- **stage 1.5 是"定制款"：** 装 GRUB 时安装程序会问"你的 `/boot` 是什么文件系统"，然后把对应驱动塞进 stage 1.5。`/boot` 是 ext4 就塞 ext4 驱动，是 xfs 就塞 xfs 驱动。

- **质变：从按扇区读盘升级到按文件名读盘。**

    - **以前：** 读 LBA 2048 的 512 字节（按扇区号）
    - **现在：** 打开文件 `/boot/grub/i386-pc/core.img`（按文件名）

    这就是"认识文件系统"带来的超能力。

### 第 3 步：stage 1.5 → 读 stage 2（真正的主程序）

- **stage 1.5 现在干的事：**

    1. 遍历硬盘所有分区，找到带 `/boot/` 目录的那个分区。
    2. 在该分区的文件系统里，打开文件 `/boot/grub/i386-pc/core.img`。
    3. 把这个文件（几 MB）完整读到内存。
    4. 跳过去执行。

### 第 4 步：stage 2（真正的 GRUB） → 显示菜单

- **stage 2 就是你平时说的 "GRUB"。** 能干的事一下子多起来：

    1. **读配置文件** `/boot/grub/grub.cfg`（现在能按文件名读了），解析出菜单项：

        ```
        menuentry "Ubuntu"               { linux /boot/vmlinuz... initrd /boot/initrd... }
        menuentry "Windows Boot Manager" { chainloader ... }
        menuentry "Recovery Mode"        { ... }
        ```

    2. **初始化显卡（文本模式），把菜单画出来：**

        ```
        ┌──────────────────────────────┐
        │ *Ubuntu 22.04                │  ← ★ 你终于看到 GRUB 的脸了 ★
        │  Ubuntu (recovery mode)      │
        │  Windows Boot Manager        │
        └──────────────────────────────┘
        ```

    3. **读键盘，等你选。**
    4. **倒计时**（你不选就走默认）。

- **"bootloader 起来了"指的就是这一刻：** 你眼睛看到 GRUB 菜单，意味着 stage 2 已经完全在内存里跑起来了。

### 比喻：叫醒一个巨人

```
MBR 只能发出一声呼喊（446 字节太小了）
  │
  ▼
呼喊声叫醒了"中巨人"(stage 1.5)
  中巨人睡觉时身体分成了 60 节（扇区）
  MBR 只唤醒了他的头（第 1 个扇区）
  头一醒，自己把身体其他部分也叫起来（再读 59 个扇区）
  │
  ▼
中巨人站起来，学会了"认字"（文件系统驱动）
  他去图书馆（文件系统）里找"大巨人"(stage 2)
  把大巨人整本书都搬进内存
  │
  ▼
大巨人醒了！
  他能说话（显示菜单）、能听（读键盘）
  他知道操作系统的位置
  接下来他要把操作系统搬进来
  │
  ▼
★ Bootloader 完全启动 ★
```

### 时间轴总览

```
T=0     BIOS 读 LBA 0 (512B) → 内存 0x7C00 → 跳过去
          │
          ▼  [MBR 跑完]
T=0.01  MBR 读"stage 1.5 的第 1 个扇区"(512B) → 0x7C00 → 跳过去
          │
          ▼  [stage 1.5 第 1 扇区跑起来]
T=0.02  stage 1.5 第 1 扇区读自己剩下的 59 个扇区到内存
          │
          ▼  [stage 1.5 完整到齐，30 KB]
T=0.05  stage 1.5 加载文件系统驱动
          遍历分区找 /boot/
          以"文件"方式读 /boot/grub/i386-pc/core.img → 内存高地址
          跳过去
          │
          ▼  [stage 2 跑起来，几 MB]
T=0.5   stage 2 读 /boot/grub/grub.cfg
          初始化显卡、键盘驱动
          画菜单
          │
          ▼
T=1     ★ 屏幕上出现 GRUB 菜单 ★
          (这就是"bootloader 起来了")
          │
          ▼  等你按键 / 倒计时结束
T=3     stage 2 读 /boot/vmlinuz 到内存
          读 /boot/initrd.img 到内存
          切 CPU 模式，跳到内核
          │
          ▼
         内核启动
```

### 三条核心规律

- **每一段的第一个扇区是"自引导头"——** 它负责把自己剩下的部分读进来。

- **能力是逐步升级的：** 按扇区读 → 认文件系统 → 认用户输入。

- **"下一段在哪"不是凭空来的，是安装 GRUB 时硬编码写死在上一段里的。** 这就是为什么 GRUB 有时候会坏——你挪动了硬盘分区，那个写死的地址就对不上了。
