# CBT 的标准工作流：从基线同步到持续增量

## 一、核心机制

- **两层分工** CBT 无法独立工作，必须与快照配合使用：快照负责“定格”，CBT 负责“对比”。

    - Snapshot 在某一瞬间冻结磁盘状态，提供一份一致性的数据视图。
    - CBT 计算两个快照 ID，也就是 `changeId` 之间的差异，告诉程序哪些扇区发生了变动。

- **VDDK 角色** VDDK 读取快照时刻的磁盘数据，首次备份读取全量，增量备份则通过 CBT 只读取上次备份后变化的块。

- **备份流程** 一次基于快照和 CBT 的备份流程，本质上按以下顺序执行。

    1. 通知 vCenter 给 VM 创建快照。
    2. 快照冻结某一时刻的磁盘状态，VM 继续运行，新的写入进入 delta 文件。
    3. VDDK 读取快照时刻的磁盘数据。
    4. 首次备份读取全量数据。
    5. 增量备份通过 CBT 只读取上次备份后变化的块。
    6. 备份完成后删除快照。

- **核心结论** CBT 是 VDDK 增量备份能力的核心，但它必须依赖快照提供稳定的数据视图。

## 二、初始化与启用

- **硬件要求** 使用 CBT 前，需要确认虚拟机硬件版本大于等于 7。

- **显式开启** 通过 `ReconfigureVM` 将 `ctkEnabled` 置为 `true`，例如使用 `govmomi` 调用 vSphere API。

- **启用效果** 开启 CBT 相当于向 ESXi 注册：从现在开始记录该虚拟磁盘的 I/O 变化，并生成对应的 `.ctk` 元数据文件。

## 三、基线同步阶段

- **创建快照** 为 VM 创建首个快照，获取标识当前状态的 `changeId`。

- **全量查询** 调用 `QueryChangedDiskAreas` 方法，并传入通配符 `"*"` 作为起点标识。

- **跳过空白区** API 返回所有已分配块的偏移量与长度；只提取已分配数据，可以跳过磁盘中的大量空白区域，缩短全量同步耗时。

## 四、迭代增量阶段

- **再次快照** 后续同步时再次创建快照，拿到新的磁盘状态视图。

- **对比查询** 使用上一次保存的 `changeId` 调用 `QueryChangedDiskAreas`，让 VMware 计算两次快照之间的变化区间。

- **精准读取** VMware 只返回自该 `changeId` 以来发生变化的底层扇区区间，备份程序根据这个列表针对性读取，实现真正的增量传输。
