# AI Good Answer: 用 “伪代码 + JSON 形式的数据结构” 解释代码的逻辑

好，我们完全不用真实代码，只用：

- 伪代码
- 逻辑说明
- JSON 形式的数据结构

来把这次我写的逻辑讲清楚。

这段逻辑的本质是：

**先问 CBT：哪些块是真实有数据的。  
再去 ESXi：只把这些块读出来。  
最后写到本地 raw 文件的相同偏移。**

这就是里程碑 3 的最小核心。

---

## 1. 我们想解决的问题

源虚拟磁盘逻辑大小可能是 16GB，但真正有数据的可能只有 2.17GB。

所以我们不想这样做：

```text
下载整个 16GB flat.vmdk
```

我们想这样做：

```text
先拿到“哪些位置有数据”
再只下载这些位置
```

---

## 2. 整个流程的总览

伪代码总流程：

```text
输入:
  vm
  snapshotName
  localRawPath

步骤:
  1. 找到指定快照
  2. 找到快照对应的磁盘
  3. 用 QueryChangedDiskAreas(changeId="*") 查询所有已分配块
  4. 创建一个本地 raw 文件
  5. 遍历每个块:
       从 ESXi 的 flat.vmdk 读取这个块
       写入本地 raw 文件的相同偏移
  6. 输出基线 raw 文件
```

---

## 3. 先理解“输入数据”长什么样

### 3.1 虚拟机和快照的概念数据

```json
{
  "vm": {
    "name": "test-xiaoze",
    "snapshots": [
      {
        "name": "BIFROST_BASELINE",
        "ref": "snapshot-123"
      }
    ],
    "disks": [
      {
        "key": 2000,
        "label": "Hard disk 1",
        "capacityBytes": 17179869184,
        "backing": {
          "type": "VirtualDiskFlatVer2BackingInfo",
          "fileName": "[datastore1] test-xiaoze/test-xiaoze.vmdk"
        }
      }
    ]
  }
}
```

这里最重要的三个字段是：

- `snapshot.ref`
- `disk.key`
- `disk.backing.fileName`

---

### 3.2 CBT 查询结果的目标形态

调用 `QueryChangedDiskAreas(changeId="*")` 后，我们最终想拿到这种数据：

```json
{
  "snapshotRef": "snapshot-123",
  "diskKey": 2000,
  "diskCapacityBytes": 17179869184,
  "backingPath": "[datastore1] test-xiaoze/test-xiaoze.vmdk",
  "allocatedExtents": [
    {
      "start": 0,
      "length": 1048576
    },
    {
      "start": 8388608,
      "length": 524288
    },
    {
      "start": 16777216,
      "length": 2097152
    }
  ]
}
```

意思是：

- 从 `0` 开始的 `1MB` 有数据
- 从 `8MB` 开始的 `512KB` 有数据
- 从 `16MB` 开始的 `2MB` 有数据

只有这些块需要下载。

---

## 4. 第一部分逻辑：先找到“快照 + 磁盘”

### 4.1 目标

从 VM 里找出：

- 指定快照是谁
- 要处理的是哪块磁盘
- 这块磁盘在 datastore 上的路径是什么

### 4.2 伪代码

```text
函数: 找到快照磁盘信息(vm, snapshotName)

读取 vm 的:
  - snapshot 树
  - hardware.device 列表

在 snapshot 树中递归查找:
  如果 snapshot.name == snapshotName
    记录 snapshot.ref

在 hardware.device 中查找:
  找第一块 VirtualDisk
  取出:
    - disk.key
    - disk.capacityBytes
    - disk.backing.fileName

如果没有找到快照:
  报错

如果没有找到磁盘:
  报错

返回:
  {
    snapshotRef,
    disk,
    backingPath
  }
```

### 4.3 返回的数据长这样

```json
{
  "snapshotRef": "snapshot-123",
  "disk": {
    "key": 2000,
    "capacityBytes": 17179869184
  },
  "backingPath": "[datastore1] test-xiaoze/test-xiaoze.vmdk"
}
```

---

## 5. 第二部分逻辑：分页查出所有“已分配块”

这里是最关键的 CBT 查询部分。

### 5.1 为什么要分页

`QueryChangedDiskAreas` 不保证一次把整盘结果全给你。  
所以必须从 `offset=0` 开始，一页一页往后查。

### 5.2 伪代码

```text
函数: 查询已分配块列表(vm, snapshotName)

info = 找到快照磁盘信息(vm, snapshotName)

offset = 0
extents = []

循环:
  如果 offset >= disk.capacityBytes:
    结束循环

  发起查询:
    snapshot = info.snapshotRef
    deviceKey = info.disk.key
    startOffset = offset
    changeId = "*"

  得到响应 response

  把 response.changedArea 追加到 extents

  如果 response.length <= 0:
    结束循环

  nextOffset = response.startOffset + response.length

  如果 nextOffset <= offset:
    报错
    因为这意味着分页没有前进，会死循环

  offset = nextOffset

返回:
  {
    snapshotRef,
    disk,
    backingPath,
    extents
  }
```

---

### 5.3 单次响应长什么样

一次分页查询可能返回：

```json
{
  "startOffset": 0,
  "length": 16777216,
  "changedArea": [
    {
      "start": 0,
      "length": 1048576
    },
    {
      "start": 8388608,
      "length": 524288
    }
  ]
}
```

这表示：

- 这次描述的是磁盘 `[0, 16MB)` 这段范围
- 在这 16MB 里，只有两个小块真的有数据

然后下一轮查询从：

```json
{
  "nextOffset": 16777216
}
```

继续往后查。

---

### 5.4 最终聚合结果

多次分页后，得到总 extents：

```json
[
  {
    "start": 0,
    "length": 1048576
  },
  {
    "start": 8388608,
    "length": 524288
  },
  {
    "start": 16777216,
    "length": 2097152
  }
]
```

这就是后续下载的工作清单。

---

## 6. 第三部分逻辑：把 extents 写进本地 raw 文件

### 6.1 目标

有了块清单之后，下一步就是：

- 打开本地 raw 文件
- 对每个块：
  - 去 ESXi 读取该块
  - 写到本地同样的偏移

### 6.2 伪代码

```text
函数: 下载变更块到本地raw(dsPath, localRawPath, diskSize, extents)

把 dsPath 解析成:
  - datastore 名字
  - 相对路径

例如:
  "[datastore1] test-xiaoze/test-xiaoze.vmdk"

解析后变成:
```json
{
  "datastoreName": "datastore1",
  "relativePath": "test-xiaoze/test-xiaoze.vmdk"
}
```

然后把描述文件路径改成 flat 文件路径：

```json
{
  "flatPath": "test-xiaoze/test-xiaoze-flat.vmdk"
}
```

然后：

```text
打开本地文件 localRawPath
把文件逻辑大小设置为 diskSize

遍历 extents:
  对每个 extent:
    从 ESXi 读取 flatPath 在 [start, start+length) 这一段
    写到本地文件的 start 偏移
```

---

## 7. 第四部分逻辑：怎么“只读取某一段”？

这里的关键思想不是 VMware 特有 API，而是 HTTP 的 Range 请求。

### 7.1 例子

如果某个 extent 是：

```json
{
  "start": 8388608,
  "length": 524288
}
```

那我们要读取的是：

```json
{
  "rangeStart": 8388608,
  "rangeEnd": 8912895
}
```

因为：

```text
rangeEnd = start + length - 1
```

然后向 ESXi 发起一个带 Range 头的下载请求：

```json
{
  "method": "GET",
  "headers": {
    "Range": "bytes=8388608-8912895"
  }
}
```

意思是：

```text
只给我这 512KB，不要整个文件
```

---

### 7.2 预期响应

理想情况下，ESXi 返回：

```json
{
  "status": 206,
  "statusText": "Partial Content",
  "contentLength": 524288
}
```

这表示：

- 服务器接受了 Range 请求
- 返回的是文件的一部分
- 长度和我们要的一样

如果服务器返回：

```json
{
  "status": 200,
  "statusText": "OK"
}
```

那往往说明：

```text
服务器没有按“部分内容”处理，
而是想把整个文件都返回
```

这个对我们的按块下载逻辑是危险的，所以我让它直接报错，不接受。

---

## 8. 第五部分逻辑：怎么写到文件的指定偏移？

网络返回的是一个连续字节流，但本地文件要求：

```text
把这段数据写到 start 偏移
```

所以逻辑上用了一个“带当前位置的写入器”。

### 8.1 它的行为

假设当前块是：

```json
{
  "start": 16777216,
  "length": 2097152
}
```

那么这个写入器的初始状态是：

```json
{
  "currentOffset": 16777216
}
```

当网络流分几次吐数据时：

第一次吐 64KB：

```json
{
  "writeToOffset": 16777216,
  "writtenBytes": 65536,
  "nextOffset": 16842752
}
```

第二次再吐 64KB：

```json
{
  "writeToOffset": 16842752,
  "writtenBytes": 65536,
  "nextOffset": 16908288
}
```

持续这样推进，直到这一个 extent 写完。

---

## 9. 最后生成的本地 raw 文件是什么样？

假设磁盘逻辑大小是 16GB，本地 raw 文件的逻辑状态可以理解成这样：

```json
{
  "path": "baseline.raw",
  "logicalSize": 17179869184,
  "writtenExtents": [
    {
      "start": 0,
      "length": 1048576
    },
    {
      "start": 8388608,
      "length": 524288
    },
    {
      "start": 16777216,
      "length": 2097152
    }
  ],
  "unwrittenAreas": "保持空洞或零值"
}
```

所以它不是“把 16GB 全下载到本地”，而是：

- 逻辑上是完整盘
- 实际只写入有数据的块

这正是你想要的基线形态。

## 10. 一句话总结整个逻辑

可以压缩成这一段伪代码：

```text
函数: 构建基线raw(vm, snapshotName, localRawPath)

1. 找到 snapshotRef、diskKey、diskCapacity、backingPath
2. 用 QueryChangedDiskAreas("*") 分页查出所有 allocatedExtents
3. 创建本地 raw 文件，大小设为 diskCapacity
4. 对每个 allocatedExtent:
     用 HTTP Range 从 ESXi 的 flat.vmdk 读取这一段
     写入本地 raw 文件相同偏移
5. 返回完成
```

- S1 基线
- S2 增量
- patch 合并

也就是把 Phase 3 和 Phase 5 连起来。
