# Go：用 Build Tag 分离 Dev / Release 版本

## 一、目标：同一个包两份实现，编译期择一

- **核心思路** Go 的 build tag（构建标签）让编译器根据命令行参数选择性地"看见"或"看不见"某些 `.go` 文件，从而让同一个包里共存两份实现，编译时决定用哪一份。

## 二、机制：`//go:build` 指令

- **文件头的编译器指令** 文件第一行（或紧跟在前导空行之后）的 `//go:build` 不是注释，是给编译器看的条件。

    ```go
    //go:build release   // 只有命令行带 -tags=release 才编我
    //go:build !release  // 只有命令行没带 -tags=release 才编我
    ```

- **互斥文件对** 把两份实现写成条件互斥的两个文件，任何一次编译里只有一份会进二进制，另一份不存在于产出物里。

    ```
    license/
    ├── license_dev.go       // //go:build !release
    └── license_release.go   // //go:build release
    ```

## 三、用法：两种编译命令

- **开发 / 测试：不带 tag，走 dev 版**

    ```
    go run ./cmd/bifrost
    go build ./cmd/bifrost
    go test ./...
    ```

- **发布给客户：带 `-tags=release`，走 release 版**

    ```
    go build -tags=release -trimpath -ldflags="-s -w" -o bifrost.exe ./cmd/bifrost
    ```

    各参数作用：

    - `-tags=release` 打开 `release` 构建标签，让 `//go:build release` 的文件被编入、`//go:build !release` 的被跳过。
    - `-trimpath` 把编译产物里的绝对路径全部剥掉，只留模块相对路径。避免 `C:\Users\liyuz\...` 这种开发机路径和用户名泄露进二进制（反编译、panic 栈都看不到）。
    - `-ldflags="-s -w"` 传给链接器的两个开关。
        - `-s` 去掉符号表（symbol table）。
        - `-w` 去掉 DWARF 调试信息。
        - 合起来：二进制体积明显变小，反编译难度提高（看不到函数名、源码行号）。
    - `-o bifrost.exe` 指定输出文件名，Windows 下带 `.exe` 后缀。
    - `./cmd/bifrost` 要编译的 main 包路径。

## 四、为什么不用 `os.Getenv("DEV_MODE")` 替代

- **反例写法** 很多人第一反应会在运行时用环境变量分支。

    ```go
    // 反例：千万别这么干
    func Bootstrap() error {
        if os.Getenv("DEV_MODE") == "1" {
            return nil
        }
        return realBootstrap()
    }
    ```

- **安全缺口** release 版二进制里两个分支的代码都在，客户只要 `set DEV_MODE=1`，或用二进制编辑器把判断改成 true，校验就废了。

- **build tag 是编译期物理剔除** `license_dev.go` 的代码根本不会出现在 release 版二进制里，反编译也看不到。

## 五、易错点：`//go:build` 与 `package` 之间必须空一行

- **格式要求** `//go:build` 必须在文件最开头（可以有空行但不能有其它内容），和 `package` 声明之间必须隔一行空行。

    ```go
    //go:build release
                        ← 这行空行不能少
    package license
    ```

- **少一行的后果** 少了那行空行，go 会把 `//go:build` 当成普通注释，build tag 不生效，两个文件都会被编进去，报"函数重复定义"错误。
