# go generate

## 一、是什么

- **定义：** `go generate` 是 Go 工具链里的一个命令，作用是扫描源码中的特殊注释指令，并执行注释里指定的命令。

- **它只是一个 “命令的触发器”：** 真正干活的是你让它去运行的那些工具。

## 二、用法

- **写注释指令：** 在 `.go` 文件里写一行特殊格式的注释，`//go:generate command argument...`。

    - **格式硬性要求：** 必须以 `//go:generate` 开头，`//` 和 `go:generate` 之间不能有空格（这点和普通注释不同）。
   
    - **放置位置：** 可放在文件任意位置，但通常放在文件顶部或相关类型定义的上方。

- **命令行触发：** 执行 `go generate ./...`，Go 会找到所有这类注释，按文件中出现的顺序依次执行后面的命令。

- **生成物要提交：** 生成的代码也应该提交到版本库里。这是 Go 刻意的设计——避免编译过程依赖外部工具。

## 三、简单的例子

- **文件结构：**

    ```text
    demo/
    ├── main.go    # 用到生成的变量
    └── gen.go     # 生成器，负责写文件
    ```

- **`main.go`：** `BuildTag` 还不存在，直接 `go run .` 会报“未定义”；关键就是那行 `//go:generate go run gen.go`。

    ```go
    package main

    import "fmt"

    //go:generate go run gen.go

    func main() {
        fmt.Println("构建版本:", BuildTag)
    }
    ```

- **`gen.go`：** 真正的生成器，`//go:build ignore` 让它在正常编译时被跳过，只有手动 `go run gen.go` 才执行。

    ```go
    //go:build ignore

    package main

    import (
        "fmt"
        "os"
        "time"
    )

    func main() {
        content := fmt.Sprintf("package main\n\n// 这是自动生成的,不要手改\nconst BuildTag = %q\n", time.Now().Format("2006-01-02 15:04:05"))
        os.WriteFile("version_gen.go", []byte(content), 0644)
        fmt.Println("已生成 version_gen.go")
    }
    ```

- **执行过程：** 没生成时报错，`go generate` 后再跑就正常了。

    ```bash
    ~/Desktop/fuckgo via 🐹 v1.26.3
    ❯ go run .
    # fuckgo
    ./main.go:8:34: undefined: BuildTag

    ~/Desktop/fuckgo via 🐹 v1.26.3
    ❯ go generate ./...
    已生成 version_gen.go

    ~/Desktop/fuckgo via 🐹 v1.26.3
    ❯ go run .
    构建版本: 2026-06-25 14:48:30
    ```

- **生成产物：** 执行完目录里会多出一个 `version_gen.go`。

    ```go
    package main

    // 这是自动生成的,不要手改
    const BuildTag = "2026-06-25 14:48:30"
    ```

- **本质总结：** `go generate` 自己什么都不生成，它只是读到 `//go:generate go run gen.go` 这行注释，然后替你去命令行敲了一遍 `go run gen.go`；真正把 `version_gen.go` 写出来的是 `gen.go`。

## 四、包路径通配符 `./...`

- **拆成两半理解：** `./...` 是 Go 命令行里的一个包路径通配符。

    - **`.`：** 表示当前目录这个包。
    
    - **`/...`：** Go 特有的通配符，意思是“这个目录下的所有子目录，递归地全找一遍”。
    
    - **合起来：** 当前目录，以及它下面所有层级子目录里的包，全都算上。

- **示例项目结构：**

    ```text
    myapp/
    ├── main.go
    ├── internal/
    │   ├── service/
    │   └── data/
    └── api/
    ```

- **在 `myapp/` 目录下的效果对比：**

    - **`go build .`：** 只处理当前目录这一个包。
    - **`go build ./...`：** 处理 `myapp`、`internal/service`、`internal/data`、`api` 全部子包。
    - **`go build ./internal/...`：** 只递归处理 `internal` 这一支下面的所有包。

- **用在 go generate 上：** 项目根目录跑 `go generate ./...`，就是让 Go 把整个项目里所有带 `//go:generate` 注释的文件一次性全扫一遍并执行，不用一个目录一个目录手动进去跑。

- **同理可推广：** `go test ./...`（跑全项目测试）、`go vet ./...`（全项目静态检查）也是同一个道理——`./...` 基本就是“对整个项目作用”的代名词。
