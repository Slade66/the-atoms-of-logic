# 为什么迁移黑屏

### 问题日志

- **启动表现** 一台运行 `RHEL 7.9` 的虚拟机从 `VMware` 迁移到华为 `FusionCompute` 后，`GRUB` 界面可以正常显示，但进入内核加载阶段后黑屏。

    - `GRUB` 内核选择界面可以正常渲染。

    - 选定内核并开始加载操作系统后才出现黑屏。

- **临时绕过** 在 `GRUB` 的内核命令行末尾追加 `nomodeset` 后，虚拟机可以绕过黑屏并进入命令行界面。

- **源端显卡** 虚拟机在 `VMware` 环境中运行时，硬件环境和显卡驱动都指向 `VMware`。

    - `dmidecode` 提取的 `SMBIOS` 数据显示硬件环境为 `VMware7,1`。

    - `lspci` 枚举到的显示适配器为 `VMware SVGA II Adapter`。

    - 当前使用的内核驱动为 `vmwgfx`。

- **目标显卡缺驱动** 代码级调查确认，`FusionCompute` 创建目标虚拟机时，底层 `QEMU/KVM` 进程固定使用 `cirrus` 虚拟显卡，但迁移后系统的 `initramfs` 中不存在对应的 `cirrus.ko` 内核驱动模块。

### 根因分析


在迁移完成后首次启动虚拟机时，GRUB 内核选择界面能够被正常渲染和观察到。这一现象是缩小故障排查范围的关键证据。在 Linux 内核接管系统控制权之前，底层 Hypervisor 提供的虚拟图形管道是完全健康且处于工作状态的。

帧缓冲遵循极其古老但绝对通用的工业标准，这些标准确保了任何基础软件，只需将像素颜色值按顺序写入这段特定的内存地址，虚拟显卡便会将其转化为屏幕上的可视图像。GRUB 引导加载程序在设计上极其精简，它并不包含、也不需要像  cirrus 或 vmwgfx 显卡驱动。相反，GRUB 纯粹依赖帧缓冲来绘制其菜单界面。这正是为什么用户能够在没有任何专用驱动程序的情况下，清晰地看到 GRUB 界面的根本原因 。

当用户在 GRUB 界面中选定 RHEL 7.9 的内核条目并按下回车键后，GRUB 会将编译好的 Linux 内核二进制映像（vmlinuz）以及初始内存文件系统映像（initramfs）从主存储设备加载到系统的随机存取存储器（RAM）中。随后，GRUB 将 CPU 的执行指针跳转至内核的入口点，并在此时将控制权连同系统启动参数（Kernel Command Line）以及一个指向固件帧缓冲内存物理地址的指针一并传递给 Linux 内核

在内核初始化的极早期阶段，Linux 同样尚未加载任何复杂的原生硬件驱动。为了在这一真空中期向用户反馈引导进度（例如打印启动日志或显示简单的 Plymouth 启动画面），内核内置了一组称为早期控制台驱动程序（Early Console Drivers）的基础组件，最典型的是 vesafb（对应 VESA 标准）或 efifb（对应 UEFI 标准）。这些早期驱动直接接管了 GRUB 遗留下来的固件帧缓冲内存。因此，从固件启动、GRUB 运行，一直到 Linux 内核早期初始化的这段时间内，屏幕的显示输出是连续且未被中断的。真正的危机，潜伏在内核试图跨越这一基础状态，加载原生高性能图形驱动的那一瞬间。

虚拟机首次在 FusionCompute 上通电启动时，其内部的 Linux 内核在扫描 PCI 总线时，将不再能够发现任何带有 VMware 标识的 SVGA 设备。取而代之的是，PCI 探测机制会报告发现了一个全新的、供应商 ID 属于 Cirrus Logic 的图形控制器。这种硬件拓扑的剧烈变化本身并不会直接导致系统崩溃，但它对操作系统内部的驱动程序预加载机制提出了严苛的自适应要求。如果系统在早期的引导序列中无法正确处理这一硬件突变，随之而来的便是严重的设备初始化失败。
