# Kratos 的响应 envelope

## 一、什么是 envelope

- **定义** envelope（信封）把所有接口的响应体都套进一个固定外层结构，业务数据塞进 `data` 字段。

- **没有 envelope 的样子** 每个接口 JSON 响应长相各异，前端要为每个接口写一套解析。

    ```json
    // /users/1
    {"id":1,"name":"alice"}

    // /orders
    [{"id":"a"},{"id":"b"}]
    ```

- **有 envelope 的样子** 所有接口外层一致，业务数据塞进 `data`。

    ```json
    {"code":0, "message":"ok", "data": <真正的业务数据>}
    ```

- **对前端的价值** 前端只要写一份公共解析逻辑（检查 `.code == 0` → 取 `.data`），所有接口都能共用。

## 二、反模式：把 envelope 写进 proto

- **错误示范** 把 `code` / `message` 直接写进每个 reply message。

    ```proto
    // ❌ 错误示范
    message ListEsxiHostsReply {
      int32 code = 1;
      string message = 2;
      repeated EsxiHost list = 3;
    }
    ```

- **主要坏处** 每个 rpc 都要重复声明 `code` / `message`，样板代码无穷无尽，业务数据形状被污染。

## 三、正确做法：只覆盖成功编码器，失败交给 Kratos

- **只覆盖成功编码** 成功路径由自己实现 `http.ResponseEncoder(fn)` 把 reply 套进 envelope；失败路径直接走 Kratos 默认的 `ErrorEncoder`（它已经基于 `*errors.Error` 输出 `{code, reason, message, metadata}`），不用我们管。

    | Option | 负责 | 处理方式 |
    | --- | --- | --- |
    | `http.ResponseEncoder(fn)` | 成功响应编码 | 自己覆盖，产出 envelope |
    | `http.ErrorEncoder(fn)` | 失败响应编码 | 用 Kratos 默认，不覆盖 |

- **统一入口** 只要写成功分支的 encoder 并注入，rpc 直接返回业务 message，proto 保持干净；失败交给 Kratos。

## 四、请求生命周期中 encoder 的位置

- **整体流程** 从请求进来到响应出去，encoder 是最后一环。

    ```
    客户端
      │ POST /v1/esxi-hosts/list  {...}
      ▼
    ┌─────────────────────────────────────────┐
    │  http.Server (kratos)                   │
    │                                         │
    │  1. RequestDecoder  ── 解 body → pb     │
    │  2. Middleware chain (recovery/log/...) │
    │  3. Service.ListEsxiHosts(ctx, req)     │
    │       └─→ Usecase → Repo → DB           │
    │  4. 返回 (*ListEsxiHostsReply, error)   │
    │                                         │
    │  5a. err == nil → ResponseEncoder(v)  ◄── 我们的 responseEncoder
    │  5b. err != nil → ErrorEncoder(err)   ◄── Kratos 默认 encoder
    └─────────────────────────────────────────┘
      ▼
    { "code":0, "message":"ok", "data": {...} }   ← envelope 出现在最外层
    ```

- **解耦点** Service 层根本不知道 envelope 存在，只负责返回业务 pb 对象或 error；envelope 只在最后一步的 encoder 里拼装。

- **Middleware 与 Encoder 的区别** Middleware 包裹"调用链"（收到 req → 返回 reply 的整个过程），可以短路 / 修改 ctx；Encoder 只负责最后一步把 reply 或 error 序列化成字节。

## 五、实现要点

- **注入位置** 在 `internal/server/http.go` 用 `http.ResponseEncoder(...)` 覆盖默认成功编码器；`ErrorEncoder` 不传，沿用 Kratos 默认。

- **成功分支** 把 reply 包成 `{code:0, message:"ok", data: <reply>}`。

- **失败分支** 不用自己写，Kratos 默认的 `ErrorEncoder` 直接基于 `*errors.Error` 输出 `{code, reason, message, metadata}`，业务层只管 `return errors.XXX(...)`。

- **`interface{}` 是关键** Kratos 把各种 pb 类型（`*EsxiHost` / `*ListEsxiHostsReply` / `*emptypb.Empty`）都塞进来，encoder 不关心具体类型，直接往 `data` 里填。

- **List 响应天然兼容分页** `data` 直接是 `ListEsxiHostsReply`，即 `{list: [...]}`，未来加分页字段不破坏结构。

## 六、为什么放在 transport encoder 层

- **职责分层** proto 只描述业务数据形状，传输层的 envelope 由 encoder 负责，互不污染。

- **改造成本低** 未来要换 envelope 格式（比如 `code` 改成 `status`），只改一个文件。
