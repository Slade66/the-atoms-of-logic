# kratos `make api` Windows 失败报告

---

## 一、事件日志

根因：Makefile 原来用 where git 算 git-bash 路径，但你的 Git 装在 A:\Program Files\Git\，路径中有空格，导致 Git_Bash 变量计算出 A:/ Files/Git/bin/bash.exe，shell 调用失败，API_PROTO_FILES 始终为空。

修复：把 Windows 分支改为用 PowerShell 的 Get-ChildItem 枚举 proto 文件，不再依赖 git-bash。

makefile
INTERNAL_PROTO_FILES=$(shell powershell -NoProfile -Command "Get-ChildItem -Path internal -Filter *.proto -Recurse | Resolve-Path -Relative")
API_PROTO_FILES=$(shell powershell -NoProfile -Command "Get-ChildItem -Path api -Filter *.proto -Recurse | Resolve-Path -Relative")
现在 make api 可以直接用了，后续不需要手动跑 protoc。


```
GOHOSTOS:=$(shell go env GOHOSTOS)
GOPATH:=$(shell go env GOPATH)
VERSION=$(shell git describe --tags --always)

ifeq ($(GOHOSTOS), windows)
	#the `find.exe` is different from `find` in bash/shell.
	#to see https://docs.microsoft.com/en-us/windows-server/administration/windows-commands/find.
	#changed to use git-bash.exe to run find cli or other cli friendly, caused of every developer has a Git.
	#Git_Bash= $(subst cmd\,bin\bash.exe,$(dir $(shell where git)))
	Git_Bash=$(subst \,/,$(subst cmd\,bin\bash.exe,$(dir $(shell where git))))
	INTERNAL_PROTO_FILES=$(shell $(Git_Bash) -c "find internal -name *.proto")
	API_PROTO_FILES=$(shell $(Git_Bash) -c "find api -name *.proto")
else
	INTERNAL_PROTO_FILES=$(shell find internal -name *.proto)
	API_PROTO_FILES=$(shell find api -name *.proto)
endif


api:
	protoc --proto_path=./api \
	       --proto_path=./third_party \
 	       --go_out=paths=source_relative:./api \
 	       --go-http_out=paths=source_relative:./api \
 	       --go-grpc_out=paths=source_relative:./api \
 	       --go-errors_out=paths=source_relative:./api \
 	       --validate_out=lang=go,paths=source_relative:./api \
	       --openapi_out=fq_schema_naming=true,default_response=false:. \
	       $(API_PROTO_FILES)
           

```

**现象**：Windows 上执行 `make api`，protoc 还没开始跑，make 展开阶段就报错。

**错误输出**：
```
'A:/' is not recognized as an internal or external command,
operable program or batch file.
Missing input file.
make: *** [Makefile:41: api] Error 1
```

**关键证据 1 —— `where git` 的结果**：
```
A:\Program Files\Git\cmd\git.exe
                ↑ 注意这里有空格
```

**关键证据 2 —— `make -n api`（仅打印命令不执行）同样报错**：
说明错误发生在 **make 变量展开阶段**，protoc 根本没被调用。

**关键证据 3 —— Makefile 里构造 Git Bash 路径的代码**：
```makefile
Git_Bash = $(subst \,/,$(subst cmd\,bin\bash.exe,$(dir $(shell where git))))
API_PROTO_FILES = $(shell $(Git_Bash) -c "find api -name *.proto")
```

## 二、根因分析

**根本原因：GNU make 的 `$(dir)` 函数按空格切分词，你的 Git 装在带空格的路径下，导致路径被拦腰斩断。**

### 逐步展开推演

起点：`$(shell where git)` = `A:\Program Files\Git\cmd\git.exe`

**第 1 步：`$(dir ...)`**

GNU make 的 `$(dir)` 是对**空格分隔的每个 word** 分别取目录。`A:\Program Files\Git\cmd\git.exe` 因为含空格，被 make 看成**两个 word**：

| word | $(dir word) 结果 |
|---|---|
| `A:\Program` | [A:\](cci:9://file:///:0:0-0:0) |
| `Files\Git\cmd\git.exe` | `Files\Git\cmd\` |

合并后：`A:\ Files\Git\cmd\`

**第 2 步：`$(subst cmd\,bin\bash.exe, ...)`**

把 `cmd\` 替换为 `bin\bash.exe`：
`A:\ Files\Git\bin\bash.exe`

**第 3 步：`$(subst \,/, ...)`**

把所有 `\` 替换为 `/`：
`A:/ Files/Git/bin/bash.exe`
  ↑ 空格还在

**第 4 步：最终 `Git_Bash` 被当作命令执行**

```
$(shell A:/ Files/Git/bin/bash.exe -c "find api -name *.proto")
         ↑ 第一个 token
```

cmd.exe 拿到这串，把第一个空格前的 [A:/](cci:9://file:///A:/:0:0-0:0) 当作命令 → 报 `'A:/' is not recognized`。

### 为什么 `API_PROTO_FILES` 展开失败导致整个 protoc 命令崩溃

`API_PROTO_FILES` 是 protoc 的输入文件列表。它展开失败返回空，而 `$(shell)` 执行过程中 cmd.exe 的报错文本直接泄漏到 make 的 stderr，于是你先看到 `'A:/' is not recognized`，然后 protoc 因为拿不到输入文件报 `Missing input file`。

### Linux 上为什么没问题

Makefile 在 `ifeq ($(GOHOSTOS), windows)` 分支之外用的是 `find` 直接调用，不走 Git Bash 那套路径构造，所以不会触发 `$(dir)` 的空格问题。

## 三、以后如何操作

**方案 A：Windows 上不用 `make api`，手写 protoc 命令**（当前已验证可用）

```powershell
protoc --proto_path=./api --proto_path=./third_party `
       --go_out=paths=source_relative:./api `
       --go-http_out=paths=source_relative:./api `
       --go-grpc_out=paths=source_relative:./api `
       --go-errors_out=paths=source_relative:./api `
       --validate_out="lang=go,paths=source_relative:./api" `
       --openapi_out="fq_schema_naming=true,default_response=false:." `
       api/esxi/v1/esxi_host.proto `
       api/fc/v1/fc_host.proto `
       api/helloworld/v1/greeter.proto `
       api/helloworld/v1/error_reason.proto
```

**方案 B：把 Git 迁到无空格路径**，例如 `C:\Git\`，改完环境变量 PATH 后 `make api` 立刻可用。

**方案 C：修 Makefile 让它兼容带空格的 Git 路径**（用 `$(firstword)` + 引号处理），改动较重，收益有限。

**方案 D（推荐）：写一个 `gen-api.ps1` 封装方案 A**，Windows 用户跑 `./gen-api.ps1`，Linux/Jenkins 继续用 `make api`，各走各路互不干扰。
