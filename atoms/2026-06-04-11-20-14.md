# 什么时候建表？什么时候用 JSON

核心原则:你能预见到要按某个字段去查询、过滤、排序、连接(JOIN)、或保证它的完整性吗?如果能,就拆成列(用表);如果不能,塞进 JSON 没问题。

字段会频繁出现在 WHERE、ORDER BY、JOIN 里。比如订单的 status、user_id、created_at,这些你几乎一定会拿来过滤和排序,做成列可以建索引,查询又快又清晰。
需要数据库帮你保证完整性。外键、唯一约束、非空、枚举值校验,这些只有列才能享受到。JSON 里的字段数据库基本不管,垃圾数据很容易混进去。
存在一对多或多对多关系。比如一个订单有多个商品项,标准做法是单独建一张 order_items 表,而不是把商品列表塞进订单行的 JSON 数组里——后者一旦要按商品聚合统计就很痛苦。

几个容易踩的坑:
别用 JSON 来逃避建模。"反正以后可能要加字段"是最常见的借口,结果是核心业务字段全埋在 JSON 里,半年后查询和报表全得绕路。已经确定的、稳定的、要查的字段,老老实实建列。
现代数据库(Postgres 的 jsonb、MySQL 的 JSON 类型)其实可以对 JSON 内部字段建索引、做查询,所以"用了 JSON 就完全查不了"已经不成立了。但即便如此,索引和查询的体验、可维护性仍然不如真正的列。
混用是常态,也是好做法。同一张表里,把稳定核心字段做成列,把灵活的、长尾的属性放进一个 JSON 列。比如 products(id, name, price, category, ...) 加一个 attributes jsonb。这往往是最实用的折中。


倾向用 JSON 的情况:
结构多变或因记录而异。典型例子是商品属性:书有作者和页数,衣服有尺码和颜色,手机有屏幕尺寸。给每种品类都建列会爆炸,这种"半结构化"数据放 JSON 很合适。

你只是整存整取,不需要在数据库内部查询它的内部字段。比如第三方 API 的原始响应、前端的 UI 配置、审计日志的快照。存下来留档,需要时整块读出来在应用层处理。

schema 还在快速演化,你不想每加一个字段就跑一次迁移。JSON 给你灵活性,代价是把校验责任推给了应用代码。


## 一个"父对象拥有一个子列表"的取舍

场景:一个对象拥有一个从属的小列表(一对多),纠结要不要为这个列表单独建中间表。

**只有当你需要从子项反查父项**(给一个子项,用 SQL 查出引用它的所有父对象),这种跨表 join/聚合才值得建表 + 索引。

**JSON 数组列就够**,当这个列表:① 只作为父对象的一部分被整行读出(读父对象 → 顺带拿到它的列表);② 不需要按子项做 SQL 反查。

判断方法 —— 看真实访问模式:如果所有读写都是"加载父对象的整体数据,再在内存里过滤/遍历这个列表",根本不走 SQL join,那就没有任何地方需要反查,内存过滤就够。

此时中间表是冗余的:

- 它唯一的边际价值(数据库强制 pair 唯一)用业务层校验就能做,不值一张表。

- 领域模型里它本就是 `[]string`,biz↔data 直接字段对拷,落库事务还少写一张表。

**结论:这种情况去掉中间表,改 JSON 数组列。**
