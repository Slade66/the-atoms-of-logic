# 虚拟机跨平台迁移：驱动注入的底层逻辑与机制

- 在将虚拟机从 VMware 迁移到华为 FusionCompute（底层为 KVM）的过程中，“驱动注入”是决定系统能否成功启动的关键。

### 1. 核心痛点：为什么会发生 Kernel Panic / 蓝屏？

- 这是典型的“硬件环境变更导致驱动脱节”问题。

- **环境差异**：VMware 虚拟出的硬盘控制器通常是 LSI Logic 或 PVSCSI；而 FusionCompute（KVM）底层使用的是 VirtIO。

- **驱动缺失**：当你在 VMware 上安装系统时，系统的 `initramfs`（Linux）或引导驱动（Windows）只打包了适配 VMware 的驱动。

- **启动冲突**：
    
    - **形象比喻**：内核醒来后，从口袋里掏出“VMware 钥匙 A”试图开启眼前的“VirtIO 锁”，发现钥匙不匹配。
    
    - **后果**：系统将因找不到启动盘而无法正常引导。Linux 通常会直接跌入 dracut 救援模式（Rescue Mode），Windows 则会出现典型的蓝屏。

### 2. 解决方案：劫持分区的“外科手术”

- “注入驱动”本质上是在虚拟机开机前，对其虚拟硬盘执行一场“离线外科手术”。

-   **操作原理**：将虚拟机巨大的磁盘镜像（如 `test-xiaoze-flat.vmdk`）当成一个离线硬盘挂载到跳板机。

-   **强行注入**：命令系统：“把你冷宫底下藏着的 `virtio` 驱动翻出来！立刻马上，塞进那个用来开机的 `initramfs`急救包里去！”

-   **关键命令示例**：`virt-customize -a xxx --run-command "dracut -f --add-drivers 'virtio...'"`

### 3. 不同操作系统的适配核心

为了确保迁移后的系统能适应新环境，需要针对性地进行深度改造：

-   **Linux 系统改造**：
    
    -   **路径更新**：修改 `/etc/fstab`，将 `/dev/sda` 类路径换成 UUID 挂载，防止盘符识别偏移。
    
    -   **模块载入**：显式注入 `virtio-blk`（块设备）和 `virtio-net`（网卡）内核模块，并重新生成 `initramfs` / `initrd` 镜像。

-   **Windows 系统改造**：
    
    -   **注册表手术**：由于无法开机，必须使用离线注册表编辑（Offline Registry Edit）技术。
    
    -   **驱动部署**：将 VirtIO 的 `.sys` 驱动文件塞入 `System32/drivers`，并在 `HKLM\SYSTEM\ControlSet001\Services` 中注册这些驱动为“引导级启动”。

### 4. 实战建议：效率与自动化的权衡

- **利器推荐**：虽然手动注入有助于理解底层逻辑，但开源工具 `virt-v2v` 已经将上述复杂的“手术过程”封装得非常成熟。

- **开发路径**：如果是进行 PoC 开发，建议直接在后台封装并调用 `virt-v2v` 命令。它会自动处理磁盘解析、驱动查找、路径修改到注入的全生命周期工作。
