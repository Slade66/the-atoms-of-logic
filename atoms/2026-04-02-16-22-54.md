# 为什么会迁移失败？

### 缺少存储驱动

- VMware 默认使用的可能是 LSI Logic SAS 或 VMware Paravirtual 磁盘控制器，而 FusionCompute（基于 KVM）使用的是 VirtIO 磁盘控制器。如果源端虚拟机内核的 initramfs 中没有预先打包 virtio_blk、virtio_scsi 等 VirtIO 驱动，系统在启动时就会处于“瞎子”状态——它根本认不出插在主板上的新硬盘，自然就报 root does not exist。

- Linux 的引导程序（initramfs）在启动时找不到它的根文件系统（C盘/系统盘所在的硬盘分区），导致无法完成启动流程，最终跌落到了紧急命令行模式（Dracut Emergency Shell (Dracut 救援模式)）。

### 磁盘设备名改变

- 在 VMware 中，磁盘可能被识别为 /dev/sda，而在 KVM/FusionCompute 变为了 /dev/vda。如果引导配置（Grub/fstab）对路径或名称有硬编码依赖，也会找不到。

- 在 Linux 世界，根据底层连接磁盘的控制器和驱动的不同，块设备（硬盘）的名称会有严格的约定：

    - VMware (IDE/SCSI/SATA/SAS)： 通常加载 mptspi, vmw_pvscsi, ahci 等驱动，内核把它们识别为 SCSI/SATA 设备，因此磁盘设备名叫做 /dev/sda、/dev/sdb。

    - KVM / FusionCompute (VirtIO 半虚拟化)： 为了追求极致的 IO 性能，KVM 使用了 VirtIO 技术。加载 virtio_blk 驱动后，内核会把它们识别为虚拟块设备，名称变成了 /dev/vda、/dev/vdb。

- **为什么会引发灾难？** 如果您的 CentOS 系统当年在 VMware 安装时，它的 /etc/fstab 或 Grub2 引导参数写死了绝对路径，像这样：
    
    - `/etc/fstab` 里面的内容：`/dev/sda3  /  xfs  defaults 0 0`
    
    - `Grub` 启动参数带了：`root=/dev/sda3`
    
    - 当虚拟机被搬到 FusionCompute 上并以 VirtIO 方式启动时，原先的 `/dev/sda3` 在物理意义上变成了 `/dev/vda3`。Linux 内核启动后，傻乎乎地去寻找叫 sda3 的设备作为它的“家（根目录）”，怎么找也找不到，于是就会崩溃并跌落到应急 Shell。
