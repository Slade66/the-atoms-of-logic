# data 层按领域拆分子包

## 一、遇到的问题

### 表层：文件即将爆炸

- **原布局** `internal/data/` 下平铺 `data.go`、`greeter.go`、`esxi_host.go`、`fc_host.go`，看起来还好（4 个业务文件）。

- **即将新增 7 个文件** 再加进来会把平铺目录撑爆。

    | 待加文件 | 归类 |
    | --- | --- |
    | `esxi_dial.go` / `esxi_vm.go` / `esxi_datastore.go` / `esxi_portgroup.go` | ESXi 远端查询（4 个） |
    | `fc_dial.go` / `fc_vm.go` / `fc_datastore.go` | FC 远端查询（3 个） |

### 深层：职责与命名失控

- **职责混杂** 同一层装“本地 DB CRUD”（事务、持久化）和“远端 SOAP 只读”（网络、会话），读代码只能靠文件名前缀区分。

- **命名被迫加前缀** 所有类型都要 `Esxi*` / `Fc*`，全符号名带业务前缀噪声。

## 二、解决方案

### 按领域切子包

- **目录结构** 把平铺的 `internal/data/` 拆成按领域的子包。

    ```
    internal/data/esxi/
    internal/data/fc/
    ```

- **文件落位** 子包内按资源命名，不再带领域前缀。

    ```
    internal/data/esxi/host_repo.go   # EsxiHostPO → HostPO、NewEsxiHostRepo → NewHostRepo
    internal/data/fc/host_repo.go     # FcHostPO  → HostPO、NewFcHostRepo  → NewHostRepo
    ```

### 命名约定：子包符号去领域前缀

- **规则** 包名已带领域信息，符号不再重复 `Esxi` / `Fc` 前缀。

- **示例** 用 `esxi.HostPO`，不叫 `esxi.EsxiHostPO`。

## 三、防静默破坏：显式锁表名

- **隐患** PO 改名后 GORM 默认表名会变，已有环境会 `AutoMigrate` 出新空表、历史数据“消失”（其实在旧表里只是没人读）。

- **通用原则** 凡是涉及外部可观察行为（表名、API 路径、配置 key、环境变量、默认值）的改动，都显式锁死，别依赖框架默认推导。
