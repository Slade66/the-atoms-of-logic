# bifrost CLI 配置迁移到 Web 表单的方法论反思

## 一、事件日志

- **任务背景** 2026-04-23 上午，准备把 `bifrost-migrate` 从 CLI 形式推进到 Web 形式，需要从 CLI 配置文件中推导 Web 迁移任务创建表单。

- **核心问题** 这次任务要解决两个问题：CLI 的 `migration.yaml` 字段不能直接平铺到 Web 表单里，需要判断每个字段的真实来源和 UI 形态；Proxy VM 和 Helper VM 这类“每个主机只部署一台”的对象，也需要判断它们到底属于迁移任务字段，还是宿主对象属性。

- **配置分析** 先让 AI 分析 CLI 配置文件，建立字段清单和来源分类。

    - 按 YAML 小节列出原始清单：`source` / `target` / `advanced`。
    - 按数据来源重排为 6 大分类：已持久化、远端查询、主机配属、每次必填、自动探测、默认值。
    - 通过凭证持久化，让表单从 22 个必填字段塌缩到 8 个必填字段。

- **方案判断** 和 AI 讨论 Proxy VM / Helper VM 的填写方案，重点判断 1:1 强约束下它们更适合作为 ESXi 连接的属性，而不是迁移任务表单字段。

- **文档产出** 将分析结果沉淀为文档：`docs/20260423T113300 - 需求分析 - 迁移任务创建表单字段梳理与来源设计.md`。

    - 字段清单。
    - 来源分类。
    - Proxy / Helper 方案对比。
    - 最终表单形态。
    - 后续待办。

- **方法抽象** 文档提交后，再从这次任务中抽取可复用经验：CLI / 配置文件迁移到 Web 表单时，先做字段来源归类，再做业务建模判断，最后做表单分层。

## 二、深度反思

- **配置不等于表单** 面对任何 YAML / JSON / TOML 配置，不能直接把配置项平铺成 Web 表单；必须先把每个字段按来源贴标签，因为来源不同，UI 形态完全不同，真正需要用户填写的字段通常只有原配置的 1/3 到 1/2。

    | 来源类型 | UI 形态 | 抽象例子 |
    | --- | --- | --- |
    | 已持久化 A | 下拉 | 已保存的外部系统连接、项目、账号 |
    | 远端动态查询 B | 下拉 + 搜索 | 从外部系统实时查到的资源、实例、目录 |
    | 主机配属 C | 不在表单里 | 随宿主对象绑定的代理、默认执行器、固定网关 |
    | 用户每次必填 D | 输入框 | 本次任务名称、临时凭证、业务备注 |
    | 后端自动探测 E | 完全隐藏 | 证书指纹、资源唯一标识、环境版本 |
    | 有默认值 F | 折叠高级区 | 超时时间、重试次数、并发数 |

- **一对一约束** 如果业务上明确“每个 X 只有一个 Y”，那通常意味着 Y 是 X 的属性，而不是一个需要单独出现在业务表单里的对象；Proxy VM 确实是一台 VM，但在这次业务域里，它是 ESXi 连接的一个属性。

    - 业务上数量严格为 1，不多也不少。
    - 生命周期与宿主绑定，宿主没了，它就没意义。

- **表单分层** 设计多字段表单时，先用 Must / Should / Could / Won't show 分层，决定哪些字段放主区域，哪些字段放规格区、高级折叠区或完全隐藏；不要一上来给用户砸一个 20 字段的平铺表单。

    | 层级 | 表单位置 | 判断标准 |
    | --- | --- | --- |
    | Must | 主区域 | 每次都要决策、没有合理默认值 |
    | Should | 规格区 | 经常要看一眼，但有默认值 |
    | Could | 高级折叠区 | 偶尔要调，平时全默认 |
    | Won't show | 隐藏 | 后端自动推导或从其他对象继承 |

- **文档先行**
    
    - **文档的价值** 文档的真实价值不是“写下来”，而是强迫自己在写代码前把多种方案列出来对比，避免把架构成本推到后面。

    - **返工风险** 如果跳过文档直接写 proto 和代码，很可能先把 Proxy / Helper 当作迁移表单字段写进 proto，做到一半才发现“每个主机只有一台”的约束，最后返工拆字段、改 proto、改前端、改 DB schema；一页 Markdown 换几天返工，成本很低。

    - **检索关联** 文档要可检索、可关联：用 `YYYYMMDDTHHMMSS - 类型 - 标题.md` 命名，并加 YAML front matter 关联文档字段，让后来的人能快速跳到上下文。
