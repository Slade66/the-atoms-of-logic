# bifrost V2V 迁移的网络配置问题

### 背景

在异构虚拟化迁移（V2V）中，网卡变化不是“小问题”，网络配置的无缝转换是确保业务连续性、避免迁移后服务中断的核心难题。

会导致的问题：

目标 VM 开机但网络不通
迁移割接后最容易让业务“系统已开机但没人连得上”的情况。

VMware 的 vmxnet3/e1000 变成 FusionCompute/KVM 的 virtio-net。

Linux 可能接口名变化，比如 ens160 变成 enpXsY。

Linux 静态配置可能失效：如果原配置绑定旧接口名、旧 MAC、旧 ifcfg HWADDR、旧 netplan match.macaddress
Linux 原配置可能绑定旧接口名或旧 MAC。

Windows 会把 VirtIO 网卡识别成一块新网卡，旧 VMware 网卡变成隐藏设备。Windows 静态 IP 留在旧的 VMware 隐藏网卡上。设置 IP 时可能提示该 IP 已分配给另一个隐藏适配器，原因是注册表里还保留旧网络适配器；解决方式是显示隐藏设备并卸载旧适配器。

安全策略失效
防火墙 ACL、堡垒机白名单、NAC、IP-MAC 绑定、EDR/监控资产识别，很多都依赖 IP/MAC。
网卡 MAC 变了，可能被当成新主机；IP 变了，规则可能完全不匹配。

业务许可证失效：一些软件会绑定 MAC 地址，mac 变更可能导致应用需要重新激活。

路由/DNS 错误
默认网关丢失、多个网卡都有默认网关、DNS 仍指向旧环境，都会导致“同网段能 ping，跨网段/域名/AD 不通”。
Azure 文档也特别提醒，多静态网卡都带默认网关会导致非对称路由、丢包和不可预测网络行为。

写死 IP的应用程序会炸。

### Windows

Windows虚拟机V2V迁移到华为FusionCompute后网络配置失效

华为FusionCompute用的是virtio半虚拟化网卡（一种"虚拟机专用网卡"），迁移过来后系统识别为一个"全新硬件"，老网卡的IP配置以"幽灵网卡"（隐藏设备）的形式残留在注册表里，导致手动设给新网卡时报"IP已被另一个适配器占用"



### 解决方案

保留VM 的 IP 和 MAC，应用改动最少。



迁移前采集，迁移后脚本注入：

    - 迁移前在 Windows/Linux 内运行准备脚本，采集网卡配置，创建启动任务，目标 VM 首启后应用静态配置。

    - MAC 地址保留：在调用 FusionCompute 虚拟机创建 API 时，显式传递源 VMware VM 的 MAC 地址（这对于绑定 MAC 授权的商业软件非常关键，FC API 是支持指定自定义 MAC 的）。

    - IP 地址恢复方案：
        在线预置脚本，
        应通过 GuestOps 在源 VM 内跑脚本采集 ip 地址等信息
        在源端虚机尚未关机前，通过 GuestOps 注入在目标端首启时执行的自注册网络配置恢复脚本，并在配置完成后自我销毁。

但必须确保源 VM 已关机或断网后再启目标 VM，否则同网段 MAC/IP 冲突。如果源 VM 没断网，目标 VM 复用同 IP 或 MAC，会造成 ARP 抖动、连接随机漂移。

在 Cutover 后加一个 NETWORK_CONFIG 阶段，通过 GuestOps 给目标注入启动脚本完成 guest 内配置。
不要只做“把源 IP 写到目标 VM”。要做成一个明确的 Network Migration Plan：源网卡识别、目标端口组映射、是否复用 MAC、静态/DHCP、IP/prefix/gateway/DNS/routes、多网卡主从关系、Windows 隐藏旧网卡处理、Linux 接口名/MAC 绑定修正、割接后连通性验证。这样才接近业界迁移产品的做法。
