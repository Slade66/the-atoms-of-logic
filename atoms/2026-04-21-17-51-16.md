# Proto 设计约定

## 一、空请求也写自己的空 message

- **不要用 `google.protobuf.Empty`** 请求或响应即使当前没有任何字段，也写一个自定义的空 message，而不是直接复用 `google.protobuf.Empty`。

- **理由** 未来要加 `page` / `filter` 等字段时，自己的 message 可以直接扩；一旦用了 `Empty`，相当于把这个 rpc 的签名钉死，后续加字段就是破坏性变更。

## 二、proto 按"领域"组织，而不是"资源"

### 约定：文件夹 = 领域，文件名 = 资源

- **Kratos 官方规范** 文件夹是领域（domain），一组相关资源共用的命名空间；文件名是资源（resource），一个资源一个 `.proto` 文件。

- **示例** `esxi` 是领域，领域下有 `esxi_host` / `esxi_datastore` / `esxi_vm` 等多个资源，各占一个 `.proto` 文件。

### 两种目录方案对比

- **方案 A：按领域分文件夹（Kratos / Google AIP 推荐，当前采用）**

    ```
    api/
    └── esxi/v1/                         ← 一个 Go 包：esxi.v1
        ├── esxi_host.proto
        ├── esxi_datastore.proto
        └── esxi_vm.proto
    ```

- **方案 B：按资源分文件夹（不推荐）**

    ```
    api/
    ├── esxi_host/v1/                    ← Go 包：esxi_host.v1
    │   └── esxi_host.proto
    ├── esxi_datastore/v1/               ← Go 包：esxi_datastore.v1
    │   └── esxi_datastore.proto
    └── esxi_vm/v1/                      ← Go 包：esxi_vm.v1
        └── esxi_vm.proto
    ```

### 为什么选方案 A：文件夹 = Go 包边界

- **protoc 的包规则** 生成的 `.pb.go` 以"文件夹"为 Go 包单位，跟 Go 自己的包规则一致。方案 A 是 1 个 Go 包，方案 B 是 3 个 Go 包。

- **方案 B 的导入噪音** 业务代码用到同一领域下多个资源时，要挨个 import 子包。

    ```go
    import (
        hostv1 "gitea.nt.cn/nantian/bifrost/api/esxi_host/v1"
        dsv1   "gitea.nt.cn/nantian/bifrost/api/esxi_datastore/v1"
        vmv1   "gitea.nt.cn/nantian/bifrost/api/esxi_vm/v1"
    )
    ```

- **方案 A 的导入简洁** 同领域资源共享一个 Go 包，只要一行。

    ```go
    import esxiv1 "gitea.nt.cn/nantian/bifrost/api/esxi/v1"
    ```

### 什么时候再拆分顶层文件夹

- **经验阈值：30 个 `.proto`** 同一文件夹下超过 30 个 `.proto`，说明"领域"定义过宽，该往下拆成子领域。

- **拆分示例** `esxi/v1/` 如果膨胀到"管理 + 监控 + 备份"三块、各 10+ 个资源，就拆成 `esxi_manage/v1/` / `esxi_monitor/v1/` / `esxi_backup/v1/`，每个子领域内部仍按方案 A 组织。
