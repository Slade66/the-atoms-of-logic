# 为什么 CPU 代数不同会导致迁移失败？

指令集（Instruction Sets）的代差
CPU 每一代的演进，除了主频和核心数的提升，最重要的就是引入了新的硬件指令集（如 Intel 的 AVX-512, AES-NI, SSE4.2 等）。

加速原理：软件（如数据库、加密软件、视频编码器）为了跑得快，会检测当前 CPU 支持哪些高级指令。如果发现 CPU 支持“高级指令 A”，它就会直接调用 CPU 内部的硬电路来完成计算。

迁移矛盾：

在主机 A（新）上：虚拟机启动，应用检测到支持 AVX-512 指令，开始开心地调用。

热迁移到主机 B（旧）时：虚拟机内存里的代码已经运行到了调用 AVX-512 的那一步。但主机 B 的 CPU 根本没有这个电路。

结果：CPU 遇到无法识别的代码，会抛出一个 “非法指令（Illegal Opcode）” 异常。对于操作系统来说，这就是致命错误，直接导致蓝屏（Windows）或内核恐慌（Linux）。

当操作系统或应用程序启动时，它们会执行一个叫 CPUID 的特殊指令。这个指令就像是问 CPU：“你是谁？你能干什么？”

静态性：大多数软件只在启动时执行一次 CPUID 检查。

迁移后的尴尬：虚拟机在旧主机上启动时，被告知“我不支持高级特性”。迁移到新主机后，虽然新硬件支持了，但软件并不知道，这只是性能浪费。

致命的尴尬（反向迁移）：如果虚拟机在新主机启动并记住了“我支持高级特性”，迁移回旧主机后，软件依然按原计划发射指令，就会瞬间撞墙。
