# 为什么 CPU 代数不同会导致热迁移失败

### 热迁移崩溃机制

- **根因** 操作系统启动时发送 CPUID 探测指令，CPU 回答后软件据此加载代码路径。大多数软件只在启动时执行一次，之后默认硬件特性永远存在——VM 不会在运行中重新检查硬件。

- **崩溃过程**

    1. **启动探测** VM 在新 CPU（如第 12 代 Intel）上启动，CPUID 返回支持 AVX-512 等高级指令集，软件立刻加载高级代码路径，内存中充满针对新 CPU 优化的数据流（如使用 512 位 ZMM 寄存器）。
    2. **盲目信任** 运行中软件默认硬件特性不变，计算任务以“512 位/次”的速度狂奔。
    3. **硬件背叛** 热迁移到旧 CPU（如第 10 代 Intel）后，内存和寄存器状态原封不动复制过去，VM 继续发射 AVX-512 指令，但旧 CPU 根本没有对应电路，抛出“非法指令（Illegal Opcode）”异常，操作系统蓝屏或内核恐慌。

- **寄存器不兼容** 新 CPU 的 512 位寄存器像大抽屉，旧 CPU 的 256 位寄存器像小抽屉，热迁移把 512 位的数据原样带走，塞不进 256 位的抽屉，物理上不兼容。

- **方向决定生死**

    - **旧→新（正向）** VM 在旧 CPU 上启动时未使用高级指令集，迁移到新 CPU 后只是性能浪费，不会崩溃。
    - **新→旧（反向）** VM 在新 CPU 上启动并使用了高级指令集，迁移回旧 CPU 后指令无法执行，瞬间崩溃。

### 冷迁移不受影响

- **场景** 把 ESXi 的 VM 磁盘数据拷贝到 FC 的存储里再开机，属于冷迁移（V2V 转换），不是热迁移。

- **为什么安全** 关机后搬运数据，VM 在 FC 上开机时会重新走完整的引导流程，重新发送 CPUID 探测新主机的 CPU，软件根据新答案自适应加载对应的驱动和指令路径。只要同品牌 CPU（如都是 Intel），代数不同不会导致崩溃。

- **类比** 热迁移是让正在百米冲刺的运动员中途换赛道；冷迁移是先让运动员睡一觉，搬到新赛场再重新起跑。
