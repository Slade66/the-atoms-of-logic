# virt-v2v

- virt-v2v (Virtual to Virtual) 是由 Red Hat 开发的一款开源工业级虚拟机迁移工具。它是 libguestfs 工具包中最核心、最重量级的组件。

- 当把一台虚拟机从非 KVM 平台（例如 VMware ESXi、Microsoft Hyper-V、Xen）迁移到 KVM 平台（如基于 KVM 的 FusionCompute）时，virt-v2v 就像一个极其专业的“外科医生”，不仅把虚拟机的身体（硬盘数据）搬运过去，还会替它做“器官移植和神经缝合（系统层面的深度改造）”。

- 它解决的根本问题是：“仅仅把硬盘数据复制过去是不够的，底层硬件更换后，操作系统会水土不服而死机（即您遇到的 Dracut 报错）”。

- 它全自动完成了以下工作：

    1. 自动识别系统架构： 智能分析这块盘里装的是 Windows 10、CentOS 7 还是 Ubuntu 20，并自动挂载核心分区。
    
    2. 重塑网络与存储神经（注入 VirtIO）： 自动根据识别出的操作系统，把 KVM 所需的 virtio-blk、virtio-scsi、virtio-net 驱动精准地打入内核和 initramfs（Linux）或者注册表（Windows）中。
    
    3. 清洗异体排斥物（卸载旧工具）： 自动搜寻并卸载 VMware Tools 或 Hyper-V Integration Services 等旧平台残留。
    
    4. 清理硬编码路标（修复引导和挂载）： 扫描 /etc/fstab 和 Grub 配置文件，自动将硬绑定的 /dev/sda 转换为 UUID 或者 /dev/vda 格式。
    
    5. 植入新平台神经元： 顺手帮您安装好 qemu-guest-agent，让新平台能直接读取虚拟机的 IP 并优雅关机。

- 旧流程：VMware 导出 vmdk -> qemu-img 转码为 qcow2 -> virt-customize 敲命令强插 dracut 驱动。
    
    - 这个流程本身逻辑是对的，非常适合用来做技术验证（PoC），但在工程化实现时，会遇到无数的“分支炼狱”（比如 CentOS、Debian、Ubuntu 的内核获取方式都不一样，还有 Grub/fstab 需要修改等情况）。请看对比：

    - 
        | 面对的问题 | qemu-img + virt-customize | virt-v2v 解法 |
        | --- | --- | --- |
        | Linux 发行版差异 | CentOS 用 dracut，Ubuntu 用 update-initramfs，Debian 路径又不同。您需要在代码里写海量的 if/else 来判断并下发不同 shell 命令。 | 无需操心。 它内部封装了所有主流 Linux 的驱动注入逻辑，无论是红帽系还是 Debian 系，一套代码平推。 |
        | Windows 虚拟机支持 | virt-customize 无法运行 .exe 驱动包。您如果是迁 Windows 机器，几乎无法通过这种方式自动化注入 VirtIO 驱动，必定蓝屏。 | 完美支持。 只要本地挂载并指定包含 virtio-win.iso 的目录，它会直接注入 Windows 注册表和驱动库，迁移后开箱即用。 |
        | fstab 和 Grub 错误 | /dev/sda 如果改成了 KVM 的 /dev/vda 您需要用正则或者 sed 去小心翼翼地修改虚拟机内部的系统配置表。极易改挂系统。 | 自动解析并安全替换。它会把挂载点转化为 UUID，彻底消除设备名强依赖的雷区。 |
        | 卸载 VMware Tools | 您必须自己写卸载脚本传进去执行。 | 自动卸载。 包含各代 VMware Tools 的静默卸载逻辑。 |

- 怎么用？

    1. 安装

        - 如果是 CentOS / RHEL：
            
            ```bash
            sudo yum install -y virt-v2v
            # (可选) 如果打算迁移 Windows，务必安装 virtio-win 驱动包，它会自动从中提取驱动
            sudo yum install -y virtio-win 
            ```
        
        - 如果是 Ubuntu / Debian：
        
            ```bash
            sudo apt update
            sudo apt install -y virt-v2v
            # Windows 驱动通常可以在网上单独下载 virtio-win.iso 供后续挂载
            ```

    2. 执行

        -
            ```bash
            # 1. 创建一个存放处理后产物的空文件夹
            mkdir -p ./v2v_output

            # 2. 核心魔法命令：自动检查、注入驱动并转为 qcow2
            virt-v2v -i disk test-xiaoze.vmdk \
                    -o local -os ./v2v_output \
                    -of qcow2
            ```
        
        - `-i disk`: 告诉工具“输入源(Input)只是一个裸镜像磁盘，请顺藤摸瓜处理”。它会自动去读 test-xiaoze.vmdk 并关联拉起 -flat.vmdk 的真实数据。
        
        - `-o local`: 输出模式为本地文件。
        
        - `-os ./v2v_output`: Output Storage 指定输出成果文件存放在刚刚创建的文件夹里。
        
        - `-of qcow2`: Output Format 指定我要把转化并打好药剂的成品压成华为需要的 qcow2 格式。

- 还要拉取 `.vmx`
    
    - `virt-v2v` 并不认得这台虚拟机原来有几个 CPU，多少内存，网卡原来是什么 MAC 地址（因为这些信息不在硬盘里，而在 VMware 配置文件里）。
    
    - 把 VMware 里那个几KB大小的配置文件 test-xiaoze.vmx 一起拉取到本地
    
    - `virt-v2v -i vmx test-xiaoze.vmx -o local -os ./v2v_output -of qcow2`
    
    - 执行完后，./v2v_output 目录下不仅会生成完美的 qcow2 硬盘，还会多出一个叫 test-xiaoze.xml 的文件。这个 XML 文件里把源端虚拟机的 CPU、内存、网卡原班人马参数解析得清清楚楚。 
    
    - 后续用 Go 语言 (Bifrost) 对接华为 FusionCompute APIs 创建新虚机时，直接读取解析这个 XML 里的参数发给华为，连资源规格型号都能全自动 1:1 完美复刻过去。
