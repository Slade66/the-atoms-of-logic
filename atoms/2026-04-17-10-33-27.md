# 块设备

- **一段话解释：** 块设备（block device）是 Linux 内核对"可以按固定大小的数据块、随机寻址读写的存储硬件"的统一抽象。

- **块设备（block device）vs 字符设备（character device）**

    - **块设备：** 以固定大小的数据块（通常是 512 字节或 4KB）为单位进行读写（不能只读 1 个字节），有 LBA（逻辑块地址），支持随机访问，可以装文件系统。

    - **字符设备：** 顺序的字节流，以字符为单位进行读写，不支持随机访问，不能装文件系统。

- **怎么在 `ls` 里一眼认出块设备**

    ```bash
    $ ls -l /dev/sda /dev/tty
    brw-rw---- 1 root disk 8, 0 9月  21 2025 /dev/sda
    crw-rw-rw- 1 root tty  5, 0 4月  13 14:08 /dev/tty
    ```

    - 权限位的第一个字母表示设备类型：`b` = 块设备（block），`c` = 字符设备（character）。
    - 后面的 `8, 0` 和 `5, 0` 是**主设备号, 次设备号**：主设备号标识驱动（`8` = sd 驱动），次设备号标识该驱动下的第几个设备或分区。内核通过 `(major, minor)` 找到对应的驱动来处理 I/O。

- **块设备的抽象模型：一块"可寻址的大字节数组"**

    - 无论底层是机械盘、SSD，还是一个被 loop 挂载的普通文件，块设备在内核看来都是同一种东西——一块按固定大小切片、可随机寻址的大字节数组。

        ```text
            LBA:      0    1    2    3    4    ...   N-1
                    ┌────┬────┬────┬────┬────┬─────┬────┐
            设备 =  │blk │blk │blk │blk │blk │ ... │blk │   每块 512B 或 4KB
                    └────┴────┴────┴────┴────┴─────┴────┘
                    ▲                                  ▲
                    │   可以对任意 LBA 发起 读 / 写      │
                    └──────────────────────────────────┘

            容量 = 块数 × 块大小
        ```

    - 内核对块设备只关心两件事：

        1. 给定一个 LBA 和一块内存缓冲区，把数据从内存搬到设备，或从设备搬回内存。
        
        2. 这个"搬运意图"由 `struct bio` 描述，最终交给具体的驱动去执行。

        ```text
                 应用 / 文件系统
                       │
                       │  只说："把这 8 个 4KB 写到 LBA=10000"
                       ▼
          ┌──────────────────────────┐
          │      块设备抽象层         │   ← 统一接口（bio / request）
          └──────────────────────────┘
            │        │        │       │
            ▼        ▼        ▼       ▼
           HDD      SSD     NVMe    loop(文件)
         (机械臂) (闪存FTL) (PCIe)  (再走一次 VFS)
        ```

- **把"块设备"理解成一个接口（interface）**

    - 这个接口只规定两件事：
        1. 能接受 bio（有 `submit_bio` 入口）。
        2. 完成时回调 `bi_end_io` 通知提交者。

    - 任何东西 —— 真硬件也好，软件合成的虚拟设备也好 —— 只要实现了这个接口，在上层眼里就是一块盘。
    
    - 上层（文件系统、数据库）只会说一种语言：`submit_bio`。它不关心底下是旋转磁盘还是闪存颗粒，是物理盘还是 loop 文件，是本地盘还是网络盘 —— 只要对方认得 `submit_bio`，它就把运单递过去，剩下的事交给对方。

- **块设备抽象层的价值：一个接口把两边隔开，各自都能自由变化**

    - **文件系统写一次，到处能跑** ext4 代码里没有一行 NVMe 特判，它装在机械盘、SSD、U 盘、RAID、LVM、加密盘、loop 文件上都一样工作 —— 因为它只和 "块设备接口" 对话，根本不知道底下是谁。
    
    - **新硬件出来，上层不用改** NVMe 问世时，只要写一个 NVMe 驱动实现块设备接口，ext4/xfs/btrfs 就能直接用；反过来，btrfs 出来时所有现有驱动也不用动。两边像插槽和插头一样独立迭代。

- **bio 是什么？**

    - `bio`（Block I/O）是内核里描述一次块设备 I/O 请求的数据结构，是上层和块层之间的契约对象——只描述"要做什么"，不关心"怎么做"。
    
    - 可以把它理解成一张运单："请把这几页内存，读/写到这个块设备的这个 LBA 上，完成后回调我。" 它不是数据本身，而是对一次搬运动作的描述。
    
    - 上层（通常是文件系统）调用 `submit_bio(bio)` 之后就不管了。剩下的流水线是：
        - 块层把 bio 塞进 request、交给调度器。
        - 调度器派发给驱动，驱动下发到硬件。
        - 硬件完成 → 中断 → 块层调用 `bi_end_io` → 通知原提交者。
