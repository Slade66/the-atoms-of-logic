# 日志系统的格式用 JSONL

## 一、什么是 JSONL

- **定义：** JSONL = JSON Lines，即“一行一个 JSON”——很多条 JSON 记录，每条占一行，用换行隔开。

- **格式特征：** 没有最外层的 `[ ]` 数组，行尾也没有逗号，每一行单独拎出来都是一个完整合法的 JSON 对象。

- **示例：**

    ```json
    {"seq":1,"level":"INFO","msg":"建快照成功"}
    {"seq":2,"level":"WARN","msg":"建卷失败，重试"}
    {"seq":3,"level":"INFO","msg":"传输完成"}
    ```

## 二、为什么日志适合用 JSONL

- **能一直往后追加（append-only）：** 日志是边跑边产生的，新来一条就在文件末尾加一行即可。

- **能一行一行读，不用整个加载：** 想看第 1000~1500 条，跳到那几行读出来就行，文件几个 G 也不卡。

## 三、实践中的应用

- **Docker 落盘（`json-file` 驱动）：** 就是一行一条 JSON，直接把 `{"log":...,"stream":"stdout","time":...}` 一行一条写进磁盘文件。

- **源头与传输的公认格式：** “一行一条 JSON”是大家公认的“日志源头 / 传输”格式。

- **存储层就不再用 JSONL 了：** JSONL 只负责把日志“产生出来、传输”；真正落到后端长期存储时，大厂各自换成了带索引的重型格式，目的是能跨成千上万台机器做“全文秒级检索”。

    - **Elasticsearch：** 建 Lucene 全文倒排索引，按词查得快。
    
    - **Loki：** 把日志压成 chunk，只给标签建索引，省存储。
    
    - **Datadog：** 用自家的专有格式。
