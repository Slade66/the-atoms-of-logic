# 如何防盗版

### 威胁清单

1. **拷贝机器** 客户把 bifrost.exe 和 license.json 一起拷贝到第二台机器运行，或者把整套工具转卖给第三方公司。这是最常见的盗版场景，必须防住。

2. **延期作弊** 客户试图延长已过期的 license，手段包括把系统时钟调回半年前，或者直接打开 license 文件把 expires_at 字段改掉。

3. **扩容作弊** 客户修改 license 里的迁移额度字段，把 50 台改成 500 台，企图白嫖更多配额。

4. **逆向破解** 客户反编译 Go 二进制，定位到校验函数，直接 patch 掉 return 语句让所有检查通过。这是成本最高但最彻底的攻击方式，纯软件方案只能缓解不能根治。

### 防篡改：数字签名

- **思路** license 不是"一个 JSON 文件"，而是"JSON 文件 + 一段用厂商私钥生成的签名"。客户改任何一个字节，签名就对不上，工具立刻识破。

- **比喻** license 好比一张盖了钢印的合同——合同正文（payload）写着客户名、机器指纹、有效期、配额；钢印（signature）是厂商用只有自己有的私钥对整份合同算出来的防伪码。

- **签发** 厂商在签名机上执行 `signature = Ed25519_Sign(私钥, payload)`，把 payload 和 signature 打包成 license.json 发给客户。

- **校验** 客户端启动时读取 license.json，用编译时内嵌的厂商公钥执行 `Ed25519_Verify(公钥, payload, signature)`。验不过直接退出；验过了才去读 payload 里的过期和配额字段做业务判断。

- **攻击场景** 客户想把配额从 50 改成 500，JSON 是明文可以随便改，但钢印是根据原合同算出来的，改了内容钢印就对不上。客户想自己伪造钢印做不到——厂商私钥在签名机上，客户手里没有；就算他用自己的密钥签一张假 license，工具里写死的公钥也和他的私钥不配对。

- **唯一出路** 客户想绕过签名校验，只剩逆向二进制替换掉内嵌的公钥这一条路，这已经属于威胁 4 的范畴。

### 防拷贝：硬件指纹

- **思路** license 必须包含一个"不会跟着文件一起被拷贝"的锚点，这个锚点就是目标机器的硬件指纹。指纹跟着机器走，不跟着文件走，所以单纯拷贝 license 到别的机器没用。

- **指纹构成** 把多个稳定硬件 ID 拼接后 SHA256，任何一项变化都会导致指纹失效，包括 CPU 型号和核心数、物理网卡 MAC、主板 UUID 等。

- **校验** 程序启动时现场计算本机指纹，和 license.payload 里冻结的指纹比对，不匹配直接拒启动。

- **容差策略** 指纹不能做死板的全等比对，否则客户换一块网卡、升级一根内存就跑不起来。定义"核心特征全中 + 辅助特征允许一定程度的漂移"，兼顾安全性和售后成本。

- **重签防滥发** 客户想让你重签一张给新机器，厂商侧在台账里查"这个客户这个项目已经签过"就拒绝重签，防止客户借"换机器"名义拿到多张 license。

### 已知局限：虚拟机克隆

- **问题** 虚拟机的硬件 ID 都是 hypervisor 伪造出来的，克隆 VM 时这些字段原样复制，导致两台 VM 的指纹一模一样，硬件指纹防线被绕过。

- **场景 a** 客户先克隆 VM，再申请 license，license 在两台机都可用。

- **场景 b** 客户在 VM-1 上申请到 license 后再克隆 VM，license 在两台 VM 上都能用。

### 防密钥替换：公钥内嵌

- **问题** 如果厂商公钥放在外部文件里，客户自己生成一对密钥、用自己的私钥签一张假 license、再把公钥文件替换掉，就能伪造出"合法"的 license。

- **对策** 公钥必须硬编码进 Go 源码，编译进二进制，客户没法从外部替换。

- **加固** 二进制里的公钥最好拆成多段异或后运行时拼回，避免逆向者一眼就能在内存里 dump 出干净的 32 字节 Ed25519 公钥常量。

### 防计数归零：账本加多处 hash

- **朴素方案的漏洞** 最直觉的做法是硬盘上的文件存一个数字，但会被客户直接改掉，必须做得复杂一些。

- **防止修改次数：HMAC 校验码** 客户改任何一行字段，整份 HMAC 就对不上，工具拒跑。HMAC 的密钥由机器指纹生成，不存在文件里，客户算不出正确的签名。

    - **做法一：数字 + HMAC** 存一个计数器 + 一段 HMAC 签名，客户改数字就会让 HMAC 对不上。实现简单，存储开销小，适合只关心"用了多少次"的场景。

        ```json
        {
            "used": 3,
            "mac": "HMAC(密钥, used)"
        }
        ```

    - **做法二：账本 + HMAC** 不存单个数字，存"已迁 VM 清单"——每迁一台追加一条记录，包含 VM 名和时间，整份清单用 HMAC 签名。真正的次数等于列表长度，改不了长度就改不了次数。额外的好处是出问题时能查到具体迁过哪些 VM，便于审计和排障。V1 推荐这种。

        ```json
        {
            "已迁移": [
                {"vm": "web-01", "time": "2026-05-01"},
                {"vm": "db-02",  "time": "2026-05-03"},
                {"vm": "app-03", "time": "2026-05-10"}
            ],
            "mac": "整个列表的 HMAC"
        }
        ```

- **备份还原攻击** 客户先备份账本文件（或做系统快照、整盘镜像），再运行迁移，迁完后用备份覆盖回去。账本和所有本地 hash 副本一起回到迁移前状态，工具以为这台 VM 还没迁过。这是纯本地方案的软肋。

- **防止回滚账本：多处 hash 同步** 账本写完后算一个整体 SHA256，这个 hash 额外同步写到 Windows 注册表 HKLM\SOFTWARE\Bifrost\LedgerHash、%ProgramData%、license 同目录等多个位置。启动时读账本算 hash，再读多处存储取最新 hash，两者必须一致，不一致说明账本被回滚了，直接拒跑。

    - **局限** 多处存储只能抵抗客户手动的"单点回滚"，无法抵抗整机快照同步还原。

### 防时间回滚：双锁机制

- **问题根源** license 靠 `当前时间 > expires_at` 判断过期，但"当前时间"是操作系统说了算的，而操作系统时间是客户自己随便调的。客户右键任务栏的时钟调回半年前，license 就能永远"没过期"。

- **下限锁：issued_at** license.payload 里带一个厂商签发时间，因为有签名所以客户改不了。启动时加一道检查：`当前时间 < issued_at → 拒跑`。客户想把时钟调到签发日之前就会露馅。

- **下限锁的盲区** 只靠 issued_at 不够——客户把时钟停在 [issued_at, expires_at] 区间里任意一天，工具永远觉得"在有效期内"，license 可以无限续命。issued_at 只能锁下限，锁不住"时间停滞"。

- **递增锁：T_last** 工具每次跑都把此刻的系统时间偷偷写到硬盘。下次启动对比：T_now < T_last 说明时间倒流了，拒跑；否则更新 T_last = T_now 继续跑。这样客户调回已经用过的日期也会被识破。

- **防改 T_last** T_last 文件不能只存一行时间，要存"时间 + 防伪码"。防伪码 = HMAC(密钥, 时间)，密钥由机器指纹派生，不存在于文件中。客户改时间防伪码就对不上；客户想自己算防伪码却不知道密钥；从别的机器拷 T_last 过来，指纹不同密钥不同，防伪码同样验不过。

- **防删 T_last** 和账本一样，把 T_last 同步写到多个位置——Windows 注册表 HKLM\SOFTWARE\Bifrost、%ProgramData%、license 同目录隐藏文件等。启动时读取多处取最新值，任何一处还在就能恢复。

### 注册 & 校验流程

- **注册步骤**
    1. 客户在目标机器上执行 `bifrost fingerprint -o machine.fp`，生成该机器的指纹文件
    2. 厂商销售或运维收到客户发来的 .fp 文件，在签名机上用厂商私钥签发 license.json，并在发放台账里登记一条记录
    3. 客户把收到的 license.json 放到 bifrost 工具目录，启动时工具自动加载并校验

- **校验流程**
    1. 加载 license.json
    2. 用内嵌公钥验 signature（防伪造）
    3. 现场算本机指纹，比对 license 里存储的指纹（防拷贝）
    4. 检查 issued_at ≤ now ≤ expires_at（防过期）
    5. 检查 T_last ≤ now（防时间回滚）
    6. 检查本地账本长度 < vm_quota（防超量）
    7. 启动迁移流程

- **失败处理** 任何一步失败都直接退出，错误信息统一为"license 无效"，不暴露具体是哪一步挂了，给逆向攻击者增加定位难度。
