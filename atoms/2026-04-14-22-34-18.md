# 为什么不能将 VDDK SDK 放入 Git 仓库

- 解析 VDDK 禁止进入 Git 的法律合规性差异与工程性能损耗。

### 🧠 法律风险与合规红线

- **再分发禁令**：下载 VDDK 时必须同意 VMware 许可协议，条款中明确禁止对 SDK 进行“再分发”。
- **法律定性**：将 SDK 上传至托管于远端服务器的 Git 仓库（如 GitHub、GitLab），在法律层面即被视为分发行为。
- **合规差异**：即便在私有内网仓库风险较低，但在审计和合规性操作上依然属于违规操作。

### 🧠 Git 仓库的性能噩梦

- **二进制膨胀**：VDDK SDK 体积通常达数百 MB，且内部主要是 `.so` (Linux) 或 `.dll` (Windows) 二进制文件。
- **仓库停滞**：Git 的版本追踪机制并不擅长处理大型二进制文件。每当 SDK 版本更新，仓库体积都会呈几何倍数增长。
- **工程后果**：导致克隆（Clone）和拉取（Pull）速度极剧变慢，最终拖垮整个开发流水线。

### 🧠 行业通用的工程解法

- **本地留存原则**：行业惯例是将 SDK 保留在服务器本地。在项目根目录的 `README.md` 中清晰标注“环境依赖”，说明版本要求、下载路径及安装位置。

- ❌ **灾难做法：直接提交二进制文件**
    
    - **直接后果**：数百 MB 的二进制死重永久埋入 Git 历史，无法轻易清除。
    
    - ```bash
      git add ./libs/vmware-vix-disklib-distrib/
      git commit -m "feat: add vddk sdk"
      ```

- ✅ **优雅做法：排除依赖并在文档中声明**
    
    - **防守策略**：配置 `.gitignore` 彻底封杀二进制路径。
    
    - ```text
      # 排除所有 SDK 和大型库文件
      /vendor/vddk/
      *.so*
      *.dll
      ```
    
    - **声明策略**：在 `README.md` 中编写环境初始化指引。
    
    - ```markdown
      ### VMware VDDK
      - **版本要求**：9.0.1
      - **下载地址**：[VMware 官网](https://developer.broadcom.com/sdk/vddk/)
      - **安装路径**：请确保 SDK 解压至 `/usr/lib/vmware-vix-disklib`，并配置 `ldconfig`。
      ```
