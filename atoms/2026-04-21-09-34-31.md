# Gitea 的 Organization / Team / Repository 权限模型

## 一、核心诉求

- **只做可见性控制** 我们要解决的问题是"谁能看到代码"，不是"代码质量如何"。不做 Review、不搞 PR 审批，只负责把"人"和"仓库"的关系理清楚。

## 二、三层结构：Organization / Team / Repository

### 用公司类比理解三层

- **三层对应关系** Gitea 把权限切成 Organization → Team → Repository 三层，正好对应一家公司里的"公司 → 部门 → 项目文件柜"。

    | Gitea 概念 | 类比 | 作用 |
    | --- | --- | --- |
    | Organization | 一家公司 | 顶层容器，拥有一批仓库和一批成员 |
    | Team | 公司里的部门 | 一组人 + 一组权限规则，批量给人发"门禁卡" |
    | Repository | 项目文件柜 | 真正存代码的地方 |
    | User | 员工个人 | 具体的人，靠加入 Team 或被单独邀请获得访问权 |

- **结构示意**

    ```
    Organization（组织）
        ├── Team（团队）
        │     └── Members（成员，来自 User）
        └── Repository（仓库）
              └── Collaborators（协作者，来自 User）

    User（用户）
        └── 自己拥有的 Repository（个人仓库）
    ```

### 权限的唯一来源：两条路径

- **User 本身对仓库没有权限** 一个账号拿到仓库权限，只能走下面两条路径之一。

- **路径 1：通过 Team（批发）** 加入某个 Team，一次性获得该 Team 管辖的所有仓库的权限。适合长期参与某条业务线的成员；改 Team 权限等于批量改所有仓库权限。

- **路径 2：通过 Collaborator（零售）** 被单独加为某仓库的 Collaborator，只对这一个仓库有权限。适合临时人员、跨 Org 协作、单仓库特例。

## 三、为什么要有 Team

### Team 解决的是"批量授权"问题

- **去掉 Team 的代价** 如果只用 User + Repository（每个仓库挨个加人），Gitea 也支持，但代价很大。

    - 12 人 × 5 个仓库 = 60 次配置，新增一个仓库就要重复加 12 次人。
    - 有人离职时要挨个仓库删除，容易漏。

- **加了 Team 之后** Team 新增/删除成员时自动拿到仓库权限，不用挨个配。

### 一个具体例子：产品 A 和产品 B

- **场景** 你们有产品 A（前后端 + ML）、产品 B（前后端），共 12 个人。

    ```
    Organization: product-a
    ├── Teams
    │   ├── Owners         成员：张三（技术负责人）
    │   └── Developers     成员：李四、王五、赵六、钱七
    ├── Repositories
    │   ├── product-a-backend   （Go）
    │   ├── product-a-web       （Vue）
    │   └── product-a-ml        （Python）

    Organization: product-b
    ├── Teams
    │   ├── Owners         成员：张三
    │   └── Developers     成员：孙八、周九、吴十
    ├── Repositories
    │   ├── product-b-backend
    │   └── product-b-web
    ```

- **批量授权** 李四加入 product-a/Developers，自动获得 backend、web、ml 三个仓库的 Write 权限，不用一个个加。

- **隐私隔离** 李四看不到 product-b 下的任何仓库——他不是 product-b 的成员。隔离就是这么实现的。

- **新仓库自动纳管** 新增 product-a-mobile 时，Developers Team 的所有人自动获得权限，不用挨个配置。

### Team 的本质

- **权限的批量管理单元** Team 不是组织结构，而是"权限分发的单位"。加入 Team 等于一次性拿到 Team 名下所有仓库的权限。

## 四、Team vs Collaborator：什么时候用哪个

### Team（批发，常规情况）

- **适用场景** 稳定的团队成员、长期参与某条业务线。

- **配一次管所有** 加一次 Team，就享有该 Team 名下所有仓库的权限。

- **例子** 李四是产品 A 的后端，后端的多个微服务他都有参与，把他加入 product-a/Developers。

### Collaborator（零售，临时/特殊情况）

- **适用场景** 仅需要访问单个仓库的人，不适合拉进整个 Team。

- **权限最小化** 只对这一个仓库有权限，对 Org 的其他仓库仍然不可见。

- **典型例子**

    - 产品经理老刘要看 product-a-web 写需求 → 该仓库 Read Collaborator。
    - 外包张师傅临时修 product-a-ml 的一个 bug → 该仓库 Write Collaborator，事后移除。
    - 产品 B 的孙八偶尔要参考产品 A 某个库 → 该库 Read Collaborator，不用把他拉进整个 product-a Org。

### 选择原则

- **一个问题定去留** 这个人会长期碰这条业务线的多数仓库吗？

    - 是 → 加入 Team。
    - 否 → 单仓库 Collaborator。

- **非长期参与者不要拉进 Team** 一旦拉进 Team，就自动拿到 Team 的全部权限，后续撤权也要挨个处理，反而麻烦。

## 五、Organization 的设计原则

### 按产品建 Org，不按技术栈

- **一个 Org = 一个产品的全部代码** 不同语言的仓库归在同一个 Org 下，因为它们服务同一条业务线，权限边界也应一致。

    ```
    org: product-a          # 产品 A 相关所有代码
      ├── repo: product-a-backend   （Go）
      ├── repo: product-a-web       （Vue）
      └── repo: product-a-ml        （Python）

    org: product-b
      ├── repo: ...

    org: infra              # 公共基础设施、运维脚本
    org: shared-libs        # 跨项目共享库
    ```

### Org 是可见性边界

- **隔离的实现方式** 不是 Org 成员、也不是仓库 Collaborator 的人，看不到该 Org 的任何 Private 仓库。"代码隔离"就靠这一点。

### 常用的 Team 角色分工

- **Owners** 组织管理员（1–2 人，通常是技术负责人），Org 创建时自动生成，不可删除。

- **Developers** 该业务线的开发，Write 权限到全部仓库。

- **Viewers（可选）** 产品、测试等只读人员，Read 权限。

- **External（可选）** 外部协作者，仅授权到特定仓库。

## 六、三条核心原则

- **Org = 可见性边界** 相同产品的仓库归一个 Org。

- **Team = 权限批量单元** 长期参与者通过 Team 一次到位。

- **Collaborator = 例外通道** 临时/跨界访问用 Collaborator，不污染 Team 权限。
