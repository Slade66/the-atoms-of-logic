# Fusion Compute 远端查询接口 URL 设计反思

- 从"构建 FC 远端查询接口"这次任务里，提炼了 RESTful API 层级设计的方法论，将来在设计任何涉及第三方平台代理查询的接口时都能直接套用。

## 事件日志

- 问题起因：
    - 2026-04-23 晚上在为 Bifrost 系统设计 FusionCompute (FC) 的 Datastore 和 Portgroup 远端查询接口。
    - 查询远端平台资源的接口 URL 应该怎么定？是直接将其视为顶级资源（如 `/v1/datastores/list`），还是将其挂在特定的子路径下？
    
- 解决方案：
    - 确定将 URL 设定为 `/v1/fc-hosts/datastores/list` 和 `/v1/fc-hosts/portgroups/list`。

## 深度反思

### 体现强绑定的从属上下文（子资源概念）

在我们的 Bifrost 系统中，Datastore 和 Portgroup 并没有作为独立的“顶层实体”（像 Task 任务记录或 Host 主机记录那样）存在我们的本地数据库里。它们是依靠一个已存在的 FcHost 连接凭证去远端动态拉取的资源。

因此，从 RESTful 层级设计的角度来看，它们逻辑上属于 `fc-hosts` 这种连接凭证的下游“衍生资源”。URL 加上前缀表明了“这是基于某个 FC 主机上下文才能查到的数据”，刚好也呼应了请求体中必须传入 `host_id` 这一参数。

直白点说：怎么判断一个资源能不能当顶级路由？就看一件事——**在你的系统里，这玩意儿是自立门户的，还是寄人篱下的？** 既然 Datastore 在我们的数据库里连张自己的表都没有，想查它全靠依附着 FcHost 主机连接，那它的 URL 就必须老老实实挂在 `/fc-hosts` 下面，不能自己当老大。

### 避免未来潜在的命名冲突

如果仅用 `/v1/datastores/list` 这样泛化的命名，未来如果我们要支持其他目标平台（如将迁移目标扩展为 vCenter / 另一个 ESXi 主机 / OpenStack 等等），它们可能同样会存在 datastore 或是网络组等概念。

此时，按平台前缀将路由从一开始就进行物理隔离（例如 `fc-hosts` 与 `esxi-hosts`），能让 API 路由体系具备非常好的扩展性且语义明确。

**只要是给外部系统写 API，必须把人家的“平台名字”顶在路径的最前面。** 别干那种光秃秃挂个 `/datastores` 的傻事，等以后接入第二家、第三家系统，大家都叫 datastore，连谁是谁的都分不清，到时候再想改代码就得全线伤筋动骨了。
